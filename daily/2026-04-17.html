GPU 重置 API — 第二版（共 4 个补丁）

Onur Özkan 为 Tyr 驱动提交了 GPU 重置处理支持的第二版补丁，Boqun Feng 参与了评审。这个系列在 drivers/gpu/drm/tyr/driver.rs 中新增了一个模块，封装了底层的重置控制器逻辑，并对外暴露了一个 ResetHandle API。简单来说，当 GPU 出现异常（比如死锁或硬件错误）时，驱动需要能够"重启" GPU 而不影响整个系统——就像手机上某个 App 卡死了，你可以强制关闭它而不用重启手机。这个系列依赖于 Alice Ryhl 之前提交的"在 Rust 中创建工作队列"补丁系列，说明 Tyr 驱动正在积极利用 Rust 的新基础设施。第二版主要修复了第一版评审中发现的问题。

Zulip 讨论：新 DRM 驱动开发中的设备绑定问题

Zulip 讨论：新 DRM 驱动开发中的设备绑定问题

在 Zulip 的 #Help 频道中，Alistair Popple 和 Danilo Krummrich 讨论了开发新 DRM 驱动时遇到的设备绑定安全问题。核心问题是：当一个辅助设备（auxiliary device）在操作过程中被解绑怎么办？Danilo 解释说，UnbindGuard 可以解决设备在操作期间被意外解绑的安全问题，但对于辅助设备的 parent() 方法，它无法像通用设备那样保证返回的父设备仍然处于绑定状态。DRM 的 UnbindGuard 和设备抽象应该利用 AsBusDevice trait 来处理这种情况。Daniel Almeida 还邀请 Link Mauve 加入每周四下午 2 点（UTC）的开发者会议。

添加 bitfield! 宏 — 第二版（共 3 个补丁）

添加 bitfield! 宏 — 第二版（共 3 个补丁）

Danilo Krummrich 和 Eliot Courtney 提交了在内核 Rust 中添加 bitfield! 宏的第二版补丁。在硬件驱动开发中，经常需要操作寄存器的某几个"位"（bit），比如一个 32 位寄存器中，第 0-3 位控制速度，第 4 位控制开关——这就叫位域。bitfield! 宏让开发者可以用声明式的方式定义这些位域，而不用手写复杂的位操作代码（什么左移、右移、与、或……）。John Hubbard 也参与了评审，这个宏将成为整个内核 Rust 代码的统一基础设施。

从 register! 宏中提取 bitfield! — 第二版（共 3 个补丁）

Yury Norov 和 John Hubbard 的补丁将 bitfield! 功能从之前的 register! 宏中独立出来。之前位域操作是作为寄存器宏的一部分存在的，但社区认为位域操作是更通用的功能——不仅寄存器需要用，其他地方（比如解析数据包格式）也需要位域操作。把它拆分成独立的宏，让它可以被更多模块复用，就像把一个"瑞士军刀"里最常用的功能单独做成工具一样。

添加 SRCU（可睡眠 RCU）Rust 抽象 — 第二版（共 4 个补丁）

添加 SRCU（可睡眠 RCU）Rust 抽象 — 第二版（共 4 个补丁）

Onur Özkan 提交了 SRCU 的 Rust 安全抽象，Boqun Feng 参与了评审。SRCU 是 RCU（Read-Copy-Update）的一个变体——普通 RCU 的读者在临界区内不能睡眠（不能调用可能阻塞的函数），但 SRCU 放宽了这个限制。补丁提供了 FFI 辅助函数和基于 guard 模式的安全 API。所谓 guard 模式，就是利用 Rust 的 RAII 机制，在你进入临界区时自动获取保护，离开作用域时自动释放——不会忘记"解锁"，从根本上防止了一类 bug。同时还有配套的 MAINTAINERS 补丁，将 Rust SRCU 相关文件添加到 SRCU 的维护条目中，Onur Özkan 也被添加为 Rust SRCU 部分的维护者。

Zulip 讨论：call_rcu() 的 Rust 支持

Zulip 讨论：call_rcu() 的 Rust 支持

在 Zulip 的 #General 频道中，Philipp Stanner 提出 Nova 和 Tyr 驱动都需要 call_rcu() 的支持。call_rcu() 允许在所有 RCU 读者退出后再执行回调函数（通常用于释放内存），是一种延迟回收机制。Joel Fernandes 表示之前没有添加是因为缺少实际使用场景。Boqun Feng 指出 Alice Ryhl 有一个概念验证实现（RcuBox），但关键问题是：Nova 和 Tyr 到底想用 RcuBox（一个智能指针风格的封装）还是基于对象（object-based）的方式？这个讨论将影响最终的 API 设计方向。

修复 rustc 链接器参数解析问题 — 第七版

修复 rustc 链接器参数解析问题 — 第七版

Mohamad Alsadhan 提交了一个看似简单但影响很多开发者的构建系统修复。问题在于：rustc 的 -C linker= 选项期望接收一个单独的可执行文件路径，但很多开发者会用编译器包装器（比如 ccache gcc），此时 HOSTCC 变量包含两个单词。直接把 -Clinker=$(HOSTCC) 传给 rustc 时，shell 会把空格处拆开，rustc 会把 gcc 当成输入文件名而不是链接器的一部分，导致报错："多个输入文件"。这个补丁的修复方法是确保传给 rustc 的是单个可执行文件路径。已经迭代到第七版，说明要在各种构建环境下都正确工作并不容易。

linux-next 测试树现在自动检查 CLIPPY=1

linux-next 测试树现在自动检查 CLIPPY=1

Miguel Ojeda 在 Zulip #Announcements 频道宣布了一个重要变化：linux-next 测试树现在会自动运行 CLIPPY=1 检查。Clippy 是 Rust 官方的代码质量检查工具（类似于 C 语言世界的静态分析器），它能发现很多潜在的代码问题——比如不必要的 clone、可以简化的条件判断等。Miguel 提醒所有向上游提交补丁的开发者：请务必确保你的补丁在 CLIPPY=1 模式下是"干净的"（没有警告），这样可以大大减轻维护者的负担。这意味着 Rust-for-Linux 的代码质量标准又提高了一个台阶。

Rust 1.95.0 正式发布
