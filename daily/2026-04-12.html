GPU / DRM 驱动：nova-core 大步前进

nova-core：为 Hopper 和 Blackwell GPU 添加识别与固件支持（v10，共 28 个补丁）

这是今天最大的一组补丁，来自 NVIDIA 的 John Hubbard 和 Timur Tabi。nova-core 是社区正在用 Rust 从零编写的 NVIDIA GPU 驱动，目标是替代现有的 C 语言 nouveau 驱动。今天这组补丁的核心目标是：让 nova-core 能认出 NVIDIA 最新两代 GPU 架构——Hopper（GH100，用于数据中心的 H100 等）和Blackwell（最新一代）。

具体来说，这组补丁做了以下几件事：

📬 查看补丁系列（封面信） 📬 Hopper/Blackwell 识别补丁
nova-core：引入 SizeConstants trait 统一大小常量（v6，共 2 个补丁）

同样来自 John Hubbard 的补丁。内核中有很多表示大小的常量，比如 SZ_1M（1 兆字节）、SZ_4K（4 千字节）等，它们的类型是 usize。但 nova-core 的很多地方需要 u64 类型，以前的做法是到处写 usize_as_u64(SZ_1M) 这样的转换代码，既啰嗦又容易出错。

这组补丁创建了一个 SizeConstants trait，让你可以直接写 u64::SZ_1M，干净利落。同时还删除了不再需要的 usize_as_u64、FromSafeCast 等工具函数的引入。另外还加了一个 todo.rst 条目，记录了未来要为设备地址和大小创建"newtype 包装器"的计划——这是 Rust 的一种设计模式，用不同的类型包装同一种底层整数，防止把"内存地址"和"设备地址"搞混。

📬 查看 SizeConstants 补丁
drm/tyr：添加 GPU 重置处理（v1，共 4 个补丁）

来自 Onur Özkan 的补丁系列，为 tyr 这个 DRM 驱动添加了 GPU 重置（reset）功能。GPU 有时候会"卡住"（比如跑了一个有 bug 的着色器程序），这时候就需要驱动能够重置 GPU，让它恢复正常工作——就像手机死机了按重启键一样。这是 GPU 驱动中非常基础但又非常关键的功能，没有它，GPU 一旦挂了就只能重启整台电脑。这个系列已经收到了 2 个评审意见，正在迭代中。

设备驱动：串口总线抽象与群晖 NAS 驱动

设备驱动：串口总线抽象与群晖 NAS 驱动

Rust 串口设备总线（serdev）抽象 v4（共 4 个补丁）

来自 Markus Probst 的补丁系列，为内核添加了 serdev（串口设备总线） 的 Rust 抽象。串口（Serial Port）是一种古老但依然广泛使用的通信接口——你的电脑主板上可能没有了，但嵌入式设备、NAS、路由器里到处都是。serdev 是 Linux 内核中专门管理挂在串口总线上的设备的子系统。

这套抽象将被用于群晖（Synology）NAS 设备中的微控制器驱动。补丁说明中还提到，开发者 Kari Argillander 也表示想用这套抽象来编写一个看门狗（watchdog）驱动，说明这个抽象有通用价值，不仅仅是为一个驱动服务。这是第 4 个版本了，说明社区已经经过了多轮评审和打磨。

📬 查看补丁
群晖微控制器驱动（synology-microp）v7（共 2 个补丁）

同样来自 Markus Probst，这是一个用 Rust 编写的真实硬件驱动，目标硬件是群晖 NAS 设备中的微控制器（MCU）。这个微控制器负责控制 NAS 机箱上的各种指示灯和外设。目前这个驱动已经支持控制电源灯、状态灯、警告灯和 USB 灯，未来还计划加入风扇控制、蜂鸣器和温度传感器等功能。

这是一个很好的"Rust 进内核"的实际案例——它不是一个玩具示例，而是一个控制真实硬件的生产级驱动。已经迭代到第 7 版了，说明评审非常认真。收到了来自 Onur Özkan 的评审。它依赖于上面提到的 serdev 抽象，两个补丁系列配套使用。

⚡ IO_URING 异步 I/O 抽象

⚡ io_uring 异步 I/O 抽象

Rust io_uring 命令抽象用于 miscdevice v4（共 5 个补丁）

来自 Sidong Yang 的补丁系列，由 Greg Kroah-Hartman（他是内核子系统的重要维护者之一）参与评审。这组补丁为 io_uring（Linux 内核中最强大的异步 I/O 框架）添加了 Rust 语言的抽象层，专门针对 miscdevice（杂项设备）场景。

简单来说，io_uring 让应用程序可以一次性提交一大批 I/O 请求，然后内核在后台处理完再通知你，而不是一个一个地发请求、等回复。这就像在餐厅里一次性把所有菜都点好，而不是吃完一道再点下一道，效率高了很多。这个 Rust 抽象让 Rust 驱动也能利用这个高效的 I/O 机制。已经迭代到第 4 版，收到了 2 个评审，说明正在稳步推进。

模块参数系统

模块参数系统

为 Rust 模块参数添加布尔类型支持

来自 Wenzhao Liao 的补丁，由 Greg KH 评审。Linux 内核模块（你可以理解为"内核插件"）在加载时可以接受参数，就像命令行程序接受参数一样。比如你加载一个网卡驱动时可以指定 debug=true 来开启调试模式。

这个补丁为 Rust 编写的内核模块添加了布尔（bool）类型参数的支持。具体做法是：为 Rust 的 bool 类型实现了 ModuleParam trait，并把它和内核 C 侧已有的 PARAM_OPS_BOOL 对接起来。这样 Rust 模块就可以声明 my_flag: bool 类型的参数了。补丁作者在邮件中为之前提交过程中的一些不规范操作道了歉——这是开源社区中常见的场景，新贡献者在学习流程中逐步适应。

BUG 报告：ANDROID BINDER 睡眠问题

Bug 报告：Android Binder 睡眠问题

Rust Binder 在 deferred_release() 中从错误上下文调用了睡眠函数

用户 syscaller 使用 syzkaller（一个自动化内核模糊测试工具）对 Rust 版 Android Binder 驱动进行 fuzzing 时，反复触发了一个 bug。问题出现在 deferred_release() 函数中：代码在一个不允许睡眠的上下文中调用了一个可能睡眠的函数。

这在内核编程中是一个严重的错误。打个比方：假设你在高速公路上开车（这是"原子上下文"，不能停），但代码却试图"停车休息"（调用了可能睡眠的函数）。在原子上下文中睡眠可能导致系统死锁或者崩溃。这个 bug 出现在 Linux 7.0-rc7 版本上。Miguel Ojeda（Rust-for-Linux 项目的主要维护者）已经参与了讨论。这类由 fuzzer 发现的 bug 对提高 Rust 驱动的可靠性非常有价值。

💡 小白科普
原子上下文 vs 可睡眠上下文：内核代码运行在两种"模式"下。在原子上下文（如持有自旋锁、中断处理时）中，代码必须尽快执行完毕，绝对不能睡眠；在可睡眠上下文中，代码可以等待（比如等内存分配）。在错误的上下文中睡眠是内核编程中最常见的 bug 之一。了解更多

块设备层：超时回调讨论

块设备层：超时回调讨论

RFC：为 Rust block 层添加借用式 blk-mq 超时回调

来自 wenzhaoliao 和 Miguel Ojeda 的讨论。这是一个 RFC（征求意见稿），讨论的是如何在 Rust 的块设备层抽象中添加超时回调机制。

块设备（比如硬盘、SSD）的 I/O 请求有时候会"卡住"——比如硬盘坏了，请求发出去了但永远等不到回复。这时候就需要一个超时机制来检测并处理这种情况。blk-mq 是 Linux 内核中现代的多队列块设备层，负责管理所有块设备的 I/O 请求。

这个 RFC 的关键词是"borrowed"（借用式），这涉及到 Rust 的所有权系统——回调函数如何安全地访问请求数据，而不违反 Rust 的借用规则？这是 Rust 驱动开发中的一个典型挑战：既要满足 Rust 严格的安全检查，又要适配 C 内核 API 的回调模式。目前收到了 2 个评审意见，讨论还在继续。

️ 驱动核心与基础设施

️ 驱动核心与基础设施

驱动核心 7.1-rc1 拉取请求

Danilo Krummrich 向 Linus Torvalds 发送了 GIT PULL 请求，包含了 7.1-rc1 版本的驱动核心（driver-core）改动。这意味着这些代码即将进入 Linux 主线内核的下一个开发周期。

主要改动包括：struct platform_device_info 结构体的变化，以及为设备锁（device lock）添加了条件守护（conditional guard）。邮件中还提到，这些改动与 Rust 树（因为 Rust 最低编译器版本的升级）可能会产生小冲突，但已经在 linux-next 中测试了至少五轮。这是 Rust 代码进入内核主线流程的正常一环——不同子系统树之间偶尔会有小冲突，合并时解决即可。

📬 查看拉取请求
trace_printk 的 Rust 支持

Danilo Krummrich 在 Zulip 上分享了他正在开发的 trace_printk Rust 支持的 git 分支。trace_printk 是内核开发者最常用的调试工具之一——它把调试信息写入内核的 trace buffer，而不是输出到控制台。与普通的 printk 相比，它的性能开销小得多，适合在性能敏感的代码路径中使用。

让 Rust 代码也能使用 trace_printk，对于 Rust 驱动的开发和调试来说非常重要。目前代码在 Danilo 的个人 git 分支上，还处于早期开发阶段。
