️ GPU / DRM 驱动

GEM shmem Rust 绑定 v11 — Lyude Paul 的系列补丁

这是一组非常重要的补丁，目标是让 Rust 驱动能够安全地操作 GPU 的共享内存对象（GEM shmem objects）。想象一下：GPU 驱动需要在 CPU 和 GPU 之间共享一块内存来传输图像数据、命令缓冲区等。在 C 语言的内核代码里，这些操作已经有了一套成熟的"shmem 帮助函数"。现在 Lyude 的工作就是给这些 C 函数穿上一层 Rust 的"安全外衣"，让 Rust 驱动也能用。

这个系列的大部分补丁已经合入上游了，这次提交的是剩余部分以及根据邮件列表反馈做的额外修改。这组补丁最初来自 Asahi Linux（苹果 M 系列芯片的 Linux 驱动项目），说明 Rust 驱动生态正在不同硬件平台之间共享成果。

补丁 1：命名空间整理 — 在 shmem.rs 文件中把 device::Device 简化为 Device。因为接下来要在这个文件里同时用到 DRM 设备和内核设备两种概念，提前整理好命名可以避免混淆——就像你在代码里给两个同名的东西取不同的别名一样。

📬 查看补丁

补丁 2：引入 __drm_gem_shmem_free_sgt_locked() — 这个补丁在 C 端新增了一个带锁的释放函数。为什么需要锁？因为 GPU 驱动随时可能被卸载（unbind），但如果卸载时还有 scatterlist（分散/聚集表，用于描述不连续的内存块）正在使用，直接释放就会导致崩溃。这个函数确保在持有 dma_resv 锁的情况下安全释放资源，就像你关店之前要确认所有客人都离开了一样。

📬 查看补丁

补丁 3：导出 drm_gem_shmem_get_pages_sgt_locked() — 把一个已有的 C 内部函数导出为可供外部（包括 Rust 绑定）调用的符号。Rust 端的 SGTable 绑定需要调用它来获取 shmem 对象的页面映射表，同时利用 dma_resv 锁来保护额外的共享资源。

📬 查看补丁

补丁 4：引入 shmem::SGTable — 这是这组补丁的核心。SGTable（Scatter-Gather Table）是一种描述"分散在物理内存各处的数据块"的数据结构，GPU 用它来知道要从哪些内存地址读取数据。Lyude 在 Rust 端引入了一个 Devres（设备资源）资源来包裹 SGTable，确保在驱动卸载时所有映射都会被自动清理。这里的关键设计思想是：任何从 scatterlist 创建的映射，都必须在驱动解绑时被销毁，不能留下"悬空指针"。

补丁 5：添加 vmap 函数 — vmap 是"虚拟地址映射"的意思，让 CPU 可以像访问普通内存一样访问 GEM 对象的内容。Lyude 为 Rust 绑定增加了两种映射类型：owned mapping（拥有所有权的映射，用 shmem::VMap 表示）和 borrowed mapping（借用映射）。这个设计完美契合了 Rust 的所有权模型——你要么拥有这块映射，要么只是借来看看，编译器会帮你确保不会用错。

DmaFence Rust 封装讨论

在 Zulip 的 #DRM 频道里，Philipp Stanner、Daniel Almeida 和 Boris Brezillon 正在热烈讨论 DmaFence 的 Rust 封装设计。DmaFence 是 GPU 驱动中的"同步信号灯"——当 GPU 正在处理一个任务时，CPU 或其他 GPU 任务可以通过 fence 来等待它完成。

Philipp 最近发布了一个新的 DmaFence 设计草案，吸收了之前的设计目标和反馈。他提出了一个关键问题：Daniel 之前的代码方案会在 Rust 端重复存储 C 后端已有的信息，意味着需要额外的内存分配。Philipp 认为应该找到一种方式避免这种冗余。Daniel 则表示他正在把所有相关代码变基（rebase）到 7.0 版本，所以回复会慢一些。这个讨论还在进行中，但它的结果将直接影响未来所有 Rust GPU 驱动的同步机制。

NVIDIA Nova 驱动（GPU 卸载流程）

补丁 1：引入 warn_on_err 宏

Alexandre Courbot（NVIDIA 工程师）提交了 Nova 驱动卸载流程的第 3 版补丁系列，共 6 个补丁。Nova 是 NVIDIA 正在用 Rust 开发的新一代开源 GPU 驱动（替代 Nouveau）。这次的重点是：当驱动被卸载时，要正确地"善后"。

补丁 5：发送 UNLOADING_GUEST_DRIVER 命令

这是关键补丁之一。目前驱动卸载后，GSP（GPU System Processor，NVIDIA GPU 上的一个协处理器）会继续运行，这很危险——它还在访问内核即将回收的共享内存区域！特别是在没有 IOMMU（输入输出内存管理单元）的系统上，这可能导致内存被非法访问。这个补丁通过在卸载时发送 UNLOADING_GUEST_DRIVER 命令来通知 GSP"我要走了，你别再碰我的内存了"。

📬 查看补丁
补丁 6：运行 Booter Unloader 和 FWSEC-SB

驱动加载时，FWSEC-FRTS 固件会创建一个叫 WPR2 的安全内存区域来存放 GSP 固件，Booter Loader 会把固件加载到 GSP 并启动它。卸载时，这些操作都需要被"撤销"——特别是 WPR2 安全区域必须被清除，否则下次加载驱动时会因为区域还在而失败。这个补丁就是确保驱动卸载时运行 Booter Unloader 来清理这些资源。由 Eliot Courtney 和 Alexandre Courbot 共同完成，已有 2 条审核意见。

设备驱动基础设施（SYSFS 模块符号链接）

设备驱动基础设施（sysfs 模块符号链接）

补丁 1：按需初始化 module_kset

Shashank Balaji 提交了一组共 4 个补丁，目标是让更多内置驱动（built-in drivers）在 sysfs 中拥有模块符号链接。这听起来很抽象，让我解释一下为什么这很重要。

在嵌入式设备上，内核需要尽可能小。开发者需要知道"这个驱动对应哪个内核配置选项"，这样就能关掉不需要的驱动来减小内核体积。路径是：驱动名 → 模块名 → 内核配置名。但是目前很多总线的驱动注册函数没有设置 mod_name，导致这条路径断了——sysfs 中看不到模块的符号链接。

module_kset 原本是在 subsys_initcall 阶段初始化的，但有些平台驱动在这之前就注册了。后续补丁会设置这些驱动的 mod_name，这时就需要 module_kset 已经准备好。这个补丁改为"按需初始化"——谁先用谁先创建，避免了时序问题。就像餐厅不是等所有餐具都摆好才开门，而是客人来了就给他摆。

📬 查看补丁
补丁 2：CoreSight 驱动宏化 THIS_MODULE 传递

把 coresight_init_driver() 重命名为 __coresight_init_driver()，并用一个宏包装来自动传递 THIS_MODULE。这和其他总线的做法保持一致。THIS_MODULE 是一个指向当前模块的指针，很多内核 API 需要它来做引用计数和生命周期管理。用宏自动传递可以避免开发者手动传错。

Rust 内核抽象层

task_work 抽象 — Ashutosh Desai

Ashutosh Desai 提交了一个新的 Rust 抽象：task_work。在 Linux 内核中，task_work 机制允许你"预约"一个函数，让它在当前进程返回用户空间之前被执行。这在很多场景下很有用——比如你需要在某个操作完成后做清理工作，但不想立即做，而是等到一个安全的时间点。Greg KH 已参与审核，目前有 2 条审核意见。这个抽象将让 Rust 内核代码能够利用这个已有的 C 内核机制。

修复 "multiple candidates for rmeta dependency core" 编译错误 — Mukesh Kumar Chaurasiya

这是一个构建系统的 bug 修复，已经到了第 11 版（v11）。问题是在某些情况下编译 Rust 内核代码时会遇到"core 库有多个候选版本"的错误。这通常发生在编译器缓存出现冲突时——就像你电脑里装了两个同名软件，系统不知道该用哪个。Link Mauve 参与了审核，这个补丁对于让 Rust 代码在各种架构上顺利编译至关重要。

LWN 文章：One Sized trait does not fit all

Miguel Ojeda 在 #Announcements 频道分享了 4 月 22 日发布的一篇 LWN 文章。文章讨论了 Rust 中 Sized trait 的局限性——在内核编程中，我们经常需要处理不定大小的类型（比如变长数组、尾部灵活数组等），但 Rust 默认要求所有类型都是 Sized 的（即编译时已知大小）。这篇文章探讨了为什么"一个 Sized trait 不能满足所有需求"，以及社区正在讨论的改进方案。对于理解 Rust 类型系统在内核中的适配挑战非常有价值。

💬 查看公告
call_rcu() 与模块卸载安全

Demi Obenour 在 #General 频道提出了一个有趣的安全设计思考：如果我们故意"泄漏"模块使用的内核虚拟地址（即卸载模块时不释放这些地址），那么即使发生代码 UAF（Use-After-Free，使用已释放的内存），影响也仅限于一个内核 oops（非致命错误），而不是被恶意利用。但他也指出，要彻底解决这个问题需要语言层面的改变或者直接禁用模块卸载。这是一个关于安全性和实用性权衡的深度讨论。
