Rust 工具链版本大升级（33 补丁集）

今天社区最重磅的事件是 Miguel Ojeda 提交的 33 个补丁的大型补丁集（v2 版本），核心目标是把内核要求的最低 Rust 编译器版本从之前的版本提升到 Rust 1.85.0，同时也提升了 bindgen（一个自动生成 Rust 绑定的工具）的最低版本。这个决定的背景是：在去年的 Linux Plumbers Conference（LPC 2025）和维护者峰会上，大家讨论并达成共识——跟随 Debian Stable 的 Rust 版本作为最低要求。Debian Trixie 发布时搭载了 Rust 1.85.0，因此这就成了新的基准线。

为什么这件事重要？因为一旦提高了最低版本，很多之前为了兼容旧版编译器而写的"补丁"和"绕路代码"就可以删掉了，让代码库变得更干净。这就好比你的手机系统要求从 Android 10 升级到 Android 13，之前为了兼容旧系统写的一堆判断逻辑就可以全部删除了。

Rust 工具链版本大升级（33 补丁集）

🔇 全局允许 clippy::incompatible_msrv 警告

Clippy 是 Rust 的代码检查工具，incompatible_msrv 这个规则会警告你"你用了比最低支持版本更新的 API"。但在内核开发中，这个警告带来的实际价值不大——因为内核有自己的版本管理策略。社区多次讨论后决定全局关闭这个检查，省得开发者到处加 #[allow(...)]。Gary Guo、Tamir Duberstein 等人参与了 review。

📬 查看补丁
⚙️ 构建系统：支持按版本门控全局编译标志

新增了一个构建系统功能：可以按 Rust 编译器版本来控制全局编译标志。比如某个 lint（代码检查规则）在特定版本有误报，就可以只在那个版本关闭它。之前虽然有 rustc-min-version 这样的辅助工具，但没法直接用在全局编译标志上。这次把这个能力补齐了。已合入 rust-next 分支。

📬 查看补丁
🩹 处理 Clippy precedence lint 的版本差异

Rust 1.85.0 中 Clippy 的 precedence（运算符优先级）检查被扩展了，开始警告位运算和移位操作的优先级问题，但这产生了大量误报。Rust 1.86.0 把这部分拆到了一个新的 precedence_bits lint 中，默认不开启。所以只有 Rust 1.85.0 这一个版本有"特殊行为"。这个补丁利用上面刚加的"按版本门控"功能，只在 Rust < 1.86.0 时允许这个 lint。已合入 rust-next。

Nova Core 是 NVIDIA 正在用 Rust 开发的开源 GPU 驱动，属于 Rust-for-Linux 中最受关注的项目之一。今天有多个方面的进展：

这个补丁由 NVIDIA 的开发者提交，在 Kconfig 中明确声明 Nova Core 驱动只支持小端序的处理器。其实驱动代码里早就默认按小端序来读写数据了——比如从命令队列读取 RPC 消息时，直接把字节按小端解释。但之前没有在配置层面写明这一点，导致理论上可以在大端序机器上编译（但运行必然出错）。这个补丁就是把这个隐含假设变成显式约束。John Hubbard、Gary Guo、Eliot Courtney 等参与了 review。

📬 查看补丁
🧠 TLB 刷新支持（v10）

这是一个已经迭代到第 10 版的 21 补丁系列，由 Joel Fernandes 开发，涉及 Nova Core 的内存管理（mm）子模块。TLB（Translation Lookaside Buffer，翻译后备缓冲器）是 GPU 用来加速虚拟地址到物理地址翻译的缓存。当内存映射发生变化时，必须"刷新"TLB 以确保 GPU 看到的是最新的地址映射，否则就会访问到错误的内存——这在 GPU 驱动中是一个非常关键的正确性问题。经过 10 轮迭代可以看出社区对这部分代码的质量要求非常高。

❌ 新增 NV_STATUS 错误码绑定（v3）

NVIDIA 的 GPU 固件（GSP，GPU System Processor）在运行时会返回各种状态码，类似于 HTTP 的 404、500 状态码。这个补丁（9 补丁系列，已到第 3 版）为这些错误码创建了 Rust 绑定，让 Rust 驱动代码能够正确理解和处理 GPU 固件返回的各种错误状态。John Hubbard 和 Eliot Courtney 参与 review。

I/O 子系统：内存映射抽象泛化

I/O 子系统：内存映射抽象泛化

🔄 将 Mmio 泛化为支持任意类型

Gary Guo 和 Alexandre Courbot 共同推动了一组 8 个补丁，对 Rust 内核的 I/O 子系统进行了重要改进。

之前内核中的 Mmio（Memory-Mapped I/O，内存映射 I/O）抽象只能用于特定的数据类型。这个补丁将它泛化，使其可以包裹任意类型。这是什么意思呢？想象一下，以前你的"万能遥控器"只能控制电视，现在改造成了真正的万能遥控器，可以控制任何设备。这让驱动开发者可以更灵活地定义他们设备的寄存器布局，而不用被固定的类型限制住。

📬 查看补丁（Mmio 泛化）
📐 I/O 类型泛化与投影（Projection）

这是上面工作的延续。"投影"（Projection）是一种设计模式，允许你从一个大的 I/O 资源中"投射"出其中一小块来使用，而不需要一次性管理整个资源。打个比方：如果一个 GPU 有 100 个寄存器，你可以用投影机制只获取其中"显示控制"那 10 个寄存器的访问权限，而不用把全部 100 个寄存器的引用都拿到手。这在编写安全、模块化的驱动时非常有用。

⚙️ RUST 内核基础设施改进

⚙️ Rust 内核基础设施改进

⏰ 为 fsleep 添加 pr_warn_once! 警告

这个补丁涉及内核的时间（time）模块。fsleep 是一个"灵活睡眠"函数，能根据睡眠时间长短自动选择最佳的底层实现。当传入的时间差（delta）被截断（clamp）时——比如值太大或太小，超出了合理范围——现在会用 pr_warn_once! 打印一条警告信息。这条警告只在第一次触发时打印，避免日志被刷屏。这有助于驱动开发者发现他们传入了不合理的参数。由 Greg KH（Linux 稳定版维护者）和 Miguel Ojeda 参与讨论。

📬 查看讨论
🎯 添加 likely()、unlikely() 和 cold_path()

这是从 Rust 标准库（std_vendor）中引入的三个性能提示函数。likely() 和 unlikely() 告诉编译器某个条件分支"很可能"或"不太可能"被执行，编译器会据此优化代码布局；cold_path() 则标记一段代码为"冷路径"（不常执行的路径），编译器会把它移到远处以提升热路径的缓存命中率。这些都是性能敏感代码中常用的手段，特别是在内核这种对性能极度敏感的场景中。

📬 查看补丁
🔗 为 lru_status 声明 CFI 编码

这个已合入 rust-next 的补丁由 Alice Ryhl 提交。默认情况下 bindgen 会把 C 的 enum lru_status 转换为一个整数类型的 typedef。在大多数情况下，相同大小的整数和枚举 ABI 兼容，但在启用 CFI（Control-Flow Integrity，控制流完整性） 时，类型签名不匹配会导致检查失败。这个补丁显式声明了正确的 CFI 编码，确保在开启 CFI 安全特性时驱动也能正常工作。
