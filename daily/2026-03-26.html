2. 重构固件签名查找逻辑

随着支持的 GPU 越来越多，Gsp::new() 函数开始变得臃肿。这个补丁将 .fwsignature* 段的查找逻辑提取到了一个独立函数 find_gsp_sigs_section() 中。这是一种常见的代码整理手法——把一个大函数拆成若干小函数，每个只做一件事。随着后续新 GPU 的加入，这个函数会继续增长，但不会把 Gsp::new() 弄得一团糟。

📬 查看补丁
3. 为 Spec 和 Revision 添加 Copy/Clone，新增 chipset() 访问器

Spec（GPU 规格）和 Revision（芯片版本号）这两个结构体总共才 4 个字节，非常小。这个补丁给它们加上了 Copy 和 Clone trait——在 Rust 中，这意味着可以像整数一样"随手复制"它们，不需要关心所有权转移。同时添加了一个 Spec::chipset() 方法，方便后续补丁在 gpu.rs 之外的地方获取芯片组信息。

📬 查看补丁
4. 按 GPU 架构设置 DMA 掩码宽度

之前 nova-core 里把 DMA 掩码硬编码为 47 位，也就是说驱动告诉系统"我这个 GPU 最多能访问 2^47 字节的内存地址"。但不同架构的 GPU 其实支持的地址宽度不同！这个补丁引入了 Architecture::dma_mask() 方法，用 Rust 的 exhaustive match（穷举匹配）来为每种架构返回正确的值。好处是：如果以后有人添加新架构但忘了设置 DMA 掩码，编译器会直接报错——这正是 Rust 的类型系统帮你"防呆"的地方。补丁还把 Spec 的创建移到了 probe() 里，确保设置 DMA 掩码之前已经知道 GPU 是什么架构。这个补丁有 6 位审查者参与评审，可见大家很重视。

️ GPU 驱动 nova-core：Hopper / Blackwell 架构支持（v8 大型补丁系列）

5. Hopper/Blackwell 跳过 GFW 启动等待

老一代 NVIDIA GPU 在启动时需要等一个叫 GFW_BOOT（GPU 固件启动完成）的信号。但 Hopper 和 Blackwell 架构使用了新的 FSP（Falcon Security Processor）安全启动流程，不需要等 GFW_BOOT。这个补丁把"等不等 GFW_BOOT"这个决定放到了 HAL（硬件抽象层）里：Pre-Hopper（比如 Tu102）的 HAL 会等，Hopper 及以后的 HAL（Gh100）直接跳过。这种设计很优雅——每种 GPU 架构自己决定自己的启动流程，上层代码不需要关心细节。

📬 查看补丁
6. 固件解析代码迁移到 firmware.rs

以前只有 GSP（GPU System Processor）需要解析固件头部信息。但 Hopper/Blackwell 引入了新的固件镜像 FMC（Falcon Microcode），而且用的是 ELF32 格式（GSP 用的是 ELF64）。所以需要把 ELF64 的段解析代码从 GSP 专属的位置"上移"到更通用的 firmware.rs 模块，让 GSP 和 FMC 都能复用。这就像把一个只给一个班用的教室改造成全年级的公共自习室。

📬 查看补丁
7. 添加 FMC 固件镜像（支撑 FSP 安全启动）

这个补丁正式为 Hopper/Blackwell 添加了 FMC（Falcon Microcode）固件镜像的加载支持。FMC 是 FSP 安全启动流程中的关键固件——可以把它理解为 GPU 内部安全处理器的"操作系统"。有了这个补丁，驱动就能正确加载和使用 Hopper/Blackwell 架构所需的完整固件链。

GPU 驱动 NOVA-CORE：RM 控制命令与寄存器重构

GPU 驱动 nova-core：RM 控制命令与寄存器重构

1. RM 控制命令基础设施（v3，9 个补丁）

Eliot Courtney 提交了 RM 控制命令基础设施的第三版。这里的 RM 是 NVIDIA 的 Resource Manager（资源管理器），是 GPU 固件中负责管理各种资源（显存、通道、引擎等）的核心组件。驱动和 RM 之间通过 RPC（远程过程调用）通信——驱动发一条消息说"我要分配一个计算通道"，RM 收到后去做，然后回复结果。

这个系列具体添加了：NV_STATUS 的绑定和对应的 NvStatus 枚举（RM 用这个来报告操作结果是成功还是失败）；RM 控制 RPC 所需的 C 绑定；RmControlMsgFunction 用来标识每个控制命令；以及错误处理机制。这些都是后续实现通道分配等高级功能的前提。

📬 查看补丁系列
2. 使用 KVVec 优化 SBufferIter 刷新

同样来自 Eliot Courtney 的 RM 控制系列中的一个补丁。SBufferIter 是用来迭代共享缓冲区的工具，这个补丁将其内部的刷新（flush）操作改用 KVVec（一种内核专用的动态数组）来实现，提升了内存使用效率。

📬 查看补丁
3. 将寄存器定义迁移到内核统一宏（v4，10 个补丁）

Alexandre Courbot 提交了第四版的寄存器宏迁移系列。此前 nova-core 自己维护了一套宏来声明硬件寄存器类型，当时就说好是"临时方案，等内核有统一版本就换掉"。现在内核已经合并了通用的寄存器声明宏，所以是时候"退役"nova-core 自己的版本了。这个系列分两部分：先把通用寄存器迁移过去，然后把 Falcon 处理器（GPU 内部的小 CPU）的寄存器也迁移了。代码评审由 Danilo Krummrich 参与。

DMA 与内存访问改进

DMA 与内存访问改进

1. 泛化 write_dma()，不再限制为字节切片

Danilo Krummrich 提交、Alice Ryhl 和 Gary Guo 参与评审的这个补丁改进了 uaccess（用户空间访问）模块中的 write_dma() 函数。之前这个函数只能接受 &Coherent<[u8]>——也就是一个"一致性 DMA 缓冲区里的字节数组"。但实际上函数内部只用到了 as_ptr() 和 size() 两个方法，这两个方法对任何实现了 KnownSize + AsBytes 的类型都管用。

所以补丁把签名改成了泛型 &Coherent where T: KnownSize + AsBytes + ?Sized，这样你可以直接把一个结构体的 DMA 缓冲区传进去，不用先手动转成字节数组。这是 Rust 泛型的典型用法——让函数更通用，同时编译器帮你检查类型安全。

📬 查看补丁
2. 修复 dma_free_attrs 缺少 inline 标记

Miguel Ojeda 发现在 UML（User Mode Linux）构建下编译一个即将合入的补丁系列时，编译器对 dma_free_attrs 函数报了一个 -Wstatic-in-inline 警告。原因是这个 C 函数被一个内联函数调用了，但它自己没有标记为 inline。想象一下：一个标着"可以随处复制"的函数（inline）试图调用一个"只能在本文件使用"的函数（static），编译器觉得这不太对劲。修复很简单——给 dma_free_attrs 也加上 inline 标记。Leon Romanovsky 和 Marek Szyprowski 参与了评审。

📬 查看补丁
3. 新增 const_align_up() 指针对齐工具函数

来自 Hopper/Blackwell 大系列的一个基础补丁。John Hubbard 在 kernel::ptr 模块中添加了 const_align_up() 函数。这是已有的 Alignable::align_up() 的 const 兼容版本——也就是说它可以在编译时求值。这在固件解析等需要编译期计算对齐地址的场景中很有用。Alice Ryhl 和 Gary Guo 参与了评审。

RUST 内核基础设施

Rust 内核基础设施

1. KASAN：移除 -Zsanitizer-recover 标志

Alice Ryhl 发现了一个 Rust 端 KASAN 配置的一致性问题。KASAN 是内核自带的内存错误检测工具（类似于"内存安全卫士"），能在运行时发现越界访问、使用已释放内存等 bug。在 C 代码那边，内核并没有给 clang 传 -fsanitize-recover 标志（意思是"发现内存错误就直接终止"），但 Rust 这边却给 rustc 传了 -Zsanitizer-recover（意思是"发现错误继续执行"）。这样 C 和 Rust 代码对同一类错误的处理方式不一致。修复方法是把 Rust 端多余的 recover 标志去掉，保持两边行为一致。Danilo Krummrich 和 Gary Guo 参与了评审。

📬 查看补丁
2. 为所有 Error::from() 函数添加 #[inline] 标记（v2）

Alistair Francis（来自 Western Digital）提交了第二版补丁，给内核 Rust 代码中所有 impl From<...> for Error 的 from() 函数加上了 #[inline] 属性。这些函数的作用是把各种错误类型转换成内核统一的 Error 类型，它们通常都非常简单——可能就一两行代码。如果不标记 inline，每次调用时编译器可能会生成一次真正的函数调用（压栈、跳转、返回），对这种一行代码的小函数来说完全是浪费。加上 #[inline] 后编译器会把函数体直接嵌入调用处，消除调用开销。Miguel Ojeda 和 Gary Guo 参与了评审。

📬 查看补丁
3. pin-init 宏改进（2 个补丁）

Benno Lossin（pin-init 子系统的维护者）提交了两个上游回移（backport）补丁，改进了 pin-init 宏的内部实现：

补丁一：文档化字段访问器的对齐保证。 init! 宏在初始化结构体字段时，传入的指针必须是对齐的（aligned）。这个对齐保证实际上是由字段访问器（field accessor）在创建时确保的，但之前代码里没有注释说明这一点。这个补丁加上了文档，让未来的维护者知道"这个对齐不是巧合，是有保证的"。

补丁二：用 unsafe 令牌替换遮蔽返回令牌。 初始化宏生成的闭包中，之前用了一个叫 __InitOk 的单元结构体作为返回值，并在闭包内部重新定义同名结构体来"遮蔽"外部的，防止用户提前返回 Ok。新方案改用一个只能通过 unsafe 创建的令牌类型，更清晰地表达了"这个返回值只有宏自己能构造"的意图。

⚙️ 构建系统

⚙️ 构建系统

kbuild：为 rustc -C linker 使用单一可执行文件（v5）

Yoann Congal 提交、Nicolas Schier 参与评审的这个补丁改进了内核构建系统中 Rust 编译器调用链接器的方式。当内核构建系统（kbuild）编译 Rust 代码时，需要告诉 rustc 用哪个链接器。之前的实现有些复杂，这个补丁将其简化为使用单一可执行文件来传递 -C linker 参数。这已经是第 5 版了，说明这个看似简单的改动涉及很多构建系统的细节，需要反复调整才能在各种配置下都正确工作。对于内核开发者来说，构建系统的改进虽然不那么"性感"，但直接影响每天的开发体验。

社区讨论

社区讨论

1. Zulip：能在内核模块中使用外部 Rust crate 吗？

新人 Alberto Lange 在 Zulip 的 #Modules 频道提了一个很多人都会好奇的问题：能不能在 Rust 内核模块中使用外部的 no_std crate（具体是来自 libcrux 的密码学库）？

Miguel Ojeda（Rust-for-Linux 项目的主要维护者）给出了重要回复：内核很可能永远不会支持直接使用任意 Cargo crate。这是因为内核对代码质量、许可证、维护责任等有严格要求，随便引入外部依赖会带来巨大的维护负担和安全风险。Miguel 建议仔细说明具体想做什么，才能给出更有针对性的建议。

这是一个非常重要的社区立场——如果你想在内核里复用 Rust 生态的代码，通常需要把需要的部分"移植"进内核源码树，而不是简单地 cargo add。

GitHub 上的 Issue #354 是一个由项目维护者 ojeda 维护的元议题（meta issue），持续跟踪 Rust-for-Linux 项目需要但 Rust 语言/编译器尚未稳定提供的功能特性。标签包含 toolchain 和 meta，表明这是一个长期存在的跟踪项。如果你对"内核 Rust 开发到底还缺什么语言特性"感兴趣，这个 issue 是很好的参考。
