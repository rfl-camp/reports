bus1：十年磨一剑的 IPC 系统回归

今天最大的新闻：David Rheinsberg 提交了一个由 16 个补丁组成的 RFC 系列——bus1，一个「基于能力（capability-based）的进程间通信系统」。这个项目的核心思想其实在十年前就已经提出了（LWN 上有过 Neil Brown 的详细介绍），但直到今天才以全新的 Rust 实现正式回归。

你可以把 bus1 理解成一种新的「进程之间互相打电话」的方式。现在 Linux 上已有 D-Bus、binder 等 IPC 机制，但 bus1 的独特之处在于它是「基于能力」的——每个进程只能访问它被明确授权的资源，就像你有一把钥匙只能开你自己家的门。而且这次是全 Rust 实现，天然具备内存安全优势。

Miguel Ojeda（Rust-for-Linux 的主要维护者）对这个系列的多个补丁进行了审查，提出了一些代码风格和设计上的建议。

LockedBy::access_mut_unchecked() — 新的锁访问方式

LockedBy::access_mut_unchecked() — 新的锁访问方式

这个系列的第一个补丁给 LockedBy 类型增加了一个新方法。LockedBy 是内核中用来「证明你持有某把锁才能访问数据」的工具。新增的 access_mut_unchecked() 允许你在不需要可变借用拥有者的情况下获取可变访问权。这在同一把锁保护多个 LockedBy 对象时特别有用——否则 Rust 的借用检查器会拒绝你同时访问它们。

Vec::into_boxed_slice() — 把动态数组「封装」成固定大小

这个补丁给内核中的 Vec 类型添加了 into_boxed_slice() 方法，和标准库中的同名方法类似。它解决的问题是：你经常需要用 Vec（一个可变长数组）来初始化数据，初始化完成后就不再需要改变大小了。这时可以调用 into_boxed_slice() 把它「封死」成一个固定大小的 Box<[T]>，既节省内存又更安全。就像你搬家时用纸箱装东西，装满后封箱贴好标签，后续只需要取用而不再往里塞。

修复 bitfield 的 Default 实现

Hopper/Blackwell 架构：跳过 GFW 启动等待（v9，31 个补丁）

这是一个大型补丁系列的第 9 版，由 Joel Fernandes 和 Alexandre Courbot 共同推进。它针对 NVIDIA 的 Hopper 和 Blackwell 架构 GPU（也就是 H100、B100 这类最新的数据中心级显卡），优化了驱动启动流程。具体来说，是跳过了 GFW（GPU Firmware）的启动等待步骤。GFW 是 GPU 内部的固件，在某些条件下驱动不需要等待它完全启动就可以继续初始化，这样可以加快驱动加载速度。这个系列已经迭代了 9 个版本，共 31 个补丁，可见上游对代码质量的严格要求。

ptr: 添加 const_align_up()（v9，31 个补丁）

ptr: 添加 const_align_up()（v9，31 个补丁）

这是一个大型补丁系列，由 Danilo Krummrich 主导，John Hubbard 和 Alexandre Courbot 参与，收到了 7 条 review。这个补丁给内核的 Rust 指针工具库添加了 const_align_up() 函数——它的作用是在编译时（const context）将一个地址向上对齐到指定边界。内存对齐是底层编程中极其重要的概念：CPU 访问对齐的地址更快，有些硬件甚至要求必须对齐才能访问。这个函数让你可以在编译期就完成对齐计算，而不用等到运行时，既高效又安全。迭代到第 9 版、31 个补丁，说明这是一个涉及面很广的基础设施改进。

pin-init: 用 unsafe-to-create token 替换 shadowed return token

Gary Guo 提交了一个补丁来改进 pin-init（内核中用于安全初始化被固定（pinned）数据结构的框架）的内部实现。之前的实现使用了一种 「遮蔽 return 关键字」的技巧来防止用户在初始化闭包中提前返回（因为提前返回会导致数据处于未完全初始化的不安全状态）。这个补丁改用一个「需要 unsafe 才能创建的 token」来达到同样的目的，设计上更符合 Rust 的惯用模式，也更容易被其他开发者理解。Greg KH 参与了审查，共 6 条评论。

编译错误：线程安全问题

编译错误：线程安全问题

linux-next 中出现 Rust 线程安全编译错误

linux-next 中出现 Rust 线程安全编译错误

在 linux-next（内核的"预览分支"，所有开发者的补丁最先汇聚到这里）的最新构建中，出现了一个 Rust 编译错误：error[E0277]: *const kernel::bindings::vm_uffd_ops cannot be shared between threads safely。翻译过来就是：Rust 编译器发现 vm_uffd_ops 这个 C 结构体的指针没有实现 Send/Sync trait，不能在线程间安全共享。

这实际上是 Rust 类型系统在「保护」我们——C 代码中可以随意在线程间传递指针，但 Rust 要求你显式声明「这个指针在多线程下是安全的」。Greg Kroah-Hartman 和 Alice Ryhl 正在讨论如何解决这个问题，可能需要为这个类型手动实现 Send 和 Sync（在确认 C 端确实安全的前提下），或者调整绑定方式。
