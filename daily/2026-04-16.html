️ GPU 驱动：Nova-Core 全面支持 GA100 计算卡

补丁 1：跳过 IFR 头部

今天 GPU 方向的主角是 NVIDIA GA100——这是一款纯计算用途的 GPU（没有显示输出），常见于数据中心。Timur Tabi 提交了一整套共 6 个补丁，目标是让 nova-core（NVIDIA 开源 GPU 驱动的 Rust 实现）能够驱动这款"怪咖"显卡。为什么说它怪呢？因为它虽然架构上属于 Ampere 家族（A100 那一代），但启动流程却跟上一代 Turing 一样，而且它的 VBIOS（显卡固件）里还藏着一些特殊结构需要额外处理。

GA100 的显卡 ROM 前面有一个叫做 IFR（Init-from-ROM）头部的特殊数据块，它出现在标准的 PCI 扩展 ROM 镜像之前。就好比一本书正文前面多了一页不属于目录的"说明页"，驱动在读取 VBIOS 时必须先识别并跳过这个头部，才能找到真正的固件数据起始位置。这个补丁让 nova-core 的 PROM 影射方法能够解析 IFR 头部，正确计算偏移量。该补丁收到了 11 条审查意见，讨论非常充分。

补丁 2：使用正确的固件签名

虽然 GA100 和 Turing 架构共用相同的 GSP-RM 固件文件，但 GA100 有自己专属的固件签名（fwsignature）。如果用错签名，固件校验就会失败，GPU 无法正常启动。这个补丁修复了之前的代码——之前 GA100 被错误地归为通用 Turing 签名。Gary Guo 参与了审查。

📬 查看补丁
补丁 3：不再识别已废弃的 0xBB77 PCI ROM 签名

PCI 扩展 ROM 通常以 0xAA55 签名开头，但 NVIDIA 显卡还有一些内部专用签名。其中 0xBB77 是一个已经被废弃超过十年的内部签名，nova-core 永远不会遇到使用这个签名的 GPU。这个补丁把对它的识别代码删掉了，让代码更干净。

补丁 4 & 5：FRTS 区域的条件处理

FRTS（Falcon Region Table of Signatures）是 GPU 安全启动流程中的一块特殊内存区域。GA100 比较特殊——它根本不需要 FRTS（大小为 0）。如果驱动在 FRTS 大小为 0 时还尝试去启动 FWSEC-FRTS 流程，就会出错。这两个补丁分别做了两件事：一是在 FbHal 硬件抽象层中添加 frts_size() 方法让 GA100 返回 0；二是在启动逻辑中增加判断——只有当 FRTS 区域确实被分配了（大小不为 0）时才运行安全启动。值得注意的是，不能简单地用"是否支持显示输出"来判断 FRTS 大小，因为有些 GPU（如 GA102GL）虽然禁用了显示但仍需要 FRTS。

补丁 6：正式启用 GA100 支持

前面 5 个补丁搞定了所有前置工作，这个最终补丁把 GA100 的设备 ID 正式加入 nova-core 的支持列表。从此 nova-core 可以识别并驱动 GA100 计算卡了。虽然架构上属于 Ampere，但由于启动方式与 Turing 一致，所以在驱动内部 GA100 被归类到 Turing 的启动路径。

网络子系统：Netlink 抽象 & Binder 事务上报

Rust 原生 Netlink 抽象 v3

Alice Ryhl（Google 工程师，Rust-for-Linux 核心贡献者）提交了 Rust 版 netlink 抽象的第三版补丁。Netlink 是 Linux 内核与用户空间程序之间通信的主要机制之一——你可以把它想象成内核和应用程序之间的"对讲机"。这个抽象层提供了安全且简洁的 API，让 Rust 驱动可以方便地构造 netlink 消息、添加各种属性，并通过广播发送出去。目前只支持广播模式（一对多发送），因为第一个用户只需要广播功能。Andrew Lunn 参与了审查。

📬 查看补丁
Binder 驱动通过 Netlink 上报异常事务

这是上面 netlink 抽象的第一个实际用户！补丁由 Carlos Llamas 编写、Alice Ryhl 提交，为 Android 的 Binder 驱动（Rust 版本）添加了 netlink 事务失败上报功能。Android 系统中，Binder 是进程间通信（IPC）的核心机制——应用之间互相调用服务都要通过它。当某个进程频繁出现 Binder 事务失败时（比如一个应用疯狂发送请求但对方没有响应），内核会通过 netlink 把这些失败信息上报给用户空间的守护进程，守护进程就可以采取措施（比如杀掉行为异常的应用）。这个功能对 Android 系统的稳定性至关重要。

Netlink：Linux 内核提供的一种套接字通信机制，主要用于内核和用户空间程序之间传递信息。比如你用 ip 命令配置网络时，底层就是通过 netlink 和内核通信的。
Binder：Android 系统独有的进程间通信（IPC）机制。当你的微信调用相机拍照时，底层就是通过 Binder 让微信进程和相机服务进程互相通信的。

⚡ 异步 I/O：IO_URING 命令的 RUST 抽象 V5

⚡ 异步 I/O：io_uring 命令的 Rust 抽象 v5

Sidong Yang（来自 Furiosa AI）提交了 io_uring 命令抽象的第五版补丁系列（共 4 个补丁）。io_uring 是 Linux 近年来最重要的性能创新之一——它允许应用程序批量提交 I/O 请求，内核异步处理后再统一返回结果，大幅减少了系统调用的开销。你可以把传统 I/O 想象成"每次去食堂打一个菜就回一趟座位"，而 io_uring 就像"把所有想吃的菜一次性告诉厨师，做好了一起端过来"。

这个补丁系列的结构很清晰：第一步把 io_uring 的 C 头文件引入 Rust 绑定；第二步实现核心的 io_uring Rust 抽象；第三步将它接入 miscdevice（杂项设备）框架，让 Rust 驱动可以处理 IORING_OP_URING_CMD 类型的 io_uring 透传命令。Greg Kroah-Hartman（Linux 稳定版维护者）参与了审查。这对于需要高性能 I/O 的 Rust 内核驱动来说是一个重要的基础设施。

📬 查看补丁
💡 小白科普
io_uring：Linux 5.1 引入的高性能异步 I/O 框架。传统的 I/O 方式每次操作都需要一次系统调用（从用户态切换到内核态），开销很大。io_uring 通过在用户态和内核态之间共享两个环形缓冲区（提交队列和完成队列），让批量 I/O 操作几乎零开销。
miscdevice（杂项设备）：Linux 中一类"小型"字符设备的统称。如果你的设备功能简单、不值得单独分配一个主设备号，就可以注册为 miscdevice。

社区治理：RUST 同步原语有了正式维护团队

社区治理：Rust 同步原语有了正式维护团队

Boqun Feng 提交了一个重要的 MAINTAINERS 文件更新补丁：为 Rust 同步原语（kernel/sync/ 下的代码，包括互斥锁 Mutex、自旋锁 SpinLock、条件变量 CondVar 等）建立了正式的维护团队。

这件事的背景是：最近几个开发周期中，Rust 同步原语方向已经连续出现了两个包含 10+ 补丁的合并请求，工作量越来越大。为了更好地管理代码审查和合并流程，Boqun Feng 邀请了 Gary Guo 和 Alice Ryhl 担任共同维护者，还组建了一支审查团队。Onur Özkan 也参与了讨论。

这意味着什么？简单来说，以后提交到 Rust 同步原语方向的补丁会有专人负责审查和合并，不再需要所有事情都由 Rust-for-Linux 总维护者一个人扛。这是 Rust 在 Linux 内核中逐步走向"正规军"的标志——就像一个创业公司长大后开始设立各个部门一样。

MAINTAINERS 文件：Linux 内核源码根目录下有一个 MAINTAINERS 文件，列出了每个子系统/模块的维护者和审查者是谁。当你提交补丁时，get_maintainer.pl 脚本会根据这个文件告诉你应该抄送给谁。

基础工具：位域宏与 PER-CPU 优化

基础工具：位域宏与 Per-CPU 优化

bitfield! 宏的提取与独立化 v2

John Hubbard 和 Yury Norov 合作提交了两个相关补丁系列（各 3 个补丁）。第一个系列是从现有的 register! 宏中提取出一个通用的 bitfield! 宏；第二个系列则完善了这个新宏的功能。

在内核驱动开发中，操作硬件寄存器时经常需要对一个整数的特定几个比特位进行读写——比如一个 32 位寄存器中，第 0-3 位表示设备状态，第 4-7 位表示中断使能。bitfield! 宏就是用来方便地定义和操作这些位域的工具。把它从 register! 宏中独立出来，意味着不仅硬件寄存器可以用，其他需要位域操作的场景也能复用这个宏了。Eliot Courtney 参与了审查。

📬 查看提取补丁 📬 查看 bitfield! 宏补丁
Per-CPU 变量的数值优化 v5

Mitchell Levy 提交了 Rust per-CPU 变量数值操作的"针孔优化"（pin-hole optimizations）第五版补丁（共 8 个补丁）。Per-CPU 变量是一种特殊的数据存储方式：每个 CPU 核心都有自己独立的一份副本，这样各个核心操作自己的数据时就不需要加锁，性能极高。

这个补丁系列针对数值类型（整数、计数器等）的 per-CPU 操作进行了优化。所谓"pin-hole 优化"，指的是通过更精细的类型约束（利用 Rust 的 trait 系统），让编译器能够为简单的数值操作（加、减、读、写）生成更高效的机器码，减少不必要的间接调用。Yury Norov 参与了审查讨论。

📬 查看补丁
💡 小白科普
位域（Bitfield）：把一个整数的不同比特位段赋予不同含义的技巧。比如一个 8 位数中，低 4 位表示颜色编号，高 4 位表示亮度。在硬件驱动中极其常见，因为硬件寄存器就是这样设计的。
Per-CPU 变量：每个 CPU 核心拥有独立副本的变量。好处是读写时不需要加锁（因为没有竞争），坏处是占用更多内存（N 个核心就有 N 份）。常用于计数器、统计信息等场景。

PHY 驱动：C 转 RUST 迁移工具的试验场

PHY 驱动：C 转 Rust 迁移工具的试验场

一位来自中国人民大学的博士生 wenzhaoliao 在邮件列表上发起了一个有趣的讨论：他正在开发一个 C 到 Rust 的自动迁移工具，想用一个真实的 Linux PHY 驱动作为验证目标。他选中的是 MAE0621A——一个目前只有树外（out-of-tree）C 实现的以太网 PHY 芯片驱动。

PHY 维护者 Andrew Lunn 参与了回复和讨论。这类讨论通常会涉及几个关键问题：自动生成的 Rust 代码质量如何？是否能通过社区的代码审查标准？与手写的 Rust 抽象层如何对接？从讨论的活跃程度来看（至少两轮回复），社区对这个方向是有兴趣的，但也可能存在对自动翻译代码质量的顾虑。

PHY 驱动是 Rust-for-Linux 最早成功落地的领域之一（第一个合入主线的 Rust 驱动就是 PHY 驱动），所以选择这个方向做 C-to-Rust 迁移验证是非常明智的——已有成熟的 Rust 抽象层可以复用。

PHY（Physical Layer Device）：网络设备的物理层芯片，负责将数字信号转换为电信号在网线/光纤上传输。你电脑网卡里就有一颗 PHY 芯片。PHY 驱动告诉内核如何与这颗芯片通信。

驱动核心 7.0-rc1 合并讨论 & Probe 函数注册问题

Danilo Krummrich 发起了 Driver core changes for 7.0-rc1 的 GIT PULL 请求讨论。在此过程中，Uwe Kleine-König 和 Cristian Marussi 引出了一个有趣的技术问题：一个驱动的 probe 函数是否可以在执行过程中注册另一个驱动？

Probe 函数是设备驱动的"入口点"——当内核发现一个新设备时，会调用对应驱动的 probe 函数来初始化设备。如果 probe 函数内部又去注册新的驱动，就可能引发复杂的生命周期和锁依赖问题。这个讨论涉及到 Rust 驱动框架中如何正确建模这种"嵌套注册"行为。

Zulip 热议：通用 I/O 后端的宏语法设计

Zulip 聊天的 #General 频道中，Danilo Krummrich、Alice Ryhl、Benno Lossin 和 Gary Guo——Rust-for-Linux 的四位核心开发者——围绕通用 I/O 后端的宏语法展开了热烈讨论（28 条消息！）。

争论的焦点是：用什么样的语法来表示 I/O 寄存器的映射关系？有人提议用 project!(foo, [variant..end]) 这样的写法，但 Danilo 觉得这种语法太丑了（原话 "Ick!"），建议找一种更像 Benno 第三点提议的替代方案。Gary 倾向于"就这样吧，能用就行"，但 Benno 坚持要找到更好的写法。Danilo 又提议在 variant 后面加冒号作为分隔符。这种关于语法细节的争论在编程语言和宏设计中非常常见——好的语法能让代码更易读、更不容易出错。

DRM 驱动的设备生命周期挑战

在 Zulip 的 #Help 频道，Danilo Krummrich 讨论了编写新 DRM（Direct Rendering Manager）驱动时面临的一个核心难题：DRM 的 ioctl（用户空间通过 ioctl 系统调用控制 GPU）与设备/驱动的生命周期没有同步。也就是说，当设备正在被移除（比如热拔插 USB GPU）时，用户空间可能还在通过 ioctl 发送命令——这就像你正在打电话时手机突然被抽走了。这个问题在 Rust 中尤其重要，因为 Rust 的所有权和生命周期系统要求开发者必须精确定义"谁拥有什么、什么时候释放"。Boris Brezillon、Alistair Popple 和 Link Mauve 被 CC 到了讨论中。

DRM（Direct Rendering Manager）：Linux 内核中管理 GPU 的子系统。你在 Linux 上能看到桌面画面，底层就是 DRM 在工作。它负责显示模式设置、GPU 内存管理、命令提交等。
