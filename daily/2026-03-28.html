GPU/DRM 驱动：Nova-core 大步前进

📦 统一页表抽象（v11，20 个补丁）

Nova-core 是社区正在用 Rust 从头编写的 NVIDIA GPU 开源驱动，目标是替代现有的 C 语言 nouveau 驱动。今天这个项目收到了大量补丁，几乎覆盖了从底层内存管理到固件加载的各个方面。可以说，nova-core 正在以肉眼可见的速度成长。

Joel Fernandes 提交了第 11 版的「统一页表项（PTE）包装枚举」补丁集，这是一项非常重要的架构工作。GPU 有自己的内存管理单元（MMU），用来把虚拟地址翻译成物理地址——就像 CPU 的页表一样。问题是 NVIDIA 不同代的 GPU 用的页表格式不同（MMU v2 和 v3），如果每次都写两套代码就太麻烦了。这组补丁定义了统一的 Pte、Pde、DualPde 枚举类型，构造时传入 MMU 版本参数，内部自动适配 v2 或 v3 的格式。这样页表遍历器和虚拟内存管理器只需要写一套代码就能同时支持两个版本，大大减少了重复劳动。已有 19 位审阅者参与评审，说明社区对这个设计非常重视。

📦 VBIOS 安全访问检查（5 个补丁）

Joel Fernandes 修复了 nova-core 中 VBIOS 解析代码的安全隐患。在 setup_falcon_data 和 FwSecBiosImage::ucode 函数中，之前直接用固件提供的数值做数学运算和内存访问，如果固件数据异常可能导致整数溢出或越界访问。现在改用 Rust 的 checked arithmetic（带检查的算术运算），如果计算溢出会返回错误而不是产生未定义行为。这就是 Rust 在内核中的价值体现——用类型系统和安全检查来防止传统 C 代码中常见的内存安全问题。

📦 SysmemFlush 资源泄漏修复

Danilo Krummrich 修复了一个资源清理 bug：当 Gpu::new 初始化过程中，如果在 SysmemFlush 创建之后、其他步骤失败了，SysmemFlush 不会被自动清理（因为它需要手动注销）。修复方法是为它添加 Drop 实现（Rust 的自动析构机制），确保失败时也能正确清理。同时保留了手动注销路径，因为手动路径可以保证不会出错，而 Drop 路径依赖运行时状态可能会出问题。这是一个很好的 Rust 资源管理（RAII）实践案例。

⚙️ 内核 Rust 基础设施与子系统抽象

📦 io_uring 命令抽象（v4，5 个补丁）

除了 GPU 驱动，社区也在持续为 Linux 内核的各个核心子系统构建 Rust 安全抽象。这些抽象就像是"翻译层"——把 C 语言的内核 API 包装成安全的 Rust 接口，让后续用 Rust 写驱动的开发者不需要直接跟危险的 C 代码打交道。

Sidong Yang 提交的这组补丁为 Rust 驱动接入 io_uring 打开了大门。io_uring 是 Linux 最强大的异步 I/O 框架，能让应用程序以极低的开销发起大量 I/O 操作。这组补丁分为几步：先把 io_uring 的 C 头文件加入 Rust 绑定，然后构建核心的 Rust 抽象层，最后把它接入 miscdevice（杂项设备）框架。这意味着未来用 Rust 写的字符设备驱动可以直接支持 io_uring 高性能 I/O，而不需要写任何 unsafe 的 C 交互代码。Greg Kroah-Hartman（内核维护者之一）也参与了评审。

📬 查看补丁
📦 Netlink 原始抽象（v2，4 个补丁）

Alice Ryhl（Google）为 Rust 添加了 Netlink 的底层抽象。Netlink 是 Linux 内核和用户空间通信的主要方式之一——你用 ip 命令配置网络、用 ss 查看连接，背后都是通过 Netlink 协议跟内核对话。之前 Binder（Android 的进程间通信机制）需要用到 Netlink，所以 Alice 构建了这个抽象。这是 v2 版本，说明经过了一轮社区反馈和改进。Andrew Lunn（网络子系统维护者）也参与了审阅。

📬 查看补丁
📦 Workqueue 安全取消方法（v4，3 个补丁）

Onur Özkan 为 Rust 的 workqueue（工作队列）抽象添加了安全的取消和状态查询方法。工作队列是内核中延迟执行任务的机制——比如你插入一个 USB 设备，内核不会在中断处理中做所有事情（那样会卡住系统），而是把耗时的初始化工作丢进队列里稍后执行。现在 Rust 代码可以安全地取消已提交的工作项，并查询它们的执行状态，而不需要写 unsafe 代码。已经迭代到第 4 版了。

📦 ACPI PRP0001 匹配修复（v3）

Markus Probst 修复了 Rust ACPI 子系统中的一个设备匹配 bug。当设备通过 ACPI 的 PRP0001 兼容性标识符（用于让 ACPI 设备兼容 Device Tree 的匹配方式）注册时，Rust 侧的 id_info 会返回 None，导致驱动无法正确识别设备。修复方法是导出 acpi_of_match_device 函数，并在 Rust 中使用它来进行匹配。不能直接用 device_get_match_data 是因为 Rust 在设备 ID 中存储的是索引而不是数据指针。Danilo Krummrich 和 Rafael J. 🔥 依赖性资源管理 DevRes（9 条消息）

🔥 Rust 模块的 C API 绑定问题（11 条消息）

David Rheinsberg 提出了一个很实际的问题：如果一个 Rust 内核模块需要向内核其他部分暴露 C 语言接口（比如 DRM 工具模块用 Rust 写，但需要被 C 驱动调用），怎么确保 Rust 侧和 C 侧对接口的理解是一致的？Alice Ryhl 回答说：对于常量，Rust 代码应该通过 bindgen 使用 C 头文件中定义的值；对于函数，应该使用她已经添加的 #[export] 宏来检查函数签名是否正确。但对于结构体大小，目前还没有好的方案，因为 C 头文件不能依赖 Rust 代码，只能在 Rust 侧猜测/高估大小并加断言验证。
