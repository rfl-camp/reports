Rust 版本升级与构建系统（重大更新！）

今天最重磅的事件：Rust 的最低支持版本正式从旧版本提升到了 1.85.0。这是一个包含 33 个补丁的大型补丁系列，由 Miguel Ojeda 主导，Benno Lossin、Tamir Duberstein、Gary Guo 等多位核心开发者共同贡献和审查。

你可能会问：为什么要升级编译器版本？这就好比你手机的操作系统升级一样——新版本的 Rust 编译器带来了更好的功能、更稳定的 API，也让内核开发者可以用更简洁的代码写出同样的功能。这次升级选择跟随 Debian Trixie（Debian 的最新稳定版），这意味着只要你用 Debian 稳定版的 Rust 工具链就能编译内核，大大降低了门槛。这个策略是在 LPC 2025 和维护者峰会上讨论决定的。

📬 查看补丁：版本升级主补丁
🧹 Clippy MSRV 更新与诊断改善

随着编译器升级，Clippy（Rust 的代码检查工具，相当于一个帮你找代码坏味道的"小助手"）的最低支持版本也同步更新了。更新后，Clippy 能给出更精准的诊断信息。之前有些代码里加了 allow(incompatible_msrv) 注解来"压制"不必要的警告，现在这些注解都可以安全删除了——代码变得更干净。由 Tamir Duberstein 编写，Gary Guo 和 Miguel Ojeda 审查。

📬 查看补丁
📐 简化 RUSTC_VERSION Kconfig 条件

内核的构建配置系统（Kconfig）里有一些条件判断，比如"如果 Rust 编译器版本大于 X 就启用某功能"。升级之后，很多这样的条件判断变成了"永远为真"——因为新的最低版本已经超过了它们的门槛。这个补丁把这些冗余的条件判断清理掉了，让配置文件更简洁。另外还指出，新的最低版本要求 LLVM 18（之前是 LLVM 16），但目前没有需要清理的 LLVM 版本相关代码。由 Gary Guo 编写。

📬 查看补丁
🧩 移除 RUSTC_HAS_COERCE_POINTEE 并简化代码

这个补丁删除了一个叫 RUSTC_HAS_COERCE_POINTEE 的 Kconfig 选项。以前，因为旧版 Rust 不支持 CoercePointee 这个特性（它允许自定义智能指针像标准库的指针一样自动转换类型），内核代码不得不使用一些不稳定的旧 API（coerce_unsized、dispatch_from_dyn、unsize）作为替代方案。现在 Rust 1.84.0 以上都支持这个特性了，所以可以把所有旧的替代方案删掉，统一用新写法。代码更简洁、更安全。由 Gary Guo 编写，Miguel Ojeda 和 Danilo Krummrich 审查。

🗺️ 简化 --remap-path-prefix 变通方案（两个补丁）

--remap-path-prefix 是一个编译器参数，用于在编译产物中把本地文件路径替换成统一的路径（这样不同机器编译出的二进制文件内容一致，对可重复构建很重要）。以前 rustdoc（Rust 的文档生成工具）不识别这个参数，需要绕路。现在新版本的 rustdoc 已经支持了，所以第一个补丁简化了相关逻辑。

此外还有一个编译器内部崩溃（ICE, Internal Compiler Error）的 bug，在 Rust 1.87.0 中被修复了。第二个补丁把这个 ICE 的临时修补方案改成了有条件应用——只在旧版本编译器上才生效，这样可以在将来更容易地确认什么时候可以完全移除这些变通代码。由 Gary Guo 编写，Miguel Ojeda 审查。

➕ 使用新稳定的 NonNull::add() 简化内存分配代码

在内核的内存分配模块中，之前需要把 NonNull 指针（一种保证非空的指针类型）先转成原始指针，做偏移运算后再转回来——这个过程很绕。现在 Rust 1.85.0 稳定了 NonNull::add() 方法，可以直接在 NonNull 上做指针偏移，一步到位。补丁清理了代码并删除了之前留下的 TODO 注释。由 Gary Guo 编写，Miguel Ojeda 和 Danilo Krummrich 审查。

📐 DeviceSize trait：统一设备地址空间常量（v3 & v4）

今天有两个版本（v3 和 v4）的 DeviceSize trait 补丁被提交。这个 trait 的目的是为设备地址空间的大小常量（比如 SZ_4K、SZ_1M 等，代表 4KB、1MB 这样的大小）提供一个统一的 Rust 接口。在内核中，这些常量到处都在用——描述内存映射区域的大小、DMA 缓冲区的大小等等。v3 由 Alice Ryhl 提出并获得了 John Hubbard、Miguel Ojeda 等 6 位审查者的审查；v4 由 John Hubbard（来自 NVIDIA）在 v3 基础上改进。两个版本的并行讨论说明社区在积极打磨这个 API 的设计细节。

📬 查看 v3 补丁 📬 查看 v4 补丁
🔄 TryFrom 和 Into 派生宏 v5

Alexandre Courbot（NVIDIA）提交了第 5 个版本的补丁，为内核 Rust 代码添加 TryFrom 和 Into 的派生宏（derive macros）。在 Rust 中，TryFrom 和 Into 是用于类型转换的标准 trait——比如把一个整数转成枚举值（可能失败，所以用 TryFrom），或者把枚举值转回整数（一定成功，所以用 Into）。有了派生宏，开发者只需要在类型定义上加一个 #[derive(TryFrom, Into)] 注解，编译器就会自动生成转换代码，不用手写一大堆模板代码。这个补丁已经迭代到第 5 个版本了，说明社区对 API 设计非常认真。包含 4 个子补丁，有 3 位审查者。

🛡️ Software Tag-Based KASAN 支持 v2

Alice Ryhl 和 Miguel Ojeda 正在讨论为 Rust 代码添加 Software Tag-Based KASAN 的支持。KASAN（Kernel Address SANitizer）是内核中的一个内存错误检测工具，可以在运行时发现越界访问、使用已释放内存等 bug。"Software Tag-Based"是 KASAN 的一种模式，通过给每块内存分配一个"标签"来检测非法访问——就像给每个停车位贴上车牌号，如果有车停错了位就能立刻发现。让 Rust 代码也支持这个工具，意味着即使 Rust 的安全保证已经很好了，在 unsafe 代码块中仍然能获得额外的安全检测。

David Rheinsberg 提交了一个文档改进补丁，在 Box::from_raw() 的文档中明确提到了内存布局（layout）的要求。Box::from_raw() 是一个把原始指针"重新包装"成 Box 智能指针的函数——但有个关键前提：你传入的指针必须是用相同的内存布局（对齐方式和大小）分配的。如果搞错了，轻则浪费内存，重则导致内存分配器崩溃。这个文档改进虽然不改功能代码，但对新手来说极其重要，因为这是最容易踩坑的地方之一。Danilo Krummrich 参与了审查。

设备驱动与子系统

🔧 ACPI：修复 PRP0001 设备的匹配数据丢失问题

Markus Probst 提交了一个重要的 bug 修复。问题是这样的：在 ACPI（高级配置与电源接口——你笔记本电脑能休眠、能管理电池、能检测到插入的设备，都靠它）系统中，有一种叫 PRP0001 的特殊设备标识，它允许 ACPI 设备使用 Device Tree 风格的匹配方式。但在 Rust 驱动中，匹配这类设备时 id_info（设备身份信息）总是返回 None（空），导致驱动无法正确识别设备。

修复方法是导出 C 内核中的 acpi_of_match_device 函数，让 Rust 代码能调用它来正确匹配设备。补丁还解释了为什么不能简单使用 device_get_match_data——因为 Rust 侧存储的是索引值而不是数据指针，两者机制不兼容。Greg Kroah-Hartman 和 Danilo Krummrich 参与了审查。

📬 查看补丁
⏰ 工作队列：添加安全的取消和状态检查功能

Aakash Bollineni 提交了一个补丁，为 Rust 的工作队列（workqueue）抽象添加安全的取消操作和状态检查功能。工作队列是内核中的一种"任务排队"机制——你可以把一些不需要立即完成的工作"排上日程"，内核会在合适的时候去执行。但有时候你需要取消一个已经排队但还没执行的任务，或者查看一个任务的状态（是在排队中？正在执行？已完成？）。这个补丁提供了 Rust 安全接口来做这些操作，避免了直接调用不安全的 C 函数。Miguel Ojeda 参与了审查。

🌲 RCU Trie：新的并发数据结构 RFC

在 Zulip 的 IRC 桥接频道中，有人分享了一个关于 RCU Trie（基于 RCU 机制的 Trie 字典树）的 RFC。RCU（Read-Copy-Update）是 Linux 内核中一种高效的读写同步机制——当多个 CPU 核心需要同时读取同一个数据结构时，RCU 允许"读操作几乎零开销"，写操作则通过复制-修改-替换的方式来避免阻塞读者。在 RCU 之上构建 Trie 数据结构，可以实现非常高效的并发查找。虽然这个 RFC 本身是 C 代码，但它对未来 Rust 抽象的设计有参考意义。
