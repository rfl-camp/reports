️ GPU / DRM 驱动（Nova-Core & Asahi）

📦 GSP 固件通信：提取可用显存区域（v9，23 个补丁）

今天 GPU 驱动方向是最热闹的板块。Nova-core 是社区正在用 Rust 从零开始编写的 NVIDIA 开源 GPU 驱动，目标是替代现有的 C 语言驱动 nouveau。今天有大量补丁在推进它的核心功能。

这是一个大型补丁系列，由 Joel Fernandes 和 Eliot Courtney 共同开发。NVIDIA 显卡内部有一个叫 GSP（GPU System Processor）的微处理器，负责管理 GPU 的各种底层功能。这组补丁的核心工作是：让驱动能从 GSP 固件那里"问出"显卡上哪些显存（Frame Buffer）区域是可以使用的。这就像你搬进一间新房子，得先弄清楚哪些房间能住人、哪些被锁住了。同时，这个系列还暴露了物理显存的总大小信息，让驱动能更好地规划显存的使用。

📦 内存管理：TLB 刷新支持（v10，21 个补丁）+ 页表条目封装

Eliot Courtney 提交的这组大型补丁，为 nova-core 的内存管理子系统添加了 TLB 刷新功能。TLB（Translation Lookaside Buffer）是 GPU 里的一个高速缓存，用来加速虚拟地址到物理地址的翻译。当页表（记录地址映射关系的"通讯录"）发生变化时，必须通知 GPU 去刷新 TLB，否则 GPU 可能会读到错误的地址——就像你朋友换了手机号，但你手机通讯录里还是旧号码。此外，还有一组补丁为页表条目（PTE）设计了统一的封装枚举类型，让代码在处理不同类型的页表映射时更加类型安全。

📬 查看补丁（TLB 刷新 v10） 📬 查看补丁（页表条目封装 v10）
🐛 GSP 代码内存屏障修复

Gary Guo 发现 nova-core 中 GSP 与 CPU 通信路径上的内存屏障（memory barrier）使用有误。具体来说，在 GSP 向 CPU 发消息的路径中，代码在读取数据之前缺少了一个读屏障，而且读取之后使用的屏障类型也不对——原来用的是 Rust 的 SeqCst（最强排序），被修正为更合适的 DMA 屏障。这类 bug 非常隐蔽，因为在大多数情况下代码看起来能正常工作，但在特定的硬件时序下可能导致读到过期数据，就像你收到了一封信，但里面的内容其实是上一封信的。

📦 SizeConstants trait 与 DeviceSize trait

Joel Fernandes 提交了一个补丁，让 nova-core 使用 SizeConstants trait 来定义 u64 类型的大小常量。Alexandre Courbot 则提交了 DeviceSize trait（v3），专门用于设备地址空间的常量定义。这两个补丁的目的是一样的：让代码中表示"大小"的常量（比如 1GB、4KB 这些）有统一的接口，避免到处硬编码魔法数字，也让不同模块之间共享这些常量更方便。

🐛 Bitfield 默认实现修复

Eliot Courtney 修复了 nova-core 位域（bitfield）模块中 Default trait 实现的一个 bug。位域是一种节省内存的技术，把多个小数据打包存储在一个整数的不同二进制位上。这个 bug 导致位域的默认值生成不正确，v2 版本修复了这个问题。

📦 DRM shmem-helper：添加 lockdep 断言 + Asahi GEM SHMEM 进展

Bence Csókás 为 DRM 子系统的 shmem-helper（共享内存辅助模块）中的 vmap/vunmap 操作添加了 lockdep 断言。lockdep 是 Linux 内核自带的锁依赖检测工具，能在运行时检查你有没有按正确的顺序获取锁，防止死锁。这组补丁确保在映射/解映射共享内存时，调用者已经正确持有了必要的锁。

在 Zulip 的 DRM 频道，Asahi Linux（苹果 M 系列芯片的 Linux 驱动）的开发者 Lyude 表示，GEM SHMEM 相关的改动大部分已经进入上游，目前只剩 SGTable/VMap 部分还没有，她计划明天将这部分补丁发到邮件列表。

⚙️ 工作队列（WORKQUEUE）

⚙️ 工作队列（Workqueue）

📦 新增安全取消与状态检查方法

Onur Özkan 和 Aakash Bollineni 联合提交了一组共 3 个补丁，为 Rust 的工作队列基础设施添加了安全取消（safe cancellation）和状态检查方法。工作队列是内核中一种"把任务丢到后台执行"的机制——比如驱动程序不想在中断处理中做太多事情，就把耗时任务放到工作队列里让内核稍后去执行。新增的取消方法对驱动的安全卸载至关重要：当设备被移除时（比如你拔掉了一个 USB 设备），驱动需要确保所有后台任务都已停止，然后才能释放资源。如果不做这一步，就像你在搬家公司还在搬东西的时候就把房门锁了——结果会很糟糕。这组补丁让驱动可以在 Drop 实现中安全地清理工作队列任务。

📬 查看补丁
🧪 新增 KUnit 测试和压力测试

配合上面的功能补丁，Aakash Bollineni 还提交了一组完整的测试基础设施，包括：（1）KUnit 测试——在 rust/kernel/workqueue.rs 内部添加了名为 rust_kernel_workqueue 的测试套件，用于验证入队、取消、状态检查等功能的正确性；（2）压力测试——模拟高并发场景，确保工作队列在大量任务同时操作时不会出问题。有了这些测试，后续开发者修改工作队列代码时就能快速发现是否引入了 bug。

💬 Zulip 讨论：能否按值入队工作项？

在 Zulip 的 Help 频道，Philipp Stanner 向 Alice Ryhl 提问：通过 workqueue::system().enqueue(foo) 入队的每个任务是否都必须实现 Send 和 Sync trait？能不能按值（by value）传递？Onur Özkan 查看了实现代码后回复：目前的设计确实要求工作项实现 Send + 'static，如果你的类型没有实现 Send，暂时无法按值传递。这是因为工作队列会在不同线程上执行任务，Rust 的 Send trait 正是用来保证"这个类型可以安全地跨线程传递"。

📦 支持覆盖 crate_name + Binder 驱动命名修复（v4）

Alice Ryhl 提交了一组两部分的补丁来解决一个令人困扰的命名问题。Rust Binder 驱动编译后的目标文件叫 rust_binder_main.o，因为 rust_binder.o 这个名字已经被链接最终产物占用了。但问题是，crate 的名字也因此变成了 rust_binder_main，这导致内核 panic 时的堆栈跟踪、符号表中出现的名字都带着多余的 _main 后缀，看起来很不专业也不直观。补丁分两步走：第一步修改构建系统，让 RUSTFLAGS_REMOVE 能正确过滤 --crate-name 参数，使得开发者可以覆盖自动生成的 crate 名；第二步在 Binder 驱动中使用 #![crate_name] 注解，将名字设置回正确的 rust_binder。Miguel Ojeda、Gary Guo、Greg Kroah-Hartman 等核心维护者都参与了审核。

📦 通用内存屏障接口

Gary Guo 提交了 3 个补丁，为 Rust 内核代码实现了一套通用的内存屏障接口。这套接口支持三种屏障作用域：完整系统屏障、DMA 屏障和 SMP（对称多处理器）屏障，每种都提供 Read（读屏障）、Write（写屏障）和 Full（全屏障）三种排序方式。它们分别映射到 C 内核中的 rmb()、wmb() 和 mb() 等宏。设计上的一个亮点是：接口通过参数强制让开发者明确表达"我想要什么类型的屏障"，而不是像以前那样可能随意选一个——这利用了 Rust 的类型系统来降低出错概率。这个补丁与上面 GSP 屏障修复是配套的。

📬 查看补丁
📦 新增 TryFrom 和 Into 派生宏（v5）

Jesung Yang 提交了 4 个补丁，为内核 Rust 代码添加了 TryFrom 和 Into 的派生宏（derive macro）。在 Rust 中，TryFrom 和 Into 是用于类型转换的标准 trait。有了派生宏，开发者只需在类型定义上加一行 #[derive(TryFrom, Into)]，编译器就会自动生成转换代码，不用手写那些重复的样板代码。Alexandre Courbot（NVIDIA）对此进行了审核。这对 nova-core 等大量使用枚举类型和整数转换的驱动特别有用。

⚡ CPU 调频（cpufreq）

📦 amd-pstate：将策略对象传递给 amd_pstate_update()（v4）

Mario Limonciello 提交了 2 个补丁，修改了 AMD CPU 频率调节驱动 amd-pstate。具体改动是将 cpufreq_policy（调频策略）对象直接传递给 amd_pstate_update() 函数，而不是让函数自己去获取。这是一个常见的代码改进模式——通过"依赖注入"让函数的输入更明确、更容易测试。Viresh Kumar（cpufreq 子系统的资深维护者）参与了审核，目前已经到了第 4 个版本。虽然这个补丁本身是 C 代码，但它出现在 Rust-for-Linux 邮件列表上，说明未来 cpufreq 子系统可能也会有 Rust 绑定的计划。
