️ GPU / Nova-Core 驱动

1. 修复命令队列中的未定义行为（UB）— 第 5 版

今天 nova-core（NVIDIA 开源 GPU 驱动的 Rust 实现）是绝对的主角，围绕它的补丁多达十余个。我们一个一个来看。

这是 Alexandre Courbot（NVIDIA 工程师）提交的补丁，已经迭代到第 5 版了，可见修复难度不小。问题出在 driver_read_area 和 driver_write_area 这两个内部方法上——它们返回的是命令队列缓冲区里驱动可以独占读写的那一段切片（slice）。虽然最终返回的值是正确且安全的，但在内部实现过程中，代码临时创建了一个引用，而这个引用的创建方式违反了 Rust 的安全规则，属于"未定义行为"。打个比方：就像你搬家时把所有东西都安全送到了新家，但搬运过程中你违反了交通规则——结果是对的，过程是危险的。这个补丁就是修正这个"搬运过程"。经过 John Hubbard、Gary Guo 等 6 位审查者的检查，目前已经到了 v5。

2. 回退之前的命令队列 UB 修复

有趣的是，上面那个修复的早期版本（已经合并的那个）被发现带来了新问题，所以 Miguel Ojeda 和 John Hubbard 提交了一个 Revert（回退）补丁。原因是：早期版本使用了 ptr::project!() 宏对运行时从 DMA 一致性内存中读取的值进行"不可失败的范围索引"。当编译器无法在编译时证明索引一定在范围内时，build_error!() 回退机制就会保留到最终二进制中，导致构建错误。简单说：修复 A 引入了问题 B，所以先撤掉 A，等 v5 版的修复来替代。这在内核开发中很常见——安全第一，宁可退一步也不要带着隐患前进。

📬 查看补丁
3. 修复 Clippy 警告：运算符优先级问题

John Hubbard 提交了多个补丁来消除 Clippy（Rust 的代码质量检查工具）发出的 clippy::precedence 警告。问题出在位移运算符 << 和按位或 | 混用时没有加括号——虽然代码逻辑是对的，但人类读起来很容易误判优先级。修复方式很直白：在 read_sysmem_flush_page_ga100() 函数里给运算加上显式括号；在 PCI ROM 头部解析代码中，用更清晰的写法替代手动字节拼接。这类补丁虽然不改变任何功能，但能让代码更易读、更不容易被未来的开发者误改。在 drm-rust-next 分支中一共消除了 3 个这样的警告。

4. Falcon 固件 DMA 对象大小对齐修复

这个补丁由 Alexandre Courbot 提交，修复了一个因为类型替换引发的"对齐"问题。之前的提交把 nova-core 本地的 DmaObject 类型换成了内核全局的 dma::Coherent 类型，结果发现一个副作用：旧的 DmaObject 会自动把请求的大小向上对齐到 PAGE_SIZE（通常是 4KB），并且在查询时报告对齐后的大小；而新的 Coherent 不做这个处理。Falcon 微控制器（GPU 内部的一个小处理器）加载固件时需要固件映像的大小满足特定的块对齐要求，不对齐就会出问题。这个补丁确保在分配 DMA 内存时，大小被正确地向上取整到所需的对齐边界。

📬 查看补丁
5. 修复 Bitfield 的 Default trait 实现

Danilo Krummrich 提交了一个补丁，修复 nova-core 中 bitfield（位域）模块的 Default trait 实现。Default 是 Rust 中的一个标准特征，用来给类型提供一个"默认值"。如果这个实现有 bug，那么每次创建一个新的 bitfield 默认实例时都可能拿到错误的初始值，进而导致硬件寄存器被写入错误的配置。虽然补丁描述很简短，但这类底层数据结构的修复往往影响面很广。

📬 查看补丁
6. SizeConstants trait：让设备地址空间常量更好用（v5，3 个补丁）

这是 John Hubbard 提交的一组补丁（3 个），已经迭代到第 5 版。内核中有一组 SZ_* 常量（比如 SZ_1M 表示 1MB），它们的类型是 usize，匹配 CPU 的指针宽度。但 GPU、MMIO 窗口等设备有自己的地址空间宽度（32 位或 64 位），驱动里经常要写 SZ_1M as u64 这样的类型转换，既啰嗦又没有安全收益。这组补丁引入了一个新的 SizeConstants trait，让这些常量可以自然地用于不同宽度的整数类型。第三个补丁还为后续添加"设备地址类型包装器"预留了任务说明。这是一个让 Rust 驱动代码写起来更优雅的基础设施改进。

RUST 内核构建系统与基础设施

Rust 内核构建系统与基础设施

1. 修复"多个 rmeta 依赖候选"构建错误（V9，4 个补丁）

这是 Mukesh Kumar Chaurasiya（来自 IBM）提交的补丁系列，解决的是一个让人头疼的构建问题。当使用 LLVM=1（即用 LLVM/Clang 替代 GCC 来编译内核）并且只用单线程编译（-j1）时，rustc 会报错说找到了两个 core 库的候选——一个是系统 toolchain 自带的标准库，另一个是内核自己编译的 libcore.rmeta。这就像你的手机同时安装了两个版本的微信，系统不知道该打开哪个。这个补丁系列通过调整构建脚本中的搜索路径逻辑来消除歧义，确保 rustc 只找到内核版本的 core 库。补丁已经到了 V9，说明这个问题的边界条件很多，修复需要反复打磨。

📬 查看补丁
2. PowerPC 架构的 Rust 支持（V9，4 个补丁）

同样来自 Mukesh Kumar Chaurasiya，这组补丁为 PowerPC 架构添加 Rust 支持。PowerPC 是 IBM 的处理器架构，在服务器和嵌入式领域仍有广泛使用。目前内核的 Rust 支持主要集中在 x86_64 和 ARM64 上，添加 PowerPC 支持意味着 Rust 的"领地"在不断扩展。这个补丁和上面的 rmeta 修复是同一个系列的一部分——因为 PowerPC 的构建环境正是暴露那个 rmeta 问题的场景之一。

3. 允许 Clippy 的 MSRV 不兼容警告

Miguel Ojeda 和 John Hubbard 一起修了一个 Clippy 在 CLIPPY=1 模式下报出的警告。内核声明的最低支持 Rust 版本（MSRV）是 1.78.0，但代码中用到了 slice_ptr_len 这个方法，而它是在 Rust 1.79.0 才稳定的。Clippy 很负责任地指出了这一点。但在内核环境中，实际使用的编译器版本是高于 1.79.0 的，所以这个警告是"假警报"。补丁通过添加 #[allow(clippy::incompatible_msrv)] 注解来消除这些警告，涉及的文件是 rust/kernel/ptr/projection.rs。

📬 查看补丁
4. Prelude 模块改用"垂直导入"风格

Miguel Ojeda 提交了一个代码风格补丁，把 rust/kernel/prelude.rs（内核 Rust 代码的"前置导入"模块）中的 import 语句改成了"kernel vertical"风格——也就是每个导入项单独占一行，而不是挤在一行里用逗号分隔。这看起来是小事，但对于一个有很多贡献者的项目来说，统一的代码风格能大大减少合并冲突和代码审查时的认知负担。prelude 是几乎所有内核 Rust 模块都会导入的文件，所以这个改动的可见度很高。

设备驱动与子系统

设备驱动与子系统

1. ACPI：修复 PRP0001 设备的匹配数据缺失

Markus Probst 提交了一个修复，解决 Rust ACPI 绑定中 PRP0001 设备的匹配数据（match data）缺失问题。PRP0001 是 ACPI 规范中一个特殊的设备标识符，它表示"这个设备虽然通过 ACPI 枚举，但应该用 Device Tree 风格的 compatible 属性来匹配驱动"。如果匹配数据缺失，Rust 驱动就无法正确识别和绑定到这类设备。Gary Guo 和 Danilo Krummrich 等人参与了审查。这个修复对于在服务器平台上运行 Rust 驱动尤为重要，因为 ACPI 是 x86 服务器的标准硬件描述方式。

📬 查看补丁
2. DMA：大小溢出时返回更准确的错误码

Danilo Krummrich 和 Gary Guo 修复了 Rust DMA 模块中的一个错误码问题。之前，当计算 DMA 缓冲区大小时发生整数溢出，代码返回的是 ENOMEM（内存不足），但更准确的语义应该是 EOVERFLOW（数值溢出）。虽然对用户来说两个错误最终都意味着"分配失败"，但正确的错误码能帮助开发者更快定位问题——"内存不足"和"你请求的大小大到溢出了"是完全不同的诊断方向。这就像医生区分"发烧"和"过敏"——症状可能相似，但病因和治疗方法不同。

3. I2C 子系统：新增 Rust 开发仓库

Wolfram Sang（I2C 子系统维护者）和 Igor Korotin 更新了 MAINTAINERS 文件，为 Rust I2C 开发添加了专门的 git 代码仓库，并更新了 Igor 的邮箱地址。这个仓库将用于收集 I2C Rust 驱动相关的补丁，并为集成测试（包括 linux-next）提供基础。I2C 是一种极其常见的硬件通信总线——你手机里的传感器、笔记本的触摸板、很多嵌入式设备都在用它。有了专门的 Rust 开发仓库，意味着 I2C Rust 驱动的开发有了"正式编制"，补丁可以通过正规渠道进入主线。

4. seq_file：新增多个实用方法

Christian Benton 为 Rust 的 SeqFile 抽象新增了好几个方法：puts()（写入 C 字符串）、putc()（写入单个字节）、write()（写入字节切片）和 hex_dump()（十六进制转储）。之前 SeqFile 只暴露了 seq_printf（通过 seq_print! 宏），这让很多常用操作没法在 Rust 代码里直接调用。seq_file 是内核中 /proc 文件系统输出信息的标准方式——比如你用 cat /proc/cpuinfo 看到的内容，底层就是通过 seq_file 生成的。有了这些新方法，Rust 驱动在向 /proc 输出调试信息时就方便多了。

📬 查看补丁
5. Binder：修复持锁时释放对象的问题

Alice Ryhl（Google 的内核 Rust 开发者）修复了 rust_binder（Android Binder 的 Rust 实现）中的一个问题：在释放 delivered_death 对象时代码仍然持有锁。在并发编程中，持有锁时执行可能耗时的操作（如内存释放）是一个经典的反模式——它会阻塞其他需要这把锁的线程，降低系统响应速度，严重时甚至导致死锁。这个修复确保在释放锁之后再 drop 对象。

社区讨论与求助

社区讨论与求助

1. KUnit 测试时如何设置 kconfig 选项？

在 Zulip 的 #Help 频道，知名 DRM 开发者 Lyude（Red Hat）和项目维护者 Miguel Ojeda 展开了一场 17 条消息的讨论。Lyude 在尝试运行 KUnit 测试时遇到了麻烦——她成功设置了需要的 kconfig 选项，但 CONFIG_RUST 却莫名其妙地被设为 n（关闭），而且没有明确的错误提示告诉她为什么。Miguel 建议使用 menuconfig 的搜索功能（按 / 键）来查看 CONFIG_RUST 的 depends on 条件，逐个检查哪个前置条件没有满足。这个问题其实很多新手都会遇到——kconfig 的依赖关系像一张网，一个选项依赖另一个，而那个又依赖更多，很容易迷路。

💬 查看讨论
2. C-to-Rust 自动翻译工具寻求社区反馈

一位名为 wenzhao liao 的开发者在 #Help 频道介绍了自己正在开发的工具项目——目标是把 Linux 内核的 C 驱动自动翻译成"健全的、可供审查的"Rust 代码，并尽量减少 unsafe 的使用。Miguel Ojeda 很感兴趣地问道：这是希望找到愿意花时间审查 AI 生成代码的维护者，还是有其他目标？wenzhao 澄清说，虽然工具链中确实用到了 AI 辅助，但最终输出的代码是经过验证的、符合 Rust 安全规范的，他们希望找到合适的驱动目标来做概念验证。这个方向很有前景，但也面临不少挑战——自动翻译的代码是否真正"地道"、维护者是否有精力审查、如何保证语义等价，都是待解决的问题。

3. Asahi DRM 开发被 kconfig 卡住

还是 Lyude，在 #DRM 频道抱怨她的 Asahi GEM SHMEM（Apple Silicon GPU 驱动的共享内存管理）开发工作被 kconfig 问题卡住了——CONFIG_RUST 在 KUnit 环境下又自动变成了 n，导致无法运行测试。她感叹道"我们真的需要一个不那么糟糕的 kconfig"。这反映了 Rust-for-Linux 社区面临的一个实际痛点：现有的内核配置系统对 Rust 的支持还不够顺畅，开发者经常在配置环节浪费大量时间。
