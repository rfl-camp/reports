️ GPU 驱动 · Nova-Core（NVIDIA 开源驱动）

1. 命令队列（Cmdq）加锁改造 v5

今天 nova-core 方向有大量活动，几乎涵盖了驱动的各个子系统。nova-core 是用 Rust 为 NVIDIA GPU 编写的全新开源驱动，目标是替代现有的 nouveau 驱动。下面逐一介绍：

Eliot Courtney 提交了一组 5 个补丁，给 GPU 的"命令队列"加上了互斥锁（Mutex）。你可以把命令队列想象成 CPU 和 GPU 之间传话的邮箱——CPU 把指令放进去，GPU 去取。问题是：当驱动卸载（比如你拔掉显卡或卸载驱动模块）时，需要从另一个线程发送"我要走了"的消息，但如果这时候还有别的命令在排队，就会出乱子。这组补丁把命令队列的内部状态抽成了一个新的 CmdqInner 结构体，再用 Mutex 包起来，这样方法签名就能从 &mut self（独占访问）变成 &self（共享访问），多个线程可以安全地使用同一个命令队列了。此外还清理了过时的文档注释、添加了接收超时常量 RECEIVE_TIMEOUT、区分了需要回复和不需要回复的命令类型，并把 Cmdq 改成了 pinned 类型以确保内存安全。

2. RM 控制命令基础设施 v2

Eliot Courtney 和 Danilo Krummrich 合作提交了 9 个补丁，搭建了"RM 控制命令"的基础框架。RM（Resource Manager）是 NVIDIA GPU 固件里的"管家"，负责管理 GPU 的各种资源。驱动要和 GPU 做任何复杂的事（比如分配显存通道），都需要通过 RM 控制命令以 RPC（远程过程调用）的方式和固件对话。这组补丁新增了 NV_STATUS 的 Rust 绑定和 NvStatus 枚举（用来表示操作成功/失败的状态码），添加了 RM 控制 RPC 所需的底层绑定，以及 RmControlMsgFunction 来标识不同的控制命令。简单说，这相当于给驱动和 GPU 固件之间架了一条"电话线"，后续的通道分配等高级功能都得靠它。

📬 查看补丁系列
3. PMC 寄存器改用内核 register 宏

Alexandre Courbot 提交了 8 个补丁，把 nova-core 里所有的 PMC（Power Management Controller，电源管理控制器）寄存器定义从自定义写法迁移到了内核统一的 register! 宏。寄存器就像是硬件上的"开关"，你往特定地址写入特定值，硬件就会执行对应操作。之前 nova-core 自己定义了一套寄存器写法，现在统一用内核提供的宏来做，好处是代码更简洁、和内核其他部分风格一致。另外还引入了一个 nv_reg 辅助宏，因为 nova-core 的寄存器都有一些固定属性（32 位宽度、crate 可见性），用辅助宏可以省掉重复代码。Gary Guo 参与了审查。

4. MCTP/NVDM 固件通信协议类型 v7

这是一个包含 31 个补丁的大系列（v7 版本），由 Gary Guo 和 John Hubbard 推进。MCTP（Management Component Transport Protocol）是一种硬件管理通信协议，NVDM 是 NVIDIA 在此基础上定义的数据消息格式。这些类型定义是 GPU 和其内部固件（GSP）之间通信的"语言规范"——就像两个人对话之前要先约定好说中文还是英文。这组补丁定义了各种消息类型和数据结构，为后续实际的固件通信打下基础。

📬 查看补丁
5. Hopper/Blackwell 架构：更大的 GSP 堆和 FSP 启动路径 v7

John Hubbard 在同一个 v7 大系列中还有两个重要改动：一是为 Hopper 和 Blackwell 架构（NVIDIA 最新的 GPU 架构）增大了 WPR2 区域的 GSP 堆内存。GSP（GPU System Processor）是 GPU 内部的一个小处理器，负责运行固件，WPR2 是一块受保护的内存区域——新架构的固件更大更复杂，所以需要更多"工作空间"。二是把 FSP（Falcon Security Processor，另一个安全处理器）的启动路径集成到了统一的 boot() 函数中，让启动流程更清晰。内核测试机器人（kernel test robot）也参与了自动化测试和反馈。

📬 WPR2 堆补丁 📬 FSP 启动路径补丁
6. 泛化 flush_into_kvec 为 flush_into_vec

Alexandre Courbot 提交了 9 个补丁，把原来只能往 KVec（内核向量）里刷数据的 flush_into_kvec 方法泛化成了 flush_into_vec，让它能支持更多类型的容器。这是一个典型的 Rust 泛型重构——把"只能往特定杯子里倒水"改成"能往任何杯子里倒水"，让代码更灵活、复用性更好。

3. DRM 依赖队列层 RFC 讨论

Matthew Brost 提出了一个 RFC（征求意见稿）补丁，要在 DRM 子系统中添加一个"依赖队列层"（dependency queue layer）。在 GPU 编程中，很多操作之间有先后依赖关系——比如"先渲染完这一帧，再显示到屏幕上"。这个依赖队列就是用来管理这些先后顺序的。目前已经有 2 条审查意见，讨论还在进行中。这是一个架构层面的改动，会影响所有使用 DRM 子系统的驱动。

RUST 核心基础设施

Rust 核心基础设施

1. 链表模块 v13：C 链表的 Rust 接口

Miguel Ojeda、Gary Guo、Danilo Krummrich 等人的链表模块来到了第 13 版，获得了 20 条审查意见——这是一个备受关注的补丁。Linux 内核大量使用 C 语言的链表（list_head），几乎每个子系统都离不开它。这个模块的目标是提供一个安全的 Rust 接口来操作这些 C 链表，让 Rust 代码能和现有的 C 子系统无缝协作。迭代了 13 个版本说明这个接口的设计很有挑战——既要保证 Rust 的内存安全，又要兼容 C 链表"侵入式"（intrusive）的设计哲学。20 条审查意见说明社区在非常认真地打磨这个基础组件。

📬 查看补丁
2. register! 宏 v9

Alexandre Courbot 的 register! 宏更新到了 v9，包含 10 个补丁。这个宏用来声明硬件寄存器——想象一下，每个硬件设备都有很多"旋钮和开关"（寄存器），驱动需要读写它们来控制硬件。以前每个驱动各写各的寄存器定义，风格不统一也容易出错。这个宏提供了一种统一、类型安全的方式来定义寄存器，确保你不会不小心往只读寄存器写数据、或者写入超出范围的值。前面提到的 nova-core 和 Tyr 驱动都在用它。

📬 查看补丁
3. pin-init 的 pinnable Option 讨论

Benno Lossin 在邮件列表和 Zulip 上同时发起了一个有趣的技术讨论：能不能让 Option 类型和 pin-init 兼容？具体场景是，有时候你想在一个结构体里放一个 Option<Mutex<T>>——也就是"可能有也可能没有"的互斥锁。但 Mutex 是 pinned 类型（被"钉"在内存中不能移动），这和 Option 的"有/无"切换机制有冲突。Gary Guo 提出了一个巧妙但有点"黑魔法"的思路：利用 MaybeUninit 和内存布局的兼容性来绕过这个问题。Benno 追问了关键问题：如何在不移动数据的前提下安全初始化 Option<MaybeUninit<T>>？讨论还在继续，没有最终结论。

块设备与存储

1. rnull 块设备驱动适配 PinInit v4

Andreas Hindborg 和 Benno Lossin 更新了 rnull（一个用 Rust 写的"空"块设备驱动，类似 /dev/null 但用于块设备层）。这个补丁把 QueueData 的类型从普通的 KBox 改成了 Pin<KBox<QueueData>>，以适配新版的 PinInit API。简单说就是：底层的初始化框架升级了，上层的驱动也要跟着适配新接口。rnull 虽然本身不做什么"有用"的事，但它是 Rust 块设备驱动的参考实现和测试用例，其他块设备驱动会参考它的写法。

这是一条值得关注的社区动态。Andreas Hindborg 在 Zulip 上通知 SeungJong Ha：他可能不会再发布 Rust NVMe 驱动（rnvme）的新 rebase 了。NVMe 是现代 SSD 使用的高速存储协议，rnvme 是用 Rust 重写其驱动的实验性项目。Andreas 指向了 LKML（Linux 内核邮件列表）上的一个讨论帖来解释原因。SeungJong Ha 表示理解，并承诺会继续自己维护这个驱动。这标志着 rnvme 从"个人项目"转向"社区协作"的新阶段——这在开源社区是很常见也很健康的交接模式。

其他动态

其他动态

1. DEPT 依赖跟踪器 v18

Yunseong Kim 提交了 DEPT（DEPendency Tracker）的第 18 版，包含 42 个补丁——这是一个大工程。DEPT 是一个内核锁依赖跟踪工具，用来检测死锁（两个线程互相等对方释放锁，结果谁也动不了）。你可以把它想象成交通管理系统中的"死锁检测器"——当两条路的车互相堵住时，它能提前发现并报警。虽然这个工具不是专门为 Rust 设计的，但 Rust 驱动也会使用内核的锁机制，所以 DEPT 对整个生态都有帮助。迭代到 v18 说明这个工具的复杂度很高，社区在逐步完善。

📬 查看补丁
2. MAINTAINERS 文件更新：GPU Buddy 分配器审查者变更

Christian König 和 Matthew Auld 提交了 2 个补丁更新 MAINTAINERS 文件，变更了 GPU buddy 分配器的审查者信息。MAINTAINERS 文件是 Linux 内核的"通讯录"，记录了每个子系统由谁负责维护和审查。Buddy 分配器是一种经典的内存分配算法（把内存分成大小为 2 的幂次的块来管理），GPU 子系统有自己的 buddy 分配器实现。审查者变更意味着这部分代码的"把关人"换了。

3. 模块/驱动作用域数据讨论

Boqun Feng 在 Zulip General 频道继续了关于"模块/驱动作用域数据"的讨论，提到了 hazard pointer（危险指针）相关的异步操作和避免忙等待的设计。这涉及到内核中如何安全地管理驱动和模块级别的共享数据——在多线程环境下，数据什么时候可以安全释放是一个很有挑战的问题。Hazard pointer 是一种无锁并发数据结构的保护机制，正在被探索用于 Rust 内核代码中。
