️ PowerPC 架构支持（V7 补丁集）

补丁 1：总览 — 启用 ppc64le 和 ppc32be 实验性 Rust 支持

今天最引人注目的工作是来自 IBM 的 Mukesh Kumar Chaurasiya 提交的 PowerPC Rust 支持补丁集，现在已经迭代到了第 7 版。这个补丁集一共包含 4 个补丁，目标是让 Linux 内核的 Rust 代码能在 PowerPC 处理器上运行。

你可以把 PowerPC 理解为一种和我们日常用的 x86（Intel/AMD）不同的 CPU 架构。它曾广泛用于苹果电脑、IBM 服务器，甚至任天堂 Wii 游戏机。现在 Rust-for-Linux 项目想让 Rust 代码能在这些机器上运行。

这个补丁是整个系列的封面信，说明了要同时支持两种 PowerPC 变体：ppc64le（64 位小端序，现代 IBM 服务器常用）和 ppc32be（32 位大端序，老设备如 Wii/GameCube 使用）。其中 ppc32 的支持由社区成员 Link Mauve 贡献，他在任天堂 Wii 上做了测试。ppc32 还需要一些工具链修复，正在讨论中。

补丁 2：修复 Makefile 竞态条件

这是一个有趣的 bug 修复。当你用 -j1（单线程编译）在 PowerPC 上编译内核时，构建系统会发现两个 libcore.rmeta 文件——一个来自 Rust 工具链自带的，一个来自内核本地编译目录的。这就像你家里有两把长得一模一样的钥匙，程序搞不清该用哪把了。修复方法是让 libproc_macro2、libquote 和 libsyn 这几个构建依赖项等待 core.o 编译完成，确保使用正确的工具链版本。这个补丁得到了 Gary Guo 和 Miguel Ojeda 的审查。

补丁 3：统一 jump_label 内联汇编

这个补丁引入了一个新宏 ARCH_STATIC_BRANCH_ASM，目的是让 PowerPC 上的 jump label（跳转标签）功能在 C 和 Rust 之间共享同一份内联汇编代码，避免重复维护两份。jump label 是内核中一种性能优化技巧：它可以在运行时动态修改代码中的跳转指令，比普通的 if 判断更快。这个补丁保持了与之前 C 语言侧提交的一致性。

构建系统（KBUILD）

构建系统（Kbuild）

C 辅助函数内联到 Rust（已合入 rust-next！）

这是今天一个重要的已合入进展！Gary Guo 的提交为内核构建系统添加了一个新的实验性配置选项 RUST_INLINE_HELPERS。

背景是这样的：Linux 内核中有很多用 C 语言写的"内联函数"和宏，它们非常小但被频繁调用。Rust 代码想调用这些函数时，通常需要通过一层"辅助函数"（helper）来桥接，这会带来额外的函数调用开销。新的选项允许编译器把这些 C 辅助函数直接"内联"到 Rust 代码里，就像把一小段 C 代码直接复制粘贴到 Rust 函数内部一样，省去了跳转调用的成本。而且这不需要开启完整的 LTO（链接时优化），大大降低了编译负担。这对性能敏感的内核代码来说非常重要。

from_err_ptr 文档澄清（V2）

Mirko Adžić 提交了对 from_err_ptr 函数文档的改进（第 2 版）。这个函数用于把 C 语言返回的"错误指针"转换为 Rust 的 Result 类型。之前的文档没有说清楚一个重要细节：如果传入一个空指针（NULL），函数会返回 Ok(NULL) 而不是报错——因为空指针不是错误值。新版本在文档注释中明确说明了这一点，还添加了完整的 doctest（文档测试）覆盖多种场景：非空非错误指针、空指针、错误值。这种细致的文档工作对于避免误用非常重要。

社区讨论：PIN-INIT 中的 PINNABLE OPTION

社区讨论：pin-init 中的 Pinnable Option

在 Zulip 的 #pin-init 频道中，anonymix007、Benno Lossin 和 Gary Guo 三人展开了一场关于"固定初始化中的 Option 类型"的技术讨论（共 7 条消息）。

问题背景：anonymix007 在内核之外使用 pin-init 库时，想要对 Option<T> 类型使用固定初始化（pinned initialization）。

Benno Lossin 的回应：首先指出 anonymix007 不是在内核环境下开发的。在使用 init! 宏时，数据并没有被"固定"（pin），所以要使用固定选项需要用特殊的注解标记结构体。他建议，在这种情况下，用一个类似 Box 已有的 impl Init<T> 初始化方式来处理 Option<T> 可能更合适。

核心技术障碍：Benno 指出，目前没有办法安全地写出这样的函数，因为你不能通过裸指针（unsafe）来设置枚举变体的值。这个底层限制记录在 pin-init#59 中。简单来说，Rust 的 Option 是一个枚举（enum），而当前无法在不安全代码中直接通过指针写入枚举的判别值（discriminant），因此无法在原地（in-place）初始化一个 Option。

这个讨论揭示了 Rust 语言本身在底层内存操作上的一个限制，也是 pin-init 库未来需要解决的问题之一。
