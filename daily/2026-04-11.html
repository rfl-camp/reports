DRM/Tyr：新增 GPU 重置处理

Tyr 是一个用 Rust 编写的 DRM GPU 调度器驱动。今天 Alice Ryhl、Boris Brezillon 和 Daniel Almeida 提交了 4 个补丁，为 Tyr 添加了 GPU 重置（GPU reset）功能。GPU 重置是什么意思呢？简单说，当 GPU 卡死不响应的时候（比如某个渲染任务挂了），驱动需要有办法把 GPU"重启"一下，让它恢复正常工作，而不需要用户重启整台电脑。这是一个生产级 GPU 驱动必备的功能，说明 Tyr 正在朝着可以实际使用的方向稳步前进。

网络子系统

网络子系统

Rust Netlink 原始抽象（v2）

Alice Ryhl 提交了 Rust netlink 抽象的第 2 版补丁（共 4 个补丁）。Netlink 是 Linux 内核和用户空间程序之间通信的一种机制——你可以把它想象成内核开了一个"聊天窗口"，用户空间的程序（比如 ip 命令）通过这个窗口告诉内核"帮我配置一下网络"。目前很多网络配置工具（如 iproute2）都依赖 netlink。

这个补丁系列为 Rust 提供了 原始 netlink 抽象，也就是最底层的 netlink 操作接口。有了这个基础，后续就可以在其上构建更高层的 Rust netlink API（比如 Generic Netlink），让用 Rust 写网络相关内核模块变得更方便。这是 Rust 网络子系统支持的重要基石。

ACPI 设备驱动

ACPI 设备驱动

修复 PRP0001 设备的匹配数据缺失（v3）

Markus Probst 提交了第 3 版补丁，修复了 Rust ACPI 绑定中一个关于 PRP0001 设备匹配数据缺失的 bug。这是什么意思呢？在 Linux 中，驱动和设备之间需要"对暗号"才能配对——ACPI 子系统会告诉内核"这里有个设备，它的 ID 是 XXX"，然后内核去找哪个驱动能处理这个 ID。

PRP0001 是一个特殊的 ACPI 设备 ID，用于支持设备树（Device Tree）风格的设备描述。之前 Rust 绑定在处理这类设备时，匹配数据会丢失，导致驱动无法正确识别和绑定设备。Gary Guo 参与了审查，这已经是第 3 轮迭代，说明社区对这个修复的正确性要求很高。

社区讨论

如何测试 PCI SR-IOV 补丁？

在 Zulip 的 #Help 频道中，Greg Rose 遇到了一个实际问题：他想测试邮件列表上 Zhi Wang 提交的 PCI 扩展能力和 SR-IOV 支持补丁，但直接应用到 rust-next 分支时失败了。Miguel Ojeda 指出，这个补丁是基于 Gary Guo 的 io_projection 补丁系列开发的（使用了 Io/IoCapable/View 基础设施），需要先应用那些依赖补丁才行。

这是内核开发中非常常见的情况——补丁之间存在依赖关系，就像搭积木一样，得按顺序来。如果你也遇到类似问题，记得看补丁的 cover letter（封面信），里面通常会说明依赖哪些其他补丁系列。

关于 LLM 辅助补丁的标注规范

这是一个值得关注的社区治理话题。在 #Help 频道的"寻求可审查的 Rust 驱动目标"讨论中，Miguel Ojeda 注意到有人从同一地址发送了大量补丁，并询问这些补丁是否由 LLM（大语言模型）辅助生成。他提醒：如果补丁是 LLM 辅助编写的，应该在提交时明确标注。

这反映了内核社区对 AI 生成代码透明度的重视。随着 AI 工具在开发中的使用越来越普遍，社区正在形成相关规范——贡献者有义务披露 AI 辅助的情况，这有助于审查者评估代码质量和潜在风险。如果你也在使用 AI 工具辅助内核开发，请记得在补丁中做好标注。
