️ GPU / DRM 驱动（Nova-core 大重构）

nova-core：迁移到新的 dma::Coherent API（v2，8 个补丁）

今天 GPU 驱动方向是绝对的主角，围绕 NVIDIA 开源驱动 nova-core 有大量补丁活动。核心主题是：把驱动里自己实现的 DMA 内存管理替换成内核提供的统一接口。

Danilo Krummrich 提交了一组 8 个补丁，把 nova-core 驱动中原来自己定义的 DmaObject 类型全部替换成内核 Rust 绑定提供的 dma::Coherent API。简单来说，原来 nova-core 为了做 DMA 内存分配，自己造了一套轮子；现在内核的 Rust DMA 抽象已经足够成熟了，就可以把自己造的轮子扔掉，统一使用官方 API。这样做的好处是：代码更少、维护更容易、而且能和其他驱动共享同一套经过充分测试的基础设施。Gary Guo 参与了代码审查。

📬 查看补丁
nova-core 固件模块：RISC-V 部分也迁移到 dma::Coherent（7 个补丁）

Gary Guo 和 Alexandre Courbot 合作，把 nova-core 中负责 RISC-V 固件加载的部分也改成使用 dma::Coherent。NVIDIA 的 GPU 内部有一个 RISC-V 处理器（称为 GSP，GPU System Processor），驱动需要通过 DMA 把固件传给它。这组补丁就是把固件传输相关的内存管理也统一到新接口上来。补丁描述明确说了：「用 Coherent 替换 nova-core 本地的 DmaObject，它能完成同样的功能。」

📬 查看补丁
nova-core GSP：把命令队列的 DMA 句柄改成结构体成员

Alexandre Courbot 提交、Danilo Krummrich 审查的这个补丁，把 GSP 命令队列（Cmdq）中的 DMA 句柄从原来的处理方式改成了结构体成员。这是一个代码组织上的改进——把 DMA 资源的生命周期和命令队列绑定在一起，这样当命令队列被销毁时，DMA 资源也会被自动清理，避免内存泄漏。就像把钥匙挂在门把手上而不是随便放在桌子上，不容易丢。

️ GPU / DRM 驱动（Nova-core 大重构）

nova-core：Falcon 和 PDISP 寄存器转换为内核寄存器宏（v2，共 20 个补丁）

Alexandre Courbot 提交了两组补丁（各 10 个，共 20 个），把 nova-core 中定义 GPU 硬件寄存器的方式从手动定义改成使用内核提供的寄存器宏。Falcon 是 NVIDIA GPU 中的微控制器（用于固件执行），PDISP 是 GPU 中负责显示输出的部分。使用统一的宏来定义寄存器有几个好处：代码更规范、自动生成读写方法、减少手写出错的概率。这就好比从手写 HTML 转向用模板引擎——做的事情一样，但更不容易出 bug。

DRM 子系统：引入 DeviceContext（v6，5 个补丁）

这是今天最有意思的一组补丁！Lyude Paul（来自 Red Hat）提交了 DRM DeviceContext 系列的第 6 版。这组补丁要解决的问题是：DRM 设备的初始化分好几步——先创建设备对象、做一些准备工作（比如分配显存对象）、最后才注册给用户空间。问题在于，有些操作只有在设备完全注册之后才能安全执行，但之前的代码没有办法在编译时区分"设备是否已注册"。

Lyude 的解决方案非常 Rust：引入一个 DeviceContext 类型参数，让编译器来帮你检查。如果你的代码拿到的是一个"未注册"状态的设备对象，编译器就不让你调用那些需要"已注册"才能用的方法。这就像你的门禁卡——电梯按钮只有刷卡后才亮起来，物理上就阻止了你乱按。

DMA 内存管理 API 改进

DMA 内存管理 API 改进

新增 dma::Coherent 和 dma::CoherentBox API（v2，8 个补丁）

除了 nova-core 驱动在"消费"新的 DMA API，今天也有人在改进这套 API 本身。

Alexandre Courbot 提交了 DMA 抽象层的第 2 版改进。这组补丁引入了 dma::Coherent（用于分配一致性 DMA 内存的数组类型）和 dma::CoherentBox（用于分配单个对象的 DMA 内存）两个新类型。这就像标准库里 Vec 和 Box 的关系——一个用来存多个元素，一个用来存单个元素，但都是分配在"CPU 和设备都能直接访问"的特殊内存区域里。这套 API 是上面 nova-core 迁移的基础。

📬 查看补丁
为 Coherent 添加 zeroed 构造函数（v2）

同样来自 Alexandre Courbot 的补丁，为 dma::Coherent 添加了一个 zeroed 构造方法。分配 DMA 内存后把它全部清零是一个非常常见的需求（防止旧数据泄漏给硬件），有了这个方法就不需要每次都手动写清零逻辑了。这是一个小改进，但对驱动开发者的日常使用很方便。

📬 查看补丁
移除 DMA_ATTR_NO_KERNEL_MAPPING 的公开暴露（2 个补丁）

Danilo Krummrich 提交了一个重要的安全性修复。问题是这样的：当你在分配 DMA 内存时传入 DMA_ATTR_NO_KERNEL_MAPPING 这个标志，内核返回的地址实际上不是一个有效的内存指针，而是一个不透明的句柄（比如指向 struct page）。但 Coherent<T> 会把这个值当作 NonNull<T> 存起来，并且提供解引用方法——这意味着如果有人用了这个标志，再去读写这块"内存"，就会导致严重的内核崩溃或安全漏洞。所以这个补丁把这个危险的标志从公开 API 中移除了，从根源上防止误用。

构建系统与宏

构建系统与宏

修复 rustc 的 -C linker 选项与 ccache 的兼容问题（v5）

Mohamad Alsadhan 提交了一个构建系统的 bug 修复，已经到了第 5 版。问题出在哪呢？rustc 的 -C linker= 选项期望接收的是一个单独的可执行文件路径。但在很多开发环境中，HOSTCC 变量可能是 "ccache gcc" 这样包含空格的字符串（ccache 是一个编译缓存工具，能加速重复编译）。把这整个字符串传给 rustc 后，shell 会把它拆成两个词，rustc 就会把 gcc 当成一个输入文件名，然后报错：「error: multiple input filenames」。这个补丁的解决方案是确保只传一个单独的可执行文件路径给 rustc。这种"胶水层"的小问题虽然不起眼，但直接影响到大家能不能顺利编译带 Rust 支持的内核。

整理 build_assert 宏的模块归属（v3，4 个补丁）

Miguel Ojeda、Gary Guo 和 Alexandre Courbot 参与的这组补丁，把 build_assert! 相关的宏移到了一个专门的模块中。build_assert! 是一个在编译期进行断言检查的宏——如果某个条件不满足，编译就会失败。这类似于 C 语言中的 BUILD_BUG_ON。这个改动属于代码组织优化：把相关的宏放在一起，方便查找和维护。就像把书架上散落的同类书籍归到同一个格子里。

⏰ 时钟子系统

⏰ 时钟子系统

为时钟（clk）类型实现 Send 和 Sync（v5，3 个补丁）

Miguel Ojeda 和 Maxime Ripard 合作推进的这个补丁系列，为内核时钟框架的 Rust 绑定实现了 Send 和 Sync 这两个 trait。在 Rust 中，Send 表示"这个类型可以安全地发送到另一个线程"，Sync 表示"这个类型可以安全地被多个线程同时引用"。内核中的时钟资源经常需要在多个线程/上下文之间共享——比如多个驱动可能共用一个时钟源。没有这两个 trait，Rust 编译器就不允许你在线程间传递时钟对象，很多正常的驱动代码就写不出来。这个补丁已经到了第 5 版，说明社区对并发安全的审查非常严格。

📬 查看补丁

Send 和 Sync：Rust 中最核心的并发安全机制。Send 意味着可以把值"送"到另一个线程，Sync 意味着可以安全地跨线程共享引用。编译器会自动检查，如果你的类型没实现这俩 trait 却试图跨线程使用，代码直接编译不过。这就是 Rust "无畏并发"的秘密武器。了解更多

设备树绑定

添加 Synology 嵌入式控制器（MicroP）设备绑定（v4，2 个补丁）

Markus Probst 提交、Krzysztof Kozlowski 审查的这组补丁，为群晖（Synology）NAS 设备中的嵌入式控制器添加了设备树绑定定义。群晖的 NAS 里有一个叫 MicroP 的小型微控制器，负责管理电源、风扇、LED 灯、硬盘状态等底层硬件。设备树绑定就是用一种标准格式来描述"这个硬件在哪、怎么通信、需要什么资源"，好让内核驱动知道该怎么控制它。虽然这个补丁本身是设备树 YAML 文件的改动（不是 Rust 代码），但它出现在 Rust-for-Linux 邮件列表上，说明相关的驱动可能计划用 Rust 来编写。这个补丁已经收到了 4 条审查意见，社区在仔细讨论绑定的正确写法。

社区工具与讨论

Sashiko：AI 代码审查工具上线！

这是今天最值得关注的社区动态之一。Miguel Ojeda 在邮件列表和 Zulip 上宣布：Sashiko（一个 AI 代码审查工具）现在可以用于 Rust-for-Linux 邮件列表了！你可以在 sashiko.dev 上直接查看它对每个补丁的自动审查结果。目前这个工具还没有针对 Rust 特定的提示词（prompt），所以审查结果可能还不够精准。Gary Guo 正在编写专门的 Rust 提示词，Miguel 也计划把社区的编码规范链接进去，这样 AI 就能给出更有针对性的建议。社区还在 GitHub issue #1230 中收集大家对 AI 审查工具的功能需求和 bug 报告。

这标志着 Rust-for-Linux 社区正在积极拥抱 AI 辅助代码审查——在开源社区中，代码审查人力一直是稀缺资源，AI 工具有望缓解审查瓶颈。

在 Zulip 的 #pin-init 频道中，Benno Lossin 提出了一个关于 pin-init 库 staging 分支工作流的新想法。pin-init 是 Rust-for-Linux 中用于安全初始化被 pin 住的数据的关键基础库。Benno 的提议是：所有普通 PR 都先合并到 staging 分支，然后通过 GitHub Actions 自动化流程来管理合并。Gary Guo 回复说这个方案听起来合理。这个讨论虽然小，但反映了社区在持续改进开发流程，确保代码质量。

GitHub 动态：内核所需的 Rust 不稳定特性追踪

Issue #2 持续开放——这是一个追踪 Rust-for-Linux 所依赖的所有 Rust 不稳定（nightly）特性的元 issue。内核代码目前需要使用一些还没有被 Rust 稳定化的语言特性，这个 issue 记录了所有这些依赖。每当 Rust 团队稳定化了一个特性，就可以从列表里划掉一个。最终目标是让内核可以用稳定版 Rust 编译器来编译。另外，关于 CONFIG_RUST 编译前提条件说明的 PR #955 已经关闭。
