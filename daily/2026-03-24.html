️ GPU 驱动 · Nova（NVIDIA 开源内核驱动）

统一页表入口（PTE）包装枚举 v11

Nova 是社区为 NVIDIA GPU 开发的全新开源内核驱动，用 Rust 编写。今天这个方向的补丁数量最多，主要集中在 GA100 支持、MMU 页表抽象和安全性加固三个方面。

这是一个包含 20 个补丁的大型系列！GPU 内部有自己的内存管理单元（MMU），负责把虚拟地址翻译成物理地址——就像你手机里的 GPS 把地址翻译成经纬度一样。NVIDIA GPU 有两个版本的 MMU（v2 和 v3），它们的"页表项"格式不同。这组补丁创建了统一的 Pte、Pde、DualPde 枚举类型，让上层代码不用关心底层是哪个 MMU 版本，一套代码就能同时处理两种格式。每个统一类型在构造时接受 MmuVersion 参数，内部自动包装对应版本的变体，然后把方法调用委托给具体实现。这大大简化了页表遍历器和虚拟内存管理器的逻辑。

SysmemFlush 启动失败时的清理修复

当 GPU 初始化（Gpu::new）过程中，如果 SysmemFlush 注册成功后、后续步骤失败了，之前的代码不会自动注销它——因为 SysmemFlush 需要手动注销。这个补丁添加了 Drop 实现来自动清理（Rust 的 Drop 就像 C++ 的析构函数，对象被销毁时自动执行）。同时保留了手动注销路径，因为手动路径可以保证不会失败，而 Drop 路径依赖运行时状态。

VBIOS 安全加固：使用受检算术和访问

这两个补丁修复了 VBIOS 解析代码中的潜在溢出问题。GPU 的 VBIOS 数据来自固件，属于"不可信输入"——就像你处理用户上传的文件一样，不能假设数据一定合法。补丁把 setup_falcon_data 和 FwSecBiosImage::ucode 中的算术运算和数组访问改成了"受检"版本（checked arithmetic），如果发生溢出会返回错误而不是产生未定义行为。

️ GPU 驱动 · Nova（NVIDIA 开源内核驱动）

Hopper/Blackwell GPU 基础识别 v10

这是一个 28 个补丁的大系列，为 Nova 驱动添加对 NVIDIA 最新两代架构（Hopper 和 Blackwell）的基础 GPU 识别能力。目前已经到第 10 个版本，说明经过了大量的代码审查和反复修改。GPU 识别是驱动工作的第一步——驱动需要知道"我在和什么型号的 GPU 打交道"，然后才能加载正确的固件、配置正确的参数。

Nova 要求小端序 v2

这个补丁要求 Nova 驱动只在小端（little endian）系统上编译。NVIDIA GPU 的寄存器和内存布局都是小端的，在大端系统上运行需要大量字节序转换代码。目前先限制在小端平台（绝大多数现代 PC 和服务器），未来需要时再添加大端支持。

Zulip 讨论：依赖性 DevRes 资源管理

Gary Guo 和 Danilo Krummrich 在 Zulip 上讨论了一个有趣的架构问题：当一个设备资源（DevRes）依赖另一个资源时，如何优雅地管理它们的生命周期？具体场景是 SysmemFlush——它的注销依赖于另一个资源的存在。Gary 提出了一个通用方案，让一个资源可以引用另一个资源；Danilo 担心这会导致自引用结构（Rust 中的大坑），但 Gary 解释说这不是自引用——对结构体自身来说，它只是引用了"别的东西"。Danilo 也提出了自己的方案，但那个方案只适用于 Drop 场景，Gary 的方案更通用。

DRM 图形子系统 · Tyr 驱动与 DRM 框架

Tyr 驱动：添加 GPU 重置处理 v1

Tyr 是另一个用 Rust 编写的 DRM（Direct Rendering Manager）GPU 驱动。这个 4 补丁系列为 Tyr 添加了 GPU 重置（reset）功能。GPU 有时会"卡死"（hang），就像手机 App 无响应一样，这时候驱动需要能够重置 GPU 让它恢复正常工作。这是 GPU 驱动必备的基础功能，有了它驱动才算具备基本的容错能力。补丁由 Onur Özkan 提交，Boris Brezillon、Daniel Almeida 和 Alice Ryhl 参与了审查。

Zulip 讨论：DMA Fence 的存储结构

Philipp Stanner 在 DRM 频道提出，存储 DMA Fence 指针的最佳数据结构应该是 XArray。理由是：fence 需要按顺序发信号，而且经常需要按序号查找，XArray 正好擅长这种场景。Boris Brezillon 回复说自己在实现中已经用了类似方案，表示"英雄所见略同"。DMA Fence 是 GPU 驱动中用来同步操作的核心机制——比如 CPU 告诉 GPU "渲染完这一帧再通知我"，fence 就是这个"通知"。

内核 RUST 抽象层 · 核心子系统绑定

内核 Rust 抽象层 · 核心子系统绑定

io_uring 命令抽象接入 miscdevice v4

这个 5 补丁系列非常重要！它为 Rust 驱动提供了 io_uring 命令的抽象。io_uring 是 Linux 上最高效的异步 I/O 接口，广泛用于高性能服务器和数据库。这个系列的结构是：先添加 C 头文件绑定，然后构建核心 Rust 抽象，最后把它接入 miscdevice（杂项设备）框架，让 Rust 驱动可以处理 IORING_OP_URING_CMD 透传命令。由 Sidong Yang 提交，Greg Kroah-Hartman（Linux 稳定版维护者）参与审查。

📬 查看补丁
workqueue：添加安全的取消和状态查询方法 v4

Linux 的工作队列（workqueue）是内核中最常用的"延迟执行"机制——当你不想在当前上下文立即做某事，就把它扔进工作队列稍后执行。这个补丁为 Rust 的 workqueue 绑定添加了安全的任务取消和状态查询方法。之前只能提交任务，现在可以安全地取消排队中的任务、查询任务执行状态了。已经到第 4 版，由 Aakash Bollineni 提交。

📬 查看补丁
netlink：原始 netlink 抽象 v2

Netlink 是 Linux 内核和用户空间程序之间通信的主要通道之一——比如你用 ip 命令配置网络时，底层就是通过 netlink 和内核对话的。Alice Ryhl（Google 的 Rust-for-Linux 核心开发者）提交的这个 4 补丁系列为 Rust 添加了原始 netlink 抽象，让 Rust 内核模块也能方便地与用户空间通过 netlink 通信。Andrew Lunn（网络子系统维护者）参与了审查。

seq_file：添加 puts、putc、write 和 hex_dump 方法 v2

seq_file 是内核中用来生成 /proc 文件内容的工具——比如你查看 /proc/cpuinfo 时，内核就是用 seq_file 一行行输出 CPU 信息的。这个补丁为 Rust 的 seq_file 绑定添加了更多输出方法：puts（输出字符串）、putc（输出单个字符）、write（输出字节数组）和 hex_dump（以十六进制格式输出数据，调试时特别有用）。

📬 查看补丁
ACPI：修复 PRP0001 设备的匹配数据缺失 v3

ACPI 是现代 PC 上描述硬件配置的标准。PRP0001 是一种特殊的 ACPI 设备类型，它允许 ACPI 设备使用 Device Tree（设备树）风格的兼容性字符串来匹配驱动。这个补丁修复了一个 bug：在 Rust 侧，PRP0001 设备的 id_info 总是 None。原因是 Rust 在设备 ID 中存储的是索引而非指针，导致无法直接使用 device_get_match_data。修复方案是导出 acpi_of_match_device 函数，在 Rust 中使用它来正确匹配。

module_param：支持 bool 类型参数

内核模块加载时可以接受参数（就像命令行程序接受参数一样，比如 modprobe mydriver debug=1）。这个补丁让 Rust 内核模块也能声明布尔（bool）类型的参数了。虽然功能简单，但这是 Rust 模块参数系统完善的重要一步。由 Wenzhao Liao 提交，Greg KH 和 Andreas Hindborg 参与审查。

Rust 模块的 C API 与 Bindgen

David Rheinsberg 提出了一个有趣的场景：如果一个 Rust 内核模块需要暴露 C API 给其他内核代码使用（比如 DRM 工具库用 Rust 写但要给 C 驱动调用），该怎么办？Alice Ryhl 给出了详细回答：常量应该通过 bindgen 使用 C 头文件中定义的值；函数应该使用她已经添加的 #[export] 宏来确保签名正确；但对于结构体大小，目前还没有好的解决方案——C 头文件不能依赖 Rust 代码，所以只能"猜测/高估大小"然后在 Rust 侧加断言验证。这反映了 Rust-for-Linux 在 C/Rust 互操作方面仍有待完善的地方。

安全相关：/proc/mem 权限收紧

Linus Torvalds 亲自提交了一个改动：将 PROC_MEM_FORCE_PTRACE 设为默认的 Kconfig 选项。这个选项控制是否允许强制覆盖 /proc/[pid]/mem 的内存访问权限。之前默认是"总是允许"（为了兼容历史行为），现在改为更安全的默认值。这个选项 18 个月前就引入了，文档也写明了将来会成为默认值，现在终于落地。

RCU 子系统更新

Joel Fernandes 提交的 RCU 更新合入了主线，包括：NOCB CPU 管理代码整理（减少重复代码）、提取 nocb_bypass_needs_flush() 辅助函数等。RCU（Read-Copy-Update）是内核中最重要的并发同步机制之一，专为"读多写少"场景优化。

用户态拷贝函数清理

Linus 合并了一组清理补丁，重构了特殊的用户态拷贝函数的命名和语义。具体来说，去掉了传统的双下划线命名和行为——之前的"optimize away the access_ok() check"优化方式被简化了。这种清理让内核代码更易读、更不容易出错。
