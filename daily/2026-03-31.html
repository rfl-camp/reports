️ GPU / DRM 驱动（nova-core 与图形子系统）

1. Hopper/Blackwell 架构：跳过 GFW 启动等待

今天 GPU 驱动方面的动态最多，主要围绕 NVIDIA 的开源 Rust GPU 驱动 nova-core 展开。这是一个用 Rust 从头编写的 NVIDIA GPU 内核驱动，目标是替代现有的 C 语言 nouveau 驱动。今天推送了 v9 大版本的一系列补丁，涵盖固件加载、GPU 初始化、安全启动等核心功能。

这个补丁由 Joel Fernandes 等人提交，属于 v9 系列的一部分。GFW（GPU Firmware）是 GPU 内部运行的固件程序，在某些情况下，驱动程序需要等待 GFW 完成初始化才能继续工作。但对于较新的 Hopper（H100 等）和 Blackwell（B200 等）架构的 GPU，在特定启动路径下可以安全地跳过这个等待步骤，从而加快 GPU 的初始化速度。这个补丁就是在驱动中判断当前 GPU 架构，如果是 Hopper 或 Blackwell 且满足条件，就直接跳过 GFW 等待阶段。收到了 5 个评审意见，包括来自 Gary Guo 和 John Hubbard 的反馈。

📬 查看补丁
2. Hopper/Blackwell 架构：FSP 信任链启动

这是 nova-core v9 系列中另一个关键补丁。FSP（Falcon Security Processor）是 NVIDIA GPU 上的一个安全处理器，负责在 GPU 启动时验证固件的签名，确保加载的固件没有被篡改——就像你手机开机时会验证系统签名一样。这个补丁为 Hopper 和 Blackwell 架构实现了完整的 FSP 信任链（Chain of Trust）启动流程。这意味着驱动程序现在可以通过 FSP 安全地引导 GPU 固件，是 GPU 安全启动的重要一环。由 John Hubbard 和 Alexandre Courbot 合作完成。

📬 查看补丁
3. Hopper/Blackwell 固件支持总览

John Hubbard 提交的这个补丁是 v9 系列中关于固件（firmware）层面的总体支持。它为 Hopper 和 Blackwell 架构的 GPU 添加了完整的固件加载和管理逻辑，包括如何找到正确的固件文件、如何解析固件格式、如何将固件传送到 GPU 的各个引擎上。这是整个 GPU 驱动能正常工作的基础——没有正确的固件，GPU 就是一块"砖头"。

📬 查看补丁
4. 固件签名段选择重构（fwsignature）

这个补丁把原来散落在代码各处的 .fwsignature* ELF section 查找逻辑，提取成了一个独立的函数 find_gsp_sigs_section()。简单来说，GPU 固件文件里包含多个"签名段"，驱动需要根据当前 GPU 型号找到正确的签名段来验证固件。之前这个逻辑写得比较分散，现在把它整理成一个专门的函数，代码更清晰、更容易维护。这种重构虽然不改变功能，但对代码质量很重要。

📬 查看补丁
5. 根据 GPU 架构设置 DMA 掩码宽度

不同代的 GPU 支持的内存地址范围不同——比如老一点的 GPU 可能只能访问 40 位地址空间（1TB），而新的 GPU 可以访问 47 位甚至更多。DMA 掩码就是告诉内核"这个设备能访问多大的内存地址范围"。这个补丁让 nova-core 驱动根据检测到的 GPU 架构（比如是 Turing、Ampere 还是 Hopper）自动设置正确的 DMA 掩码宽度，这样内核就能正确地为 GPU 分配它能访问到的内存。如果设置错了，轻则性能下降，重则导致数据传输出错。

6. fwctl 固件控制接口驱动

Joel Fernandes 和 Jason Gunthorpe 合作提交了这个补丁，为 nova-core 添加了 fwctl（firmware control）驱动。fwctl 是 Linux 内核中一个相对较新的子系统，专门用于让用户态程序与设备固件进行通信。通过这个接口，管理工具可以查询 GPU 固件的状态、发送控制命令、获取诊断信息等，而不需要通过传统的 ioctl 接口。这对于 GPU 的运维管理和调试非常有价值，特别是在数据中心场景下。收到了 4 个评审意见。

📬 查看补丁
7. GSP 命令队列未定义行为修复

Alexandre Courbot 提交了一个修复补丁（v3），解决了 nova-core 驱动中 GSP（GPU System Processor）命令队列代码里的未定义行为（Undefined Behavior，简称 UB）。在 Rust 中，未定义行为是非常严重的问题——它意味着编译器可能产生任何结果，程序行为完全不可预测。这个 bug 出现在驱动向 GPU 的 GSP 处理器发送命令时的内存操作中，可能导致数据损坏甚至系统崩溃。修复方法是使用更安全的内存访问模式，避免违反 Rust 的安全规则。这正是用 Rust 写驱动的好处——编译器和工具能帮你发现这类潜在的内存安全问题。

1. 不用完整 LTO 也能内联 C 函数到 Rust（v2）

这是一个很有意思的补丁！目前内核中有很多用 C 写的小工具函数（inline helpers），Rust 代码需要调用它们。但通常只有在开启 LTO（Link-Time Optimization，链接时优化）的情况下，编译器才能把这些 C 函数真正"内联"到 Rust 代码中——也就是把函数调用替换成函数体本身，避免函数调用的开销。问题是完整的 LTO 会让编译时间大幅增加。Christian Schrefl 的这个补丁提出了一种新方案，让编译器在不需要完整 LTO 的情况下也能实现跨语言内联，这对编译速度和运行性能都有好处。收到了 Arnd Bergmann 和 Geert Uytterhoeven 等 4 个评审意见，讨论了实现细节和兼容性问题。

📬 查看补丁
2. 修复 Rust Makefile 中的竞态条件（V7）

Mukesh Kumar Chaurasiya 发现并修复了内核构建系统中一个关于 Rust 编译的竞态条件 bug。竞态条件就像两个人同时要过一扇门，如果没有协调好，可能会撞在一起。在这里，问题出在 Makefile 中：当使用 make -j（多线程编译）时，某些 Rust 编译步骤可能会并行执行并产生冲突，导致编译失败或输出错误的结果。这个补丁已经到了第七版，说明修复方案经历了多轮评审和改进。虽然听起来是个小问题，但构建系统的可靠性直接影响所有开发者的日常工作效率。

3. 用 const_assert! 替换 build_assert!

Miguel Ojeda（Rust-for-Linux 项目的维护者）提交了一个代码清理补丁，把内核 Rust 代码中使用的 build_assert! 宏替换为 const_assert!。这两个宏的功能类似——都是在编译期间检查某个条件是否成立（就像考试前检查答题卡有没有填错一样）。const_assert! 的命名更符合 Rust 社区的惯例，语义也更明确："这是一个在编译时求值的常量断言"。这种命名统一虽然不改变功能，但让代码更容易被 Rust 开发者理解。

⚙️ 内核基础设施（指针、时钟、同步）

⚙️ 内核基础设施（指针、时钟、同步）

1. 指针工具函数：const_align_up()

John Hubbard 为内核 Rust 代码的指针模块（rust/kernel/ptr.rs）添加了一个新的工具函数 const_align_up()。这个函数的作用是将一个地址"向上对齐"到指定的边界——比如把地址 0x1003 对齐到 4 字节边界就变成 0x1004。在内核编程中，内存对齐非常重要：CPU 访问对齐的数据更快，某些硬件甚至要求数据必须对齐才能正常工作。const 前缀意味着这个函数可以在编译期间计算结果，不需要运行时开销。这个补丁也是 nova-core v9 系列的一部分，GPU 驱动在配置 DMA 缓冲区时需要大量的地址对齐操作。收到了 Miguel Ojeda 和 Alexandre Courbot 等 5 位评审者的反馈。

📬 查看补丁
2. 时钟子系统 Rust 绑定合入请求（v7.1）

Andreas Hindborg 提交了 Rust 时钟（timekeeping）子系统的合入请求。时钟子系统是内核的"手表"——负责追踪时间的流逝，提供各种精度的计时功能。这次的改动主要包括三部分：为 HrTimer（高精度定时器）扩展了使用示例文档，为 Delta（时间差）类型添加了构造函数，以及将 ClockSource 的 trait 标记为 unsafe。最后一点很有意思——之所以标记为 unsafe，是为了确保只有能提供正确时间值的实现才会使用它，因为返回错误的时间可能导致严重的系统问题（想象一下你的闹钟如果不准会怎样）。

📬 查看合入请求
3. 为时钟框架实现 Send 和 Sync（v5）

Miguel Ojeda 评审了为时钟（clk）框架实现 Send 和 Sync trait 的补丁（第五版）。在 Rust 中，Send 意味着一个值可以安全地从一个线程传递到另一个线程；Sync 意味着一个值可以安全地被多个线程同时访问（通过引用）。为时钟框架的类型实现这两个 trait，意味着 Rust 驱动可以在多线程环境中安全地使用时钟 API——而编译器会在编译时自动检查这种安全性。这比 C 代码中需要程序员自己记住加锁要可靠得多。

️ 架构支持：PowerPC

为 PowerPC (ppc64le) 启用 Rust 支持（V7）

Mukesh Kumar Chaurasiya（来自 IBM）提交了两组相关补丁，目标是让 PowerPC 64 位小端（ppc64le）架构也能编译和运行 Rust 内核代码。目前 Rust-for-Linux 主要支持 x86_64 和 ARM64 架构，PowerPC 的加入将扩大 Rust 在内核中的适用范围——PowerPC 架构广泛用于 IBM 的服务器和大型机，在企业级市场很重要。

这组补丁共 4 个，包括：为 PowerPC 添加 Rust 编译器的目标配置、修改构建系统以支持 PowerPC 特有的调用约定和 ABI，以及处理 PowerPC 架构特有的一些底层细节（比如函数指针的处理方式和 x86 不同）。已经到了第七版，说明社区对细节的打磨非常严格。这也是上面提到的 Makefile 竞态条件修复所在的补丁系列的一部分——在为 PowerPC 添加支持的过程中发现了构建系统的 bug，顺手修复了。

设备驱动

1. ACPI：支持 PRP0001 设备匹配（v5）

Markus Probst 和 Rob Herring 提交的这个补丁改进了内核中 ACPI（高级配置与电源接口）与 设备树（Device Tree）之间的桥接。简单来说，电脑有两种方式告诉操作系统"我有哪些硬件"：一种是 ACPI（主要用于 x86 PC 和服务器），另一种是设备树（主要用于嵌入式设备和 ARM 平台）。有些硬件设备使用一个特殊的 ACPI 标识符 PRP0001，表示"我的信息请去设备树里找"。这个补丁让 of_match_device 函数能正确识别和匹配这类设备，使得使用设备树描述的驱动也能在 ACPI 系统上工作。这对 Rust 驱动很重要，因为新的 Rust 驱动可能需要同时支持 ACPI 和设备树两种系统。已到第五版，收到 3 个评审意见。

📬 查看补丁
2. Synology MicroP 平台驱动（v5）

Conor Dooley 提交了一个新的平台驱动——Synology MicroP 驱动的第五版。Synology 就是群晖，做 NAS（网络存储）设备的那个公司。MicroP 是群晖 NAS 设备内部的一个微控制器，负责管理风扇转速、LED 指示灯、蜂鸣器、电源按钮等硬件外设——就像是 NAS 的"管家芯片"。这个 Rust 驱动实现了与 MicroP 微控制器的通信协议，让 Linux 内核能够控制这些外设。这是一个很好的例子，说明 Rust 驱动开发已经从"概念验证"走向了"实际产品"——真正在消费级产品中使用的硬件驱动。

内核需要的 Rust 不稳定特性清单（#1223）

Linux 内核的 Rust 代码使用了一些 Rust 语言中尚未稳定的特性（unstable features）。这个 issue 跟踪所有这些不稳定特性的状态——哪些已经被稳定化了、哪些还在讨论中、哪些可以用替代方案绕过。减少对不稳定特性的依赖是 Rust-for-Linux 项目的一个重要目标，因为不稳定特性随时可能改变，会给内核代码带来维护负担。

Clippy 需要的功能与修复（#349）

Clippy 是 Rust 的代码检查工具（linter），帮助开发者发现代码中的常见问题和不良模式。这个 issue 追踪了 Clippy 在内核代码场景下需要改进的地方，比如某些 lint 规则在内核上下文中可能产生误报，或者需要新的 lint 规则来检查内核特有的编码规范。
