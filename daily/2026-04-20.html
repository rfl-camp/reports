GPU 驱动：Nova Core 新增 GA100 (A100) 支持

具体来说，这 5 个补丁分别做了这些事情：

今天最密集的补丁活动来自 Gary Guo，他对 Nova Core 驱动提交了一组共 5 个补丁的代码审查意见（v5 版本，属于一个 6 补丁系列的一部分）。Nova Core 是社区正在用 Rust 从零编写的 NVIDIA GPU 驱动，目标是替代现有的 C 语言 Nouveau 驱动。这次的核心目标是让驱动能够正确识别和启动 GA100 芯片——也就是 NVIDIA A100 数据中心 GPU 所用的芯片架构。

具体来说，这 5 个补丁分别做了这些事情：

1. 修正 GA100 的固件签名 (fwsignature)：GPU 启动时需要加载固件（firmware），不同芯片需要匹配不同的固件签名来验证固件合法性。这个补丁修正了 GA100 使用的固件签名值，确保驱动能加载正确的固件文件。如果签名搞错了，GPU 根本无法初始化——就像你用错了钥匙，门是打不开的。

📬 查看补丁

2. 过滤无效的 PCI ROM 头部签名：GPU 的固件有时存储在 PCI 设备的 ROM 中。驱动在读取 ROM 时需要验证头部签名是否合法。这个补丁将 0xBB77 标记为无效签名，防止驱动把垃圾数据当作合法固件来加载。这属于防御性编程——宁可拒绝启动，也不要因为加载了损坏的固件导致 GPU 行为异常甚至系统崩溃。

📬 查看补丁

3. 仅在 FRTS 区域被分配时才启动 FRTS：FRTS（Fault Replay Table Storage）是 GPU 内存管理中的一个功能区域，用于记录和回放内存页面错误。不是所有 GPU 配置都会分配这个区域。这个补丁加了一个检查：如果 FRTS 区域没有被分配，就跳过启动步骤，而不是盲目地去访问一块不存在的内存——后者会直接导致内核崩溃。

4. 为 GA100 添加 FbHal::frts_size() 方法：这是上一个补丁的前置工作。GA100 的 FRTS 区域大小与其他芯片不同，所以需要在 framebuffer 硬件抽象层（FbHal）中添加一个专门的方法来返回 GA100 的 FRTS 尺寸。这就像给一个通用接口加了一条"GA100 专用规格"的记录。

构建系统：更精确地管理 RUST 编译器版本兼容性

构建系统：更精确地管理 Rust 编译器版本兼容性

HeeSu Kim 提交了 v6 版本的两个相关补丁，解决的问题是：内核构建系统中有一条"绕过"指令（workaround），用来关闭 Rust 编译器对 ABI 兼容性的检查。这条绕过指令是因为 rustc 1.90.0 之前的版本在生成文档（rustdoc）和运行文档测试（doctest）时存在 bug，会误报 ABI 不匹配的错误。但问题是——新版的 rustc 已经修复了这些 bug，继续无条件关闭检查就不对了，因为 ABI 检查本身是有用的安全保护。

第一个补丁在构建脚本 scripts/Makefile.compiler 中新增了一个 rustc-lt-version 宏。内核构建系统已经有 rustc-min-version（检查编译器版本 ≥ 某版本），但缺少反向的检查——"编译器版本 < 某版本"。有了这个宏，构建脚本就能表达"只在旧版编译器上才启用某个 workaround"的逻辑。设计上使用"小于"而非"小于等于"，这样版本号可以直接写整数（如 109000 表示 1.90.0），而不用写成 108999 这种不直观的数字。

📬 查看补丁

第二个补丁修改了 rust/Makefile，利用新增的 rustc-lt-version 宏，将 -Cunsafe-allow-abi-mismatch=fixed-x18 这个编译器标志限制为仅在受影响的编译器版本上生效。具体来说：rustdoc 的 bug（#144521）在 1.90.0 修复，doctest 的 bug（#146465）在 1.92.0 修复。补丁将这两个场景分别绑定到对应的版本上限，让新版编译器恢复正常的 ABI 安全检查。这是一个典型的"清理技术债务"的工作——临时补丁不应该永远留在代码里。

同步原语：给全局锁守卫加上"别丢弃我"的警告

同步原语：给全局锁守卫加上"别丢弃我"的警告

Ashutosh Desai 提交了一个补丁，给 GlobalGuard 和 GlobalLock::try_lock() 加上了 #[must_use] 标注。这看起来是个很小的改动，但它解决的是一个隐蔽的并发 bug 来源。

来打个比方：锁就像一把钥匙，你拿到钥匙（获取锁）才能进入房间（访问共享数据），出来时要还钥匙（释放锁）。在 Rust 的内核同步原语中，Guard 就是这把"钥匙"——当 Guard 被销毁（drop）时，锁会自动释放。内核中的 Guard 类型已经有 #[must_use] 标注，意思是"如果你获取了这个值却不使用它，编译器会给你发出警告"。

问题在于 GlobalGuard 是 Guard 的一个封装（wrapper），但它自身缺少 #[must_use] 标注。这意味着如果你写了 lock.lock(); 而不是 let _guard = lock.lock();，GlobalGuard 会被立即创建又立即销毁——锁一瞬间就被释放了，你的临界区代码实际上根本没有被锁保护！Rust 编译器不会对此发出任何警告。同样，try_lock() 返回的 Option<GlobalGuard> 也有这个问题，因为 Option<T> 不会自动继承 T 的 #[must_use] 属性。这个补丁修复了这个遗漏。

小白科普：
#[must_use]：Rust 的一个编译器属性（attribute），用来标记"这个返回值你不能忽略"。如果你调用了一个 #[must_use] 函数却不使用它的返回值，编译器会发出警告。这在锁的场景下特别重要，因为"忽略锁的返回值"等于"立刻释放锁"。了解更多

设备驱动：首个 RUST 磁旋转位置传感器驱动 RFC

设备驱动：首个 Rust 磁旋转位置传感器驱动 RFC

Muchamad Coirul Anwar 提交了一份 RFC（征求意见稿），为 ams OSRAM AS5600 磁旋转位置传感器编写了一个 Rust 驱动。AS5600 是一款通过检测磁铁旋转角度来测量位置的传感器芯片，常见于电机控制、旋钮和机器人关节等应用场景——你可以把它想象成一个超精确的"旋钮位置读取器"。

这个驱动属于内核的 IIO（Industrial I/O，工业输入输出）子系统中的"position"（位置）类别。目前实现的功能包括：通过 I2C 总线与传感器通信、设备探测（probe），以及严格解析 AGC（自动增益控制）硬件状态寄存器。AGC 是传感器内部用来自动调节信号强度的机制，读取它的状态可以判断传感器是否工作正常、磁铁位置是否合适。

值得注意的是，此前已经有人（Frank Zago）提交过这个传感器的 C 语言驱动。这次的 RFC 用 Rust 重新实现，一方面是为了验证 Rust 在 IIO 子系统驱动开发中的可行性，另一方面也是 Rust-for-Linux 项目积累真实驱动用例的重要一步。作为 RFC，这份代码的主要目的是收集社区反馈，尤其是关于 Rust IIO 驱动的 API 设计和代码组织方式。

I2C 子系统：RUST 绑定正式提交合入主线

I2C 子系统：Rust 绑定正式提交合入主线

Wolfram Sang（I2C 子系统维护者）发出了 [GIT PULL] rust-i2c-next for v8.0 请求，这意味着 I2C 子系统的 Rust 绑定代码已经准备好合入 Linux 内核 v8.0 版本了。GIT PULL 是内核开发流程中的一个关键节点——子系统维护者将自己分支上积累的补丁整理好，请求上游维护者（通常是 Linus Torvalds 或子系统的上级维护者）拉取合并。

I2C 的 Rust 绑定非常重要，因为 I2C 是嵌入式 Linux 中最常用的总线之一——温度传感器、EEPROM、触摸屏控制器、电源管理芯片等大量外设都通过 I2C 连接。有了 Rust 绑定，上面提到的 AS5600 传感器驱动这样的 Rust 驱动才有基础设施可以依赖。这个 pull request 能走到合入阶段，说明社区对 Rust I2C 绑定的质量和 API 设计已经达成了共识。
