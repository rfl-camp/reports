GPU 驱动 · nova-core 大规模寄存器宏迁移

今天 nova-core（NVIDIA 开源 GPU 驱动）提交了一组 10 个补丁的 v2 版本，目标只有一个：把所有自造的寄存器定义宏替换成内核统一的 register 宏。

故事是这样的：nova-core 一开始开发的时候，内核里还没有通用的寄存器定义宏，所以他们自己写了一个"临时版"来用。这个临时版一直运作良好，也帮助内核社区验证了「我们到底需要什么样的寄存器宏」。现在，内核的通用版本已经合并了，所以是时候把 nova-core 的"临时工"退休了。

这次迁移涉及多个寄存器组：

▸ PMC（电源管理控制器）寄存器迁移
Alexandre Courbot 把所有 PMC 寄存器转换成使用内核的 register 宏。为了让声明更简洁，还引入了一个 nv_reg 辅助宏——因为 nova-core 的寄存器都有一些共同属性（比如都是 32 位宽，都是 crate 内部可见），所以用这个小宏来省去重复书写。

📬 查看补丁 · PMC 寄存器

▸ Falcon / PFALCON / PRISCV 寄存器迁移
这是最复杂的一组——Falcon 是 NVIDIA GPU 内部的一个微控制器，它有好几代架构（PFALCON、PFALCON2、PRISCV）。因为这些寄存器之间会共享一些"相对寄存器"类型定义，所以必须一起更新，不能一个一个来，否则编译就会出错。更新完成后，nova-core 自己的 register 宏就没人用了，直接删除。

📬 查看补丁 · Falcon 寄存器

▸ PDISP（显示引擎）寄存器迁移
Joel Fernandes 完成了显示相关寄存器的迁移。PDISP 负责 GPU 的显示输出部分（可以理解为"管显示器的那个模块"）。这部分迁移比较直接，按照同样的模式替换即可。

📬 查看补丁 · PDISP 寄存器

▸ 引入 bounded_enum 宏
Alexandre Courbot 还引入了一个新的 bounded_enum 宏，它是之前 impl_from_enum_to_u8 宏的升级版。这个宏能定义一个枚举类型，同时自动关联一个固定位宽的 Bounded 类型，并且自动生成 From 和 TryFrom 的实现——这样枚举就可以直接用作寄存器字段了。简单来说，它让「定义一个能塞进寄存器的枚举」变得只需要一行代码。

Lyude 和 Gary Guo 在 Zulip 的 #DRM 频道讨论了一个关于 DRM 设备生命周期的微妙问题：当设备对象还处于"未初始化"（Uninit）类型状态时，如果它被 drop（销毁），该怎么正确处理？讨论中他们还顺便吐槽了 AI 代码审查工具（疑似 Sashiko.dev）生成的一些"一本正经的胡说八道"——AI 并不理解 Rust-for-Linux 社区独特的"Context 模式"设计。Gary Guo 总结说，AI 审查整体上有用，但缺乏 Rust 特定的理解能力。

DMA 内存管理 · COHERENT API 全面升级

DMA 内存管理 · Coherent API 全面升级

Danilo Krummrich 今天提交了一组 8 个补丁的 v2 版本，对 Rust 的 DMA（直接内存访问）抽象层进行了一次全面升级。这些补丁环环相扣，让我们来看看它们分别做了什么：

▸ 新增通用容器类型
以前的 DMA 抽象只支持"切片"（slice）类型的内存，也就是一段连续的同类型元素数组。但实际驱动开发中，有时候需要把一个结构体（而不是数组）映射到 DMA 内存上。这个补丁增加了对任意类型的支持——就像从"只能装一排苹果的盒子"升级到了"能装任何东西的盒子"。

📬 查看补丁 · 通用容器

▸ Coherent 的零初始化构造函数
新增了 Coherent::zeroed() 方法。有时候我们需要分配一块 DMA 内存，并且确保里面全是零（硬件通常要求初始状态为全零）。以前需要手动做这件事，现在一个函数调用就搞定了。

📬 查看补丁 · zeroed 构造函数

▸ 引入 CoherentInit 和 CoherentBox
这两个补丁解决的是"初始化"问题。在内核中，你不能像在普通 Rust 程序里那样随便在栈上创建一个大对象然后拷贝到 DMA 内存——因为内核栈很小（通常只有 8-16KB）。CoherentInit 提供了一种新的初始化模式，而 CoherentBox 则是一个支持就地初始化（pin-init）的 DMA 内存容器。Gary Guo 参与了 CoherentBox 的设计和审阅。

📬 查看讨论 · CoherentInit 📬 查看补丁 · CoherentBox

DMA 内存管理 · Coherent API 全面升级

▸ 新增 Coherent::init() 和 Coherent::init_with_attrs()
这是这组补丁的"封顶之作"。有了前面的基础设施，现在可以提供完整的初始化 API 了。init() 用于标准初始化，init_with_attrs() 则允许传入额外的 DMA 属性（比如是否需要缓存一致性、是否允许写合并等）。这让驱动开发者有了更精细的控制能力。

📬 查看补丁 · init API

▸ Zulip 讨论：内存屏障 API 设计
Gary Guo 在 #Library 频道向 Boqun Feng 请教：Rust 里的内存屏障（memory barrier）API 应该怎么设计？Gary 提议用 Rust 风格的 Ordering 枚举（比如 mb(Full) 表示全屏障、mb(Acquire) 表示读屏障），这样更符合 Rust 的习惯。但 Boqun Feng 指出这种映射并不准确——rmb()（读内存屏障）和 Acquire 语义其实不完全一样，内核已经有了现成的 rmb()/wmb()/mb() 接口，保持 C 的命名可能更不容易出错。这个讨论还在进行中，尚未达成结论。

💬 查看讨论
💡 小白科普
DMA（Direct Memory Access，直接内存访问）：硬件设备可以直接读写内存，而不需要 CPU 当"搬运工"。但这也意味着 CPU 和设备需要看到同一份数据，所以需要"Coherent（一致性）"机制来保证双方数据同步。 📖 内核 DMA API 文档
内存屏障（Memory Barrier）：CPU 为了速度会重新排列指令执行顺序，但和硬件交互时顺序很重要。内存屏障就像一条"此路不准超车"的标线，保证屏障前的操作一定在屏障后的操作之前完成。 📖 内核内存屏障文档

RUST 核心基础设施 · 指针对齐与错误类型优化

Rust 核心基础设施 · 指针对齐与错误类型优化

▸ const_align_up() 指针对齐函数 v7
David Rheinsberg 提交了第 7 版的 const_align_up() 补丁，这是一组包含 31 个补丁的大系列！这个函数的作用是"把一个指针地址向上对齐到指定的边界"。听起来很简单，但在内核中它几乎无处不在——硬件经常要求内存地址必须是 4、8、16 甚至 4096 的倍数。目前的实现不能在编译期（const）使用，这个补丁让它在编译期也能工作，这对 Rust 的 const 泛型和编译期计算非常重要。经过 13 位审阅者的反复打磨，已经到了第 7 版，Miguel Ojeda 和 Alice Ryhl 都参与了审阅。

📬 查看补丁

▸ 为所有 Error 的 from() 函数标记 #[inline]
Alistair Francis（来自 Western Digital）提交了一个看起来很小但实际影响面很广的补丁：给所有 impl From<...> for Error 的 from() 函数加上 #[inline] 标注。这些函数通常只有一两行代码（就是把一种错误类型转换成另一种），但因为它们被调用得非常频繁，如果编译器不内联它们，每次转换都会产生一次函数调用开销。加上 #[inline] 之后，编译器会把函数体直接"粘贴"到调用点，消除调用开销。Benno Lossin 和 Miguel Ojeda 参与了审阅。

📬 查看补丁
💡 小白科普
内存对齐（Alignment）：硬件访问内存时，通常要求地址是某个数的倍数（比如 4 字节对齐意味着地址必须是 4 的倍数）。就像停车时车必须停在车位线内——不对齐的访问可能很慢甚至直接报错。 📖 了解内存对齐
内联（Inline）：编译器把函数代码直接"复制粘贴"到每个调用点，省去了函数调用的开销。对于很短的函数来说，调用的开销可能比函数本身还大，所以内联特别有用。 📖 Rust inline 属性文档

⚙️ 构建系统 · AUTOFDO 支持与链接器优化

⚙️ 构建系统 · AutoFDO 支持与链接器优化

▸ 为 Rust 代码添加 AutoFDO 支持
Alice Ryhl、Gary Guo 和 Nathan Chancellor 合作，给内核的 Rust 代码带来了 AutoFDO（Automatic Feedback-Directed Optimization，自动反馈导向优化）支持。这是什么意思呢？简单来说：先运行一遍程序，记录下"哪些代码跑得最多"（性能采样），然后把这个信息反馈给编译器，让编译器针对热点路径做更激进的优化。这个技术在 C 代码的内核编译中已经使用了，但 Rust 代码之前还不支持。这个补丁收到了 4 位审阅者的反馈，正在积极推进中。

📬 查看补丁

▸ 统一 rustc 链接器为单一可执行文件 v3
Nicolas Schier 提交了第 3 版补丁，目的是在 kbuild（内核构建系统）的 host 程序编译中，让 rustc -C linker 使用单一的可执行文件。目前的构建过程中，链接器的指定方式有些混乱（可能用脚本、可能用 wrapper），这个补丁把它统一成一种简洁的方式，减少构建过程中的意外问题。

📬 查看补丁
💡 小白科普
AutoFDO（自动反馈导向优化）：想象你每天开车上班都走同一条路，如果有人帮你把那条路拓宽了，你就会更快。AutoFDO 就是这个"拓宽工人"——它分析程序实际运行时的热点路径，然后让编译器重点优化那些代码。 📖 了解反馈导向优化
kbuild：Linux 内核的构建系统，负责把成千上万个源文件编译、链接成最终的内核镜像。它基于 Make，但有很多内核特有的扩展。 📖 内核 kbuild 文档

设备驱动 · 串口设备 RUST 支持

设备驱动 · 串口设备 Rust 支持

▸ serdev 设备添加 Rust 私有数据支持 v3
Markus Probst 提交了第 3 版补丁，为 serdev_device（串口设备）添加了 Rust 私有数据（private data）的支持。在内核驱动开发中，"私有数据"是一个非常常见的模式：每个设备实例都需要保存一些自己专属的状态信息（比如当前波特率、缓冲区指针等）。这个补丁包含 4 个子补丁，由 Danilo Krummrich 主导设计，Greg Kroah-Hartman（内核稳定版维护者、设备模型的创始人之一）也参与了审阅。这说明 Rust 驱动框架正在逐步覆盖更多的设备类型——从 GPU 到串口设备，Rust 的触角在不断延伸。

📬 查看补丁
💡 小白科普
serdev（Serial Device，串口设备）：串口是一种很基础的通信接口，蓝牙模块、GPS 模块、一些传感器都通过串口和主芯片通信。serdev 是 Linux 内核中管理这类设备的子系统。 📖 内核 serdev 文档
驱动私有数据（Private Data）：就像每个人有自己的口袋装自己的东西——每个设备驱动实例也需要一个"口袋"来存放只属于它自己的运行状态。内核通过指针把这个"口袋"挂在设备结构体上。 📖 内核驱动设计模式
