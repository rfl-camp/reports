️ GPU 驱动 · NOVA CORE（NVIDIA 开源驱动）

️ GPU 驱动 · Nova Core（NVIDIA 开源驱动）

Hopper/Blackwell 架构：PCI 配置镜像位置更新 (v10)
John Hubbard 提交了一组 28 个补丁的第 10 版。这组补丁的核心工作是：在 NVIDIA 最新的 Hopper 和 Blackwell 架构 GPU 上，找到并使用新位置的 PCI 配置镜像。简单来说，GPU 作为一块 PCI 设备插在主板上，CPU 要和它通信需要读写一组"配置寄存器"。在老架构上，这些寄存器的"镜像副本"放在 GPU 内部的某个固定地址；但在新架构上，NVIDIA 把这个镜像搬了个地方。这组补丁就是让 Nova 驱动能正确找到新地址并读取配置信息。已经有 2 位审查者参与了代码评审，说明这组补丁正在逐步走向合并。

📬 查看补丁

Blackwell 架构：计算预留显存堆大小 (v10)
同样来自 John Hubbard 的另一组 28 个补丁（第 10 版）。这组补丁处理的是 Blackwell 架构 GPU 上的显存管理问题。GPU 有自己的显存（Frame Buffer，简称 FB），其中一部分需要被"预留"出来给固件、内部硬件队列等使用，剩下的才能分配给用户程序和图形渲染。这组补丁实现了正确计算"到底需要预留多少显存"的逻辑。这对于驱动能否正确初始化 GPU 并高效利用显存至关重要——预留太多浪费资源，预留太少会导致硬件崩溃。已有 1 位审查者参与评审。

Zulip：NVIDIA 驱动问题已解决
在社区 Zulip 聊天中，此前有人在 #General 频道报告了一个 NVIDIA 驱动相关的问题。Nova 项目的核心维护者 Danilo Krummrich 已将该话题移至 #Nova 频道，并标记为已解决（resolved）。虽然具体问题细节没有在摘要中展开，但这说明 Nova 驱动的 bug 追踪流程在正常运转，维护者也在积极响应社区反馈。

RUST 内核基础设施 · 大小常量

Rust 内核基础设施 · 大小常量

新增 SZ_4G 常量
Deborah Brouwer 提交了一组 4 个补丁，为内核中的 Rust 代码添加一个名为 SZ_4G 的常量，表示 4GB（即 4,294,967,296 字节）。你可能会问：这不就是一个数字常量吗？为什么值得单独提一个补丁？原因是内核代码里经常需要用到各种固定大小（比如页大小、内存区域大小），用命名常量而不是"魔法数字"可以让代码更易读、更不容易出错。Linux 内核的 C 代码中早已有 SZ_4K、SZ_1M 等一系列常量，而 Rust 侧还缺少 4GB 这个级别的定义。这个补丁正是在补齐 Rust 内核代码的基础工具箱。已经有 1 位审查者参与了评审。

上游内核合并动态 · Linux 7.1-rc1

rust-next 分支今天同步了多个上游的合并提交，这些都是 Linux 7.1-rc1 合并窗口期间各子系统向 Linus Torvalds 的主线发起的 Pull Request。虽然这些合并本身不直接涉及 Rust 代码，但它们反映了 rust-next 分支正在与上游保持同步——这对于后续 Rust 补丁能顺利在最新内核上工作非常重要。以下是具体的合并内容：

PA-RISC 架构更新
来自 Helge Deller 的 PA-RISC 子系统更新。主要修复了 32 位 PA-RISC 架构上内核模块无法正常加载的问题，还去掉了 ip_fast_csum() 中的内联汇编以避免未对齐内存访问带来的 bug。PA-RISC 是惠普早年的处理器架构，现在很少见了，但内核仍然在维护对它的支持。

Memblock 内存管理更新
来自 Mike Rapoport 的 memblock 子系统更新。memblock 是内核在启动早期、正式内存管理器还没初始化之前使用的"简易内存分配器"。这次更新改进了 reserve_mem 内核参数的错误处理——如果预留内存失败，现在会打印更清楚的错误信息，还能通过 debugfs 查看实际预留了什么。就像一个启动日志的"调试模式"被打开了，方便开发者排查问题。

I2C 子系统更新 · 维护者交接
来自 Wolfram Sang 的 I2C 子系统更新。除了常规的驱动修复和改进外，这次 PR 中有一个重要消息：Wolfram Sang 将在 7.2 周期后将 I2C 子系统的维护权交给 Andi Shyti。I2C 是一种非常常见的硬件通信总线，你的笔记本里的温度传感器、触摸板控制器很可能都通过 I2C 连接。维护者的交接在开源社区是一件大事，意味着新的人将负责审查补丁和决定代码合入。

上游内核合并动态 · Linux 7.1-rc1

IPMI 子系统更新
来自 Corey Minyard 的 IPMI 子系统更新。IPMI（智能平台管理接口）是服务器上用来远程监控和管理硬件的协议——想象一下你可以在另一台电脑上远程查看服务器的温度、风扇转速甚至远程开关机。这次修复了一个宿主侧驱动的 bug：在某些情况下一个内核线程会在特定内存分配失败后被错误地保留运行，可能导致资源泄漏。

🔗 查看提交

perf 性能分析工具更新
来自 Namhyung Kim 的 perf 工具更新。perf 是 Linux 自带的性能分析工具，开发者用它来找出程序中哪些函数最耗时。这次更新中最有趣的新功能是 comm_nodigit 排序键：当你的程序有很多线程，名字只是末尾数字不同（比如 worker-1、worker-2、worker-3），现在可以把它们合并在一起分析，而不是分散成几十个独立条目，极大简化了多线程程序的性能分析。

社区与工具链

持续追踪：内核使用的 Rust 不稳定特性
GitHub 上有一个从项目初期就一直打开的跟踪 Issue（#2），由项目发起人 Miguel Ojeda 创建，标签为 toolchain 和 meta。这个 Issue 的作用是集中记录"内核 Rust 代码用到了哪些还没稳定的 Rust 语言特性"。为什么这很重要？因为 Rust 语言本身也在不断演化，有些功能虽然可以用但还标记为"不稳定"——意味着它们的接口将来可能会变。内核代码依赖这些特性，所以需要持续跟踪它们的稳定化进度。最终目标是所有用到的特性都进入 Rust 稳定版，这样编译内核就不再需要 nightly 编译器了。这个 Issue 今天仍然处于打开状态，说明还有一些特性没有完成稳定化。
