️ GPU 驱动 · Nova-core GA100 全面支持

今天 nova-core（NVIDIA 开源内核 GPU 驱动项目）的重头戏是 Timur Tabi 提交的 v4 版 GA100 支持系列补丁，一共 6 个补丁环环相扣，把 GA100 这块"特立独行"的 GPU 接入了 nova-core。GA100 是 NVIDIA 安培架构的一款纯计算卡（没有显示输出），但它的启动流程却跟上一代图灵架构一样，还有一些自己独特的怪癖——比如 VBIOS 前面会多出一个 IFR 头、FRTS 窗口大小必须设为 0 等等。下面逐个看看：

① 修正 GA100 固件签名

虽然 GA100 和图灵 GPU 使用同一套 GSP-RM 固件文件，但 GA100 有自己专属的数字签名。之前的代码错误地给 GA100 用了图灵的签名，这个补丁修正了这个问题。就好比两个人共用一把钥匙开门，但门锁其实是不一样的——现在给 GA100 配上了正确的"钥匙"。

📬 查看补丁

② 不再把 0xBB77 当作有效的 PCI ROM 签名

NVIDIA GPU 的 PCI 扩展 ROM 有些段落会使用 NVIDIA 自定义的签名（而非标准的 0xAA55）。其中 0xBB77 是一个已经废弃十多年的内部签名，nova-core 实际上不可能碰到使用这个签名的 GPU。这个补丁把对 0xBB77 的检查去掉了，让代码更干净、逻辑更明确。

📬 查看补丁

③ 跳过 IFR 头部

有些 GPU 的 ROM 开头并不是直接的 PCI 扩展 ROM 镜像，而是先有一个叫做 Init-from-ROM (IFR) 的头部。读取 VBIOS 时需要先解析这个头部，算出真正的 PCI ROM 起始偏移量，然后再进行后续读取。GA100 就是这样的 GPU。这个补丁给 PROM（可编程只读存储器）影子方法加上了对 IFR 头部的检测和跳过逻辑。

📬 查看补丁

④ FRTS 区域未分配时不启动 FWSEC-FRTS

在 GA100 上，FRTS（Firmware Reserved Table Storage）区域的大小为 0，意味着它没有被分配。如果仍然尝试启动 FWSEC-FRTS 固件安全模块，就会出错。这个补丁加了一个判断：只有当 FRTS 区域确实被分配了（大小 > 0），才执行启动流程。

⑤ 新增 FbHal::frts_size() 方法

为了让不同 GPU 能灵活地报告自己的 FRTS 窗口大小，引入了一个新的硬件抽象层（HAL）方法 frts_size()。GA100 的实现直接返回 0。值得注意的是，不能简单地用"是否支持显示输出"来判断 FRTS 大小，因为有些其他 GPU（如 GA102GL）虽然禁用了显示但 FRTS 大小不为 0。

GSP-RM（GPU System Processor - Resource Manager）：现代 NVIDIA GPU 内部有一个独立的小处理器叫 GSP，用来管理 GPU 资源。驱动需要先给 GSP 加载固件，然后通过它来控制 GPU。可以把它想象成 GPU 内部的一个"管家"。

RUST 内核基础设施

Rust 内核基础设施

① 新增 bitfield! 位域宏（v2）

在内核开发中，经常需要从一个整数的特定几位中提取或设置值——这叫做"位域操作"。C 语言有原生的位域语法，但 Rust 没有。Mark Brown、Danilo Krummrich 和 Miguel Ojeda 合作提交了一个 bitfield! 宏，让 Rust 内核代码也能方便地定义和操作位域。这个补丁收到了多达 11 条 review，说明社区对这个基础设施非常关注。位域宏是硬件驱动开发的基石——几乎所有硬件寄存器都是用位域来组织数据的。

📬 查看补丁

② XArray 新增 Guard::find() 查找助手

Alvin Sun 等人为 Rust 的 XArray 封装添加了一个 find() 方法。XArray 是 Linux 内核中一种高效的稀疏数组数据结构，常用来存储和查找内核对象（比如通过 ID 找到对应的页面或文件）。新的 find() 方法可以找到数组中第一个存在的条目并返回其索引，如果数组是空的就返回 None。这对即将到来的 Tyr 驱动中 ioctl 的实现至关重要。

③ 新增 SZ_4G 常量

📬 查看补丁
💡 小白科普
位域（Bitfield）：硬件寄存器通常是一个 32 位或 64 位的整数，但不同的位（bit）代表不同的含义。位域就是把一个整数的不同位段命名并单独访问的技术。比如一个 8 位寄存器，低 3 位表示速度，第 4 位表示开关——位域宏让你可以写 reg.speed() 和 reg.enabled() 来读取它们。

内存管理 · KREALLOC 数据丢失修复

内存管理 · krealloc 数据丢失修复

Harry Yoo（Oracle）提交了一个针对 SLUB 分配器中 krealloc() 函数的 bug 修复。krealloc() 的作用是"重新调整已分配内存的大小"——就像你买了一个盒子装东西，发现太小了，想换一个大一点的，同时把原来的东西搬过去。

这个 bug 可能导致两个严重问题：数据丢失和溢出。也就是说，在某些情况下，搬家的时候可能会弄丢一部分数据，或者新盒子的大小计算出错。Marco Elver 参与了 review，这个修复对内核稳定性非常重要，因为 krealloc() 在内核各处被大量使用。虽然这不是 Rust 特有的问题（这是 C 代码的 bug），但它出现在 Rust-for-Linux 邮件列表上，说明这个修复可能与 Rust 代码的内存分配路径有关。

SLUB 分配器：Linux 内核默认的内存分配器。内核代码需要频繁地申请和释放小块内存，SLUB 就是负责高效管理这些内存块的组件。krealloc() 是它提供的一个功能，类似于 C 标准库中的 realloc()。

安全协议 · SPDM 的 RUST 实现讨论

安全协议 · SPDM 的 Rust 实现讨论

Alistair Francis 提出了一个重量级的 RFC（征求意见稿）：用 Rust 在 Linux 内核中实现 SPDM（Security Protocol and Data Model） 协议。这个 RFC 已经到了 v3 版，包含多达 27 个补丁。

SPDM 是一种安全协议，用于在设备和主机之间建立可信的通信通道。简单来说，它能让你的电脑在使用一个硬件设备之前，先验证这个设备"是不是假冒的"，并建立加密通信。这对数据中心、边缘计算和物联网场景非常关键。

Lukas Wunner 和 Dan Williams 也参与了讨论。用 Rust 实现 SPDM 的一个核心优势是 Rust 的内存安全特性——安全协议的实现如果出了内存 bug，那安全性就形同虚设了。这个项目如果成功合入，将成为内核中最大的 Rust 安全相关子系统之一。

💡 小白科普
SPDM（Security Protocol and Data Model）：DMTF 组织定义的一套安全协议，主要用于设备身份验证和安全通信。想象你在网上购物时浏览器会检查网站的 SSL 证书——SPDM 做的是类似的事情，只不过是在硬件设备之间进行的。
RFC（Request For Comments）：在内核社区，RFC 表示"我有一个想法/方案，请大家提意见"。它是正式补丁之前的讨论阶段。

代码生成 · YNL 生成 RUST 代码

代码生成 · YNL 生成 Rust 代码

Alice Ryhl 和 Donald Hunter 正在推进 ynl_gen 工具的 v3 版本，这个工具可以从 YAML 规范文件自动生成 Rust 代码。这是什么意思呢？

Linux 内核的网络子系统使用一种叫 Netlink 的机制让用户空间程序和内核通信。定义 Netlink 接口时，需要写大量模板代码——参数怎么序列化、消息怎么解析等等。YNL（YAML Netlink）项目用 YAML 文件来描述这些接口，然后用代码生成器自动生成对应的代码。之前只能生成 C 和 Python 代码，现在这个补丁系列让它也能生成 Rust 代码了！

这对 Rust 网络驱动开发者来说是个好消息——他们不再需要手动编写大量繁琐的 Netlink 接口代码了。

📬 查看补丁
💡 小白科普
Netlink：Linux 内核提供的一种通信机制，让用户空间程序（比如 ip 命令）能和内核网络子系统"对话"。你用 ip addr 查看网卡信息时，底层就是通过 Netlink 和内核通信的。
代码生成（Code Generation）：用程序自动生成代码，避免手写大量重复的模板代码。就像用模板批量打印文件，而不是一份一份手抄。

① call_rcu() 延迟释放机制大讨论（44 条消息）

这是今天 Zulip 上最热烈的讨论！Philipp Stanner、Gary Guo 和 Alice Ryhl 围绕 call_rcu() 展开了深入交流。

背景是这样的：Philipp 在开发 DRM（图形显示）驱动中的 fence（栅栏/同步原语）功能时，遇到了一个棘手的问题——他有一个用纯 Rust 编写的 DriverFence::data 结构体，其中一些 C 回调函数（虽然实际上是 Rust 实现的）可能仍在通过 RCU 保护机制访问这个数据。当引用计数降到 0 时，不能立即释放这个数据，而是需要"延迟释放"——等所有可能在读这个数据的代码都执行完了再释放。

Alice 问到 C 的 fence 结构体是否已经有 rcu_head 字段。Philipp 回答说虽然有类似的东西，但在引用计数归零之前似乎无法使用。这说明 Rust 封装需要自己管理 RCU 延迟释放的生命周期，这是一个关乎 Rust 内核安全抽象设计的重要问题。

💡 小白科普
RCU（Read-Copy-Update）：一种内核中常用的同步机制。它的核心思想是：读操作几乎没有开销（不需要加锁），而写操作则通过"先复制一份、修改副本、然后等所有读者都不再使用旧数据后才释放旧数据"来保证安全。call_rcu() 就是那个"等所有读者都走了再释放"的函数。
DRM（Direct Rendering Manager）：不是"数字版权管理"！在 Linux 内核语境下，DRM 是图形显示子系统，负责管理 GPU 和显示输出。所有 Linux GPU 驱动都是 DRM 驱动。

Rust 内核所需的不稳定特性跟踪（GitHub Issue #2）

Rust-for-Linux 项目在 GitHub 上维护着一个长期 Issue（#2），追踪所有"Rust 语言中还不稳定但内核开发必须用到的特性"。这是一个 meta issue——也就是汇总性质的 issue。由项目维护者 ojeda 创建，标记为 toolchain 和 meta。

为什么这很重要？因为 Rust 语言有些功能还处于"实验阶段"（unstable），正式项目一般不建议使用。但内核开发有些场景确实需要这些功能（比如内联汇编、自定义分配器等）。这个 issue 就是记录"我们到底依赖了哪些不稳定特性"，同时也推动 Rust 官方尽快把这些特性稳定化。

🔗 查看 Issue
💡 小白科普
Unstable Features（不稳定特性）：Rust 语言的新功能会先作为"不稳定特性"发布，只能在 nightly（每日构建）版编译器中通过 #![feature(...)] 启用。经过足够的测试和反馈后，它们才会被"稳定化"，进入正式发布版。内核使用 nightly Rust 编译器，因此可以使用这些特性，但也需要承担它们可能变化的风险。
