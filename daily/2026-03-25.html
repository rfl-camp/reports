覆盖了哪些寄存器模块：

① GPU_CONTROL 寄存器转换 — 把 GPU 控制相关的寄存器定义全部迁移到 register! 宏。手动的位掩码和位移操作被替换为带类型的寄存器和字段访问器，编译器能帮你检查类型是否正确，避免了手滑写错位的风险。

② GPU_ID 打印改进 — 以前 Tyr 只打印 GPU_ID 寄存器的高 16 位（架构主版本、副版本等），需要手动位移。现在改为直接打印完整的 GPU_ID 寄存器值，不仅省去了位移操作，还多输出了产品副版本号和状态信息，调试时能看到更完整的 GPU 身份信息。

③ 互联一致性协议探测 — 以前 GpuInfo 直接写死了互联一致性协议为"无"，虽然结果碰巧是对的，但做法不严谨。现在改为在驱动探测（probe）阶段先写入 NO_COHERENCY，然后再读回寄存器来填充 GpuInfo。这就像"先把开关拨到关，再确认一下确实是关的"，比直接假设它是关的要可靠得多。

NVIDIA NOVA 核心驱动：修复命令队列中的未定义行为

NVIDIA Nova 核心驱动：修复命令队列中的未定义行为

NVIDIA 的 Alexandre Courbot 和 Danilo Krummrich、Gary Guo 一起审核了 Nova 核心驱动中 GSP（GPU System Processor）命令队列代码的一个 bug 修复（v2 版本）。

问题是什么？ 在 Nova 核心驱动与 GPU 系统处理器通信的命令队列代码中，存在"未定义行为"（Undefined Behavior，简称 UB）。在 Rust 中，UB 是最严重的代码问题之一——它意味着编译器对这段代码的行为不做任何保证，可能今天正常运行，明天就崩溃，或者更糟，悄无声息地产生错误结果。

为什么重要？ Nova 是 NVIDIA 官方用 Rust 编写的新一代开源 GPU 驱动，社区非常关注它的代码质量。修复 UB 不仅是消除一个 bug，更是确保 Rust 在内核中"说到做到"——即 Rust 代码确实比 C 代码更安全。这个补丁已经是第二版，说明审核者对细节要求很高。

DMA 内存管理：一致性内存 API 大幅增强

DMA 内存管理：一致性内存 API 大幅增强

Andreas Hindborg 今天在 DMA（直接内存访问）子系统的 Rust 封装上推进了多项工作，Danilo Krummrich 和 Gary Guo 参与了审核。这些补丁共同构建了一套更完善的 DMA 一致性内存管理接口。

① Coherent 和 CoherentInit API（8 个补丁） — 引入了 dma::Coherent 和 dma::CoherentInit 两个新的 API。"一致性内存"是 CPU 和设备（比如网卡、GPU）都能同时访问的一块特殊内存，系统会自动保证双方看到的数据是一致的。这套新 API 让 Rust 驱动开发者能更方便、更安全地申请和初始化这种内存。

② 通用 DMA 容器（不限于切片，v2） — 以前 DMA 的 Rust 封装只能管理"切片"（连续的同类型数据数组）类型的内存。这个补丁让它支持更多类型，比如一个包含多种字段的结构体。这就像以前你只能用 DMA 搬运"一箱苹果"，现在可以搬运"一个装了各种东西的百宝箱"。Gary Guo 参与了审核，确保泛型实现的正确性。

⚙️ 工作队列：新增 AREF 智能指针支持

⚙️ 工作队列：新增 ARef 智能指针支持

来自 Collabora 的 Daniel Almeida 提交了第三版补丁（4 个补丁），为内核工作队列添加 ARef<T> 智能指针支持。Alice Ryhl（Google）参与了审核。

什么是工作队列？ 想象一下你在餐厅点了菜，厨师不会站在你面前做完再去服务下一桌。他会把你的订单放到"待办队列"里，按顺序做。内核的工作队列也是类似的——驱动把"要做的事情"扔进队列，内核的工作线程会在合适的时候依次处理。

这次改了什么？ 以前工作队列只支持用 Arc<T>（原子引用计数指针）来管理工作项的生命周期。现在新增了对 ARef<T> 的支持。ARef 是内核 Rust 中的"always-refcounted"指针，它和 Arc 类似，但适用于那些引用计数由 C 侧管理的对象。补丁 1 和 3 实现了核心的 workqueue.rs 改动，补丁 2 和 4 为 DRM 设备添加了对应的实现，将调用分派给底层的 T::Data。

为什么重要？ 这个功能已经在 Tyr 驱动上实测验证过，说明它是从实际需求出发的改进——有些驱动场景下，必须用 ARef 而不是 Arc 来持有工作项的引用，否则无法正确管理对象的生命周期。

💡 小白科普
工作队列（Workqueue）：内核中一种延迟执行机制。当驱动需要做一些"不急着马上完成"的事情时，可以把任务放进工作队列，由内核线程稍后处理。📖 内核文档
ARef / Arc：都是"引用计数智能指针"。当多个地方需要共享同一个对象时，引用计数会追踪"还有几个人在用它"，最后一个用完时自动释放。📖 Rust Arc 文档

电源管理：修复稳压器抽象的未定义行为

电源管理：修复稳压器抽象的未定义行为

Google 的 Alice Ryhl 提交了一个关键的 bug 修复，Daniel Almeida 和 Miguel Ojeda（Rust-for-Linux 项目维护者）参与了审核。

出了什么问题？ Rust 的 Regulator（稳压器）抽象使用 NonNull 来包装底层 C 的 struct regulator 指针。NonNull 顾名思义，它承诺"这个指针永远不会是 null"。但是！当内核配置中关闭了 CONFIG_REGULATOR（即不需要稳压器功能）时，C 侧的 regulator_get() 函数会返回 NULL。

关键细节： from_err_ptr 这个辅助函数不会把 NULL 当作错误（它只识别 Linux 的 ERR_PTR 编码），所以 NULL 被悄悄地传给了 NonNull::new_unchecked，违反了它"不能是 null"的约定——这就是未定义行为（UB）！在某些编译优化下，这可能导致无法预测的崩溃或错误。

修复方案： 不再假设 regulator_get() 一定返回非 null 值，而是正确处理 null 的情况。这个 bug 提醒我们：即使在 Rust 中，与 C 代码交互时也需要格外小心边界条件。

💡 小白科普
Regulator（稳压器）：管理硬件电源供应的内核子系统。手机、开发板上的各种芯片需要不同的电压，稳压器负责给它们提供正确的电力——就像家里的变压器把 220V 转成手机充电器需要的 5V。📖 内核文档
NonNull：Rust 标准库中的一个类型，表示"保证不为 null 的指针"。在 unsafe 代码中必须由程序员保证这个约定。📖 Rust 文档

️ ACPI 子系统：新增设备 ID 匹配接口

️ ACPI 子系统：新增设备 ID 匹配接口

Markus Probst 提交了第三版（v3）的 ACPI 补丁系列（7 个补丁），Rafael J. Wysocki（ACPI 子系统维护者）参与了审核。

这是在做什么？ 新增了一个 Rust 封装的 acpi_of_match_device_ids 函数。这个函数的作用是让驱动能根据 ACPI 表和设备树（Device Tree）中的设备 ID 来"认出"硬件。打个比方：你的电脑开机时，BIOS/UEFI 会提供一份"硬件清单"（ACPI 表），这个函数就是让驱动拿着"准考证"去和清单对比，找到自己该管理的设备。

为什么重要？ 设备匹配是驱动正常工作的第一步——驱动必须先认出"这是我的设备"才能开始工作。把这个功能用 Rust 封装好，意味着用 Rust 写的驱动可以更方便地支持 ACPI 设备发现，这对于在 x86 平台（台式机、笔记本）上运行 Rust 驱动至关重要。
