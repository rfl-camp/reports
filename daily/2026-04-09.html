GPU/DRM 驱动：Nova Core 大步前进

统一页表条目封装枚举 (v10，共 21 个补丁)

今天 nova-core（NVIDIA 开源 GPU 内核驱动）是最活跃的领域，多达六组补丁同时在推进。这个驱动是用 Rust 编写的，目标是替代现有的闭源 NVIDIA 驱动，是整个 Rust-for-Linux 生态中最受关注的项目之一。

Joel Fernandes 和 Eliot Courtney 提交了第 10 版补丁，为 nova-core 的内存管理模块（mm）添加了统一的页表条目（PTE）封装枚举。你可以把页表想象成 GPU 的"地址簿"——GPU 要访问显存中的数据，就需要通过页表把虚拟地址翻译成物理地址。这组补丁把不同类型的页表条目（比如大页、小页、无效页）统一用 Rust 的枚举类型来表示，让代码更类型安全、更不容易出错。已经迭代到第 10 版，说明社区对代码质量要求很高，审查非常仔细。

Hopper/Blackwell 固件支持 (v9，共 31 个补丁)

Alexandre Courbot 提交了一组 31 个补丁的系列，为 nova-core 添加对 NVIDIA Hopper（H100 等数据中心 GPU）和 Blackwell（最新一代）架构的固件支持。这意味着这个开源驱动正在向支持最新 NVIDIA 硬件迈进。固件（firmware）就像是显卡的"操作系统"，驱动需要正确加载固件才能让 GPU 工作。这组补丁已经审查到第 9 版，由 John Hubbard 参与评审。

计算预留显存堆大小 (v9)

同样是 Hopper/Blackwell 系列的一部分，这个补丁专门处理预留的显存（Frame Buffer）堆大小计算。GPU 启动时需要预留一部分显存给固件和安全启动使用，这个补丁确保对新架构能正确计算这个大小。就像你买了一块 24GB 显存的显卡，实际可用的会少一点，因为有一部分要留给系统用。

FSP 安全启动完成等待 (v9)

FSP（Falcon Security Processor）是 NVIDIA GPU 内部的一个安全处理器，负责验证固件的签名、确保启动过程没有被篡改。这个补丁添加了等待 FSP 安全启动完成的逻辑——驱动在加载固件后需要"等一等"，确认安全处理器说"一切正常"才能继续。这就像手机开机时要先通过指纹验证才能进入系统一样。

WPR 堆大小计算改为可失败 (v9)

WPR（Write Protected Region）是 GPU 显存中一块写保护区域，用来存放敏感的固件数据。这个补丁把 WPR 堆大小的计算从"不会失败"改成了"可能失败"（fallible）。在 Rust 中，这意味着用 Result 类型来处理错误，而不是直接 panic。这是 Rust 编程的最佳实践——承认计算可能出错，并优雅地处理错误，而不是让整个内核崩溃。

PCI 配置镜像新位置 (v9)

Hopper 和 Blackwell 架构改变了 PCI 配置空间镜像（mirror）的位置。PCI 配置空间是 CPU 与 GPU 通信的"联络窗口"，驱动通过读写这个空间来控制 GPU。新架构把这个镜像挪到了不同的地址，这个补丁确保驱动能找到正确的位置。

ACPI 设备匹配修复

ACPI 设备匹配修复

修复 PRP0001 设备的匹配数据缺失 (v4)

Markus Probst 提交了第 4 版补丁，修复了一个困扰 Rust 驱动的 ACPI 设备匹配问题。PRP0001 是 ACPI 规范中的一个特殊设备 ID，它允许 ACPI 设备使用设备树（Device Tree）风格的匹配方式。简单来说，当硬件通过 ACPI 告诉系统"我是 PRP0001 类型的设备"时，驱动需要进一步查看设备的兼容性信息来决定该怎么处理它。

问题出在 Rust 驱动中，id_info（设备匹配信息）在 PRP0001 设备上会是 None（空值）。这是因为 Rust 驱动在 OF（Open Firmware）设备 ID 表中存的是索引值而不是数据指针，所以没法直接用 device_get_match_data 来获取匹配数据。补丁的解决方案是导出 C 内核中的 acpi_of_match_device 函数，让 Rust 代码能直接用它来匹配。这个补丁得到了 Rafael J. Wysocki（ACPI 子系统维护者）和 Danilo Krummrich 的审查，共有 6 条 review 意见。

️ 内存安全工具：KASAN 拥抱 RUST

️ 内存安全工具：KASAN 拥抱 Rust

为 Rust 添加软件标签 KASAN 支持 (v3，共 2 个补丁)

Alice Ryhl（Google）提交了第 3 版补丁，为 Rust 代码添加了基于软件标签的 KASAN（Kernel Address SANitizer）支持。KASAN 是内核中的"内存错误侦探"，能在运行时检测到越界访问、使用已释放内存等危险行为。虽然 Rust 本身就以内存安全著称，但内核中 Rust 代码经常需要通过 unsafe 块与 C 代码交互，这时候就需要 KASAN 来"兜底"了。

软件标签模式是 KASAN 的一种轻量实现，通过给每块内存打上"标签"（tag），访问时检查标签是否匹配来发现错误。这比硬件标签模式开销更小，适合在开发和测试中使用。Gary Guo 参与了审查。

📬 查看补丁
KASAN+Rust 必须使用 Clang 编译 (v3)

Alice Ryhl 同时提交的另一个补丁明确了一个构建限制：当同时启用 KASAN 和 Rust 时，必须使用 Clang 编译器（不能用 GCC）。这是因为 KASAN 的软件标签模式需要编译器在内存操作前后插入检查代码，而目前只有 Clang 对 Rust 代码的 KASAN instrumentation 有良好支持。Miguel Ojeda 参与了审查。

⚙️ 内核基础设施：SEQ_FILE、工作队列与 IO_URING

⚙️ 内核基础设施：seq_file、工作队列与 io_uring

seq_file 新增 puts/putc/write/hex_dump 方法 (v2)

Christian Benton 提交了第 2 版补丁，为 Rust 的 seq_file 抽象新增了四个方法：puts（输出字符串）、putc（输出单个字符）、write（写入字节数组）和 hex_dump（输出十六进制转储）。seq_file 是内核中用于在 /proc 文件系统输出信息的机制——比如你运行 cat /proc/cpuinfo 看到的内容，就是通过 seq_file 输出的。

补丁作者指出，现有的 Rust Binder 代码（drivers/android/binder/）中很多地方用 seq_print! 宏只是为了输出静态字符串，用 puts 会更高效，因为不需要格式化处理。这版补丁修复了 Alice Ryhl 的审查反馈和 kernel test robot 报告的构建问题。

📬 查看补丁
工作队列：安全的取消和状态查询方法 (v6，共 3 个补丁)

Aakash Bollineni 提交了第 6 版补丁，为 Rust 的工作队列（workqueue）抽象添加了安全的取消和状态查询方法。工作队列是内核中的"任务调度中心"——当你需要推迟执行某些操作（比如"等网络数据到了再处理"），就把任务放到工作队列里。这个补丁让你可以安全地取消一个已经排队但还没执行的任务，并查询任务的当前状态（等待中？执行中？已完成？）。"安全"意味着这些操作不需要 unsafe 代码，Rust 编译器会帮你检查正确性。

Sidong Yang（Furiosa AI）提交了第 4 版补丁，在 Rust 的 bindings_helper.h 中添加了 io_uring 的头文件引用。io_uring 是 Linux 中高性能异步 I/O 的核心机制——想象一下，你让内核帮你同时处理几千个文件读写请求，而不用每次都等上一个完成。这个补丁是为 Rust 代码调用 io_uring 功能做准备的第一步，让 Rust 能"看到" io_uring 的 C 数据结构和函数。Miguel Ojeda 参与了审查。

pin-init v7.1 合并

Miguel Ojeda 将 Benno Lossin 的 pin-init v7.1 更新合并到了 rust-next 分支。主要变化包括：把 Option<NonZero*> 类型的 Zeroable 实现替换为 NonZero* 类型的 ZeroableOption 实现（更精确的类型约束），以及改进了不稳定特性（unstable features）的 feature gate 处理。pin-init 是 Rust 内核中的一个核心库，用于安全地初始化被"钉住"（pinned）的数据结构——在内核中，很多数据结构一旦初始化就不能移动位置（比如锁、链表节点），pin-init 确保这些初始化过程是安全的。

Tyr 驱动：新增 VM 和 BO ioctl 支持

trace_printk 的 Rust 支持在哪里？

Philipp Stanner 在 Zulip 的 #General 频道提出了一个问题：Rust 中还没有 trace_printk 的支持，有人在做吗？trace_printk 是内核中的一个调试工具——它类似于 printk（内核的 printf），但输出到内核的追踪缓冲区（trace buffer）而不是控制台日志，性能开销更小，非常适合在性能敏感的代码路径中插入调试信息。目前 Rust 代码只能用 pr_info! 等宏输出到内核日志，还不能用追踪系统。这是一个值得关注的功能缺口。
