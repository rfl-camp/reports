1. 为 GA100 添加 FbHal::frts_size() 支持（v5，6 个补丁）

今天 nova-core 项目（NVIDIA 正在开发的全新开源 GPU 内核驱动，用 Rust 编写）一次性提交了大量补丁，主要围绕对 Hopper（H100 等） 和 Blackwell（B200 等） 两代最新 GPU 架构的支持。可以把这看作是给一栋新大楼搭建不同楼层的地基——每组补丁负责一个子系统。下面逐一介绍：

1. 为 GA100 添加 FbHal::frts_size() 支持（v5，6 个补丁）

这组补丁由 Timur Tabi 和 Gary Guo 等人协作完成，为 GA100（Ampere 架构数据中心 GPU）的帧缓冲硬件抽象层（FbHal）添加了一个新方法 frts_size()。FRTS（Fault Recovery Table Size）是 GPU 用来记录和恢复显存故障信息的表格大小。简单来说，就像给 GPU 的"内存体检报告"指定了存放空间。这个补丁已经经过了 6 轮评审，Alexandre Courbot 也参与了讨论，整体质量比较成熟。

2. Hopper/Blackwell：更大的 non-WPR 堆（v10，28 个补丁）

2. Hopper/Blackwell：更大的 non-WPR 堆（v10，28 个补丁）

WPR（Write-Protected Region）是 GPU 上一块受保护的内存区域，用于存放安全固件。non-WPR 堆则是不受保护的普通内存堆。Hopper 和 Blackwell 架构的 GPU 需要更大的 non-WPR 堆来存放初始化数据和临时缓冲。这组补丁把堆的大小限制放宽了，好比给新一代的 GPU "仓库"扩容，让它有足够空间存放启动时需要的各种数据。这已经是第 10 版修订，说明社区对内存布局的正确性非常谨慎。

3. Hopper/Blackwell：PCI 配置镜像的新位置（v10，28 个补丁）

3. Hopper/Blackwell：PCI 配置镜像的新位置（v10，28 个补丁）

PCI 配置空间是 CPU 与 GPU 沟通的"名片夹"，里面记录了设备的各种基本信息。在新架构上，这个配置空间的镜像（mirror）被搬到了一个新地址。这组补丁就是告诉驱动"去新地址找配置信息"。这就好比你搬了家，得告诉快递员新地址一样。v10 版本意味着这个地址映射方案经过了反复验证。

4. 跳过 IFR 头部（v5，6 个补丁）

4. 跳过 IFR 头部（v5，6 个补丁）

IFR（Internal Firmware Region）头部是 GPU 固件镜像开头的一段元数据。某些 GPU 固件版本会带有这个头部，而驱动在解析固件时需要正确跳过它，否则会把元数据误当成固件代码来执行，导致 GPU 初始化失败。Eliot Courtney 提交的这组补丁让驱动在检测到 IFR 头部时自动跳过，相当于给固件解析加了一个"智能跳转"功能。

5. Hopper/Blackwell：更大的 WPR2（GSP）堆（v10，28 个补丁）

5. Hopper/Blackwell：更大的 WPR2（GSP）堆（v10，28 个补丁）

GSP（GPU System Processor）是 NVIDIA 近几代 GPU 上的一个专用处理器，负责运行 GPU 的安全固件。WPR2 堆是 GSP 固件专用的受保护内存区域。在 Hopper 和 Blackwell 上，GSP 固件变得更大更复杂，因此需要更大的 WPR2 堆。这组补丁增加了堆的容量上限，确保 GSP 固件能正常加载和运行。

7. Blackwell：使用正确的 sysmem flush 寄存器（v10，28 个补丁）

sysmem flush 寄存器用于确保 GPU 和系统内存之间的数据一致性——当 GPU 写完数据后，需要"刷新"到主内存，让 CPU 能看到最新内容。Blackwell 架构上这些寄存器的地址与之前的架构不同，如果用错了地址，就会导致数据不一致甚至系统崩溃。这组补丁修正了 Blackwell 专用的寄存器地址。

8. 用 GPU 架构简化 HAL 选择（v10，28 个补丁）

8. 用 GPU 架构简化 HAL 选择（v10，28 个补丁）

HAL（硬件抽象层）是驱动中用来屏蔽不同 GPU 型号差异的一层代码。之前 nova-core 可能需要为每个具体 GPU 型号写不同的 HAL 实现，而 Gary Guo 的这组补丁引入了"GPU 架构"的概念，让同一架构下的 GPU 共享同一套 HAL 代码。这大大减少了代码重复——就像手机 App 不用给每个型号做一个版本，只需要按"Android 14""Android 15"来适配。

9. Hopper/Blackwell：基本 GPU 识别（v10，28 个补丁）

在驱动初始化的最开始，需要先弄清楚"我面前的这块 GPU 到底是什么型号、什么架构"。Gary Guo 的这组补丁为 Hopper 和 Blackwell 添加了基本的 GPU 识别逻辑，读取 GPU 的标识寄存器并匹配到具体的架构和型号。这是所有后续初始化步骤的前提——得先认出硬件，才能加载对应的固件和配置。

10. 为 MapleTree 实现 Send 和 Sync（与 nova-core 相关）

10. 为 MapleTree 实现 Send 和 Sync（与 nova-core 相关）

Daniel Almeida 和 Joel Fernandes（来自 NVIDIA）发现 nova-core 的虚拟内存管理器（VMM）使用了 Maple Tree 数据结构，但由于 C 语言的 maple_tree 结构体内部包含一个 *mut c_void（裸指针），Rust 编译器无法自动推导出 Send 和 Sync trait，导致编译报错——因为 PCI 驱动框架要求核心结构体能安全地在线程间传递。这个补丁通过手动声明 unsafe impl Send 和 unsafe impl Sync（并附带安全论证）来解决这个问题。这在 Rust 内核开发中是一个常见的模式：C 数据结构本身是线程安全的，但 Rust 编译器看不懂 C 的并发契约，需要人工"签保证书"。

PCI（Peripheral Component Interconnect）是电脑里连接显卡、网卡、SSD 等设备的标准总线。PCI 设备有"能力"（capability）的概念，描述设备支持哪些高级功能。"扩展能力"（extended capability）是 PCIe 引入的更多功能描述，存放在配置空间的高地址区域。

Zhi Wang 的这组补丁为内核的 Rust PCI 抽象添加了读取扩展能力和操作 SR-IOV 的接口。有了这些接口，未来用 Rust 编写的 PCI 设备驱动就能方便地查询设备支持的高级特性，并启用虚拟化功能。Gary Guo 也参与了评审。

SRCU（Sleepable Read-Copy-Update）是 RCU 的一个变种。普通的 RCU 读端不能睡眠（即不能做可能阻塞的操作，比如等磁盘 I/O），但 SRCU 允许读端在临界区内睡眠。这在某些需要长时间持有读权限的场景中非常有用。Onur Özkan 的这组补丁为 Rust 代码提供了 SRCU 的安全抽象，让 Rust 驱动开发者可以方便地使用这种同步机制，而不用直接调用 C 函数。第 2 版说明作者根据社区反馈做了改进。

2. slab 分配器：编译器辅助的类型隔离分区（v2，Harry Yoo / Oracle）

2. slab 分配器：编译器辅助的类型隔离分区（v2，Harry Yoo / Oracle）

slab 分配器是内核中最常用的内存分配器之一，专门用于分配固定大小的小对象（比如文件描述符、inode 等）。这个补丁引入了一项安全增强功能：利用编译器的帮助，按对象类型自动将 slab 缓存分区隔离。这意味着不同类型的内核对象会被存放在不同的内存区域中。这样做有什么好处？如果攻击者通过漏洞破坏了某个对象，由于它和其他类型的对象物理隔离，攻击者就更难利用这个漏洞去篡改其他关键数据。Harry Yoo（来自 Oracle）和 Marco Elver 共同参与了这项工作，已经收到 3 条评审意见。

1. container_of 与泛型类型的困惑（已解决）

1. container_of 与泛型类型的困惑（已解决）

Philipp Stanner 在 Zulip 的 #Help 频道提出了一个关于 container_of 宏与泛型类型的问题。背景是这样的：在 DMA fence（GPU 同步机制）的回调中，C 代码只会传进来一个 C 结构体的指针，但 Rust 驱动需要从这个 C 指针"反推"出包含它的 Rust 结构体。container_of 宏就是干这个事的——它通过指针运算，从结构体的某个成员指针算出整个结构体的指针。Philipp 最初担心当外层结构体带有泛型参数时，container_of 无法获取正确的类型信息。但经过思考后他发现，在 trait 的默认方法实现中，Self 类型已经完整编码了所有泛型参数，所以不需要额外的 trait 来传递泛型信息。问题自己就解决了，他标记了话题为"已解决"。

Tyr-for-upstream 分支强制更新（Deborah Brouwer）

Tyr 是 Rust-for-Linux 生态中专注于 DRM（Direct Rendering Manager，Linux 的图形显示子系统）的一个项目分支。Deborah Brouwer 通知社区 tyr-for-upstream 分支进行了一次强制更新（force push），包含了两项重要变更：一是移除了 DRIVER_GEM_GPUVA 标志——GEM（Graphics Execution Manager）是 DRM 中管理 GPU 内存的组件，GPUVA 是 GPU 虚拟地址管理的接口，这个标志的移除说明相关的实现方式可能发生了架构调整。二是使用 register! 宏添加了所有全局 CSF（Command Stream Frontend）接口。CSF 是 ARM Mali GPU 使用的命令提交前端，register! 宏则是 Rust-for-Linux 中用于声明式地注册内核组件的工具。这些变更表明 Tyr 项目在积极推进其 DRM 驱动的 Rust 化工作。
