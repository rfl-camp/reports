️ GPU / DRM 驱动（NOVA-CORE）

️ GPU / DRM 驱动（nova-core）

🔒 VBIOS 安全加固系列（共 11 个补丁）

今天 nova-core（NVIDIA 开源内核 GPU 驱动的 Rust 核心模块）迎来了大量更新，集中在 VBIOS 解析的安全加固和 GA100 GPU 支持两个方向。

由 Eliot Courtney（NVIDIA）提交的 v2 版本补丁系列，是今天最大的一组改动。VBIOS 是显卡自带的固件，内核在启动 GPU 时需要读取并解析它。问题在于：固件里的偏移量、长度等数据都是"外部输入"，如果固件被篡改或本身有缺陷，直接按照这些值去访问内存数组，可能会越界访问，轻则数据错误，重则内核崩溃（panic）。

这组补丁的核心思路是：把所有基于固件数据的数组访问、算术运算都换成"带检查"的版本。比如用 checked_add() 代替普通加法，用 .get() 代替直接下标访问，这样即使固件数据有问题，也只会返回错误而不是让整个内核崩掉。

📬 查看补丁系列
🆕 GA100 GPU 支持：添加 FbHal::frts_size()

Timur Tabi（NVIDIA）提交了 6 个补丁，为 nova-core 的帧缓冲硬件抽象层（FbHal）添加了 frts_size() 方法，以支持 GA100 系列 GPU。GA100 是 NVIDIA Ampere 架构的数据中心 GPU（A100 就是这个系列），FRTS（Fault Recovery Table Structure）是 GPU 内存管理中用于错误恢复的关键数据结构。这个补丁让驱动能够正确查询 FRTS 的大小，是支持这类 GPU 的基础工作之一。

📬 查看补丁
⚡ 仅在 FRTS 存在时才启动

同样来自 Timur Tabi 的补丁，修复了一个逻辑问题：之前的代码会无条件地尝试启动 FRTS，但有些 GPU 型号根本没有这个结构。现在改为先检查 FRTS 是否存在，只有存在时才执行启动流程。这避免了在不支持 FRTS 的 GPU 上出现不必要的错误或异常行为。

构建系统与工具链

🐉 为龙芯架构添加 Clang 和 Rust 编译支持

由 Haiyong Sun（龙芯中科）提交，WANG Rui 等参与审核。这个补丁为 perf（Linux 内核自带的性能分析工具）的构建系统补上了 LoongArch（龙芯）架构缺少的两个编译标志：CLANG_TARGET_FLAGS_loongarch 和 RUST_TARGET_FLAGS_loongarch。

简单来说，之前如果你想在龙芯处理器上用 Clang 编译器编译 perf 工具，或者用 Rust 进行交叉编译，构建系统不知道该怎么设置目标架构参数，所以会失败。这个补丁把缺少的参数补上了，让龙芯架构也能享受到 Clang + Rust 的编译支持。这对推动国产处理器生态融入 Rust-for-Linux 很有意义。

🔍 perf 类型分析测试：远程 typedef 支持

Gary Guo 和 Dmitry Dolgov 参与的一个补丁，改进了 perf 工具对 Rust 类型信息的处理能力。具体来说，是让 perf 的类型分析（type profiling）功能能够正确处理"远程 typedef"——也就是在结构体上定义的类型别名。这属于让 perf 更好地理解 Rust 代码调试信息的基础工作，对于用 perf 分析 Rust 内核模块的性能很重要。

RUST 内核基础设施

Rust 内核基础设施

➕ 新增 num 模块和 PowerOfTwo 类型

Miguel Ojeda（Rust-for-Linux 项目维护者）提交了 3 个补丁，在内核的 Rust 支持库中新增了一个 num（数字）模块，其中包含 PowerOfTwo 类型。

在内核编程中，"2 的幂"是一个非常常见的概念——内存页大小（4096 = 2¹²）、对齐要求、缓冲区大小等等，很多地方都要求值必须是 2 的幂次方。以前大家只能用普通的整数来表示，然后在需要的地方手动检查"这个值是不是 2 的幂"。有了 PowerOfTwo 类型后，类型系统本身就能保证这个值一定是 2 的幂，编译器帮你检查，不需要运行时反复验证，既安全又高效。这是 Rust 类型系统优势在内核中的又一个漂亮应用。

⚙️ 设备驱动与 IO_URING

⚙️ 设备驱动与 io_uring

🔄 io_uring 命令抽象 for miscdevice（v4）

Sidong Yang 提交了第 4 版补丁（共 5 个补丁），为 Rust 的 miscdevice（杂项设备）框架添加 io_uring 命令支持。io_uring 是 Linux 近年来最重要的性能创新之一，它让应用程序可以用异步方式高效地与内核通信，而不需要反复陷入"等待"。

这组补丁的目标是：让用 Rust 编写的杂项字符设备驱动也能处理来自 io_uring 的命令请求。这意味着 Rust 驱动开发者将能利用 io_uring 的高性能异步 I/O 能力，而不用回退到传统的同步 ioctl 接口。Greg Kroah-Hartman（Linux 驱动子系统维护者）参与了审核，说明这个方向得到了上游的重视。

📬 查看补丁
🐛 Binder 驱动修复：避免在 TF_UPDATE_TXN 时错误调用

Alice Ryhl（Google）修复了 Rust 版 Binder 驱动中的一个 bug。Binder 是 Android 系统中进程间通信（IPC）的核心机制。这个补丁修复的问题是：当收到一个 TF_UPDATE_TXN（更新事务）类型的请求时，代码不应该调用 pending_oneway_finished() 函数，因为这个函数是给单向（oneway）事务完成时用的，在更新事务的场景下调用它会导致逻辑错误。补丁很小但很关键，属于正确性修复。

社区讨论（Zulip）

🔌 通用 I/O 后端：宏格式化之争

这个话题有 32 条消息，是今天最热闹的讨论。Alice Ryhl、Gary Guo、Benno Lossin 和 Danilo Krummrich 正在设计一套通用 I/O 后端抽象，涉及一个 project! 宏的语法格式问题。

Gary Guo 发现 rustfmt（Rust 代码格式化工具）会把宏里方括号 [] 内的 variant .. end 格式化成 variant..end（去掉空格），因为 rustfmt 把 .. 识别为 Rust 的范围运算符或子切片语法。这不是他们想要的效果。

Benno Lossin 分析了原因（允许子切片语法导致的），并提出了几个解决思路：一是使用 rustfmt 的一个不稳定选项来禁用这个行为（但可惜是 unstable 的，内核不太好用）；二是让大家改用花括号 project! { ... } 调用宏，因为花括号包裹的宏内容 rustfmt 不会去格式化（但这也不太美观）。讨论还在继续，大家在权衡各种方案的利弊。

🖥️ 新人求助：如何用 Rust 编写 DRM 驱动？

Link Mauve 在 Help 频道请教如何用 Rust 写一个新的 DRM（图形显示）驱动。Alistair Popple 表示自己也正在玩 Nova DRM 驱动，尝试实现一个需要调用 nova-core 的 ioctl，遇到了完全相同的问题。

核心问题是关于 DevRes（设备资源管理）的使用。Link Mauve 发现可以用 tdev.the_devres.try_access().ok_or(EBUSY)? 来安全地访问设备资源，而不是直接用 .access()。Janne Grunau 解释说：对于 DevRes<IoMem>（I/O 内存映射的设备资源），当你无法证明设备仍然绑定时，应该使用 fallible（可失败的）try_access() 方法。他还透露，长期计划是在 DRM ioctl 处理中保证设备一定是绑定状态的。

🔗 Dependent DevRes：嵌套设备资源的生命周期

Danilo Krummrich 和 Gary Guo 在讨论一个高级话题：当一个结构体内部有两个 DevRes 包装的资源，而其中一个引用了另一个时，如何处理生命周期。Danilo 指出这不是真正的"自引用结构体"（self-referential struct，Rust 中著名的难题），而是引用了"同一个外部结构体中的另一个成员"。Gary 确认只要它们被分别包装在独立的 DevRes 容器中，这种模式就是安全的。这是一个涉及 Rust 所有权模型在内核驱动中实际应用的重要设计问题。
