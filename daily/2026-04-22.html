️ GPU 驱动：NOVA CORE 多线推进

️ GPU 驱动：Nova Core 多线推进

1. 驱动卸载时通知 GSP 固件（v2）

今天 Nova Core（社区正在用 Rust 编写的开源 NVIDIA GPU 驱动）是最忙碌的模块，一共有 7 组补丁在推进。可以把 Nova 想象成"用 Rust 重新造一个 NVIDIA 显卡驱动"——它需要跟显卡上的一个叫 GSP（GPU System Processor）的小固件"对话"，才能让显卡工作。

作者：Alexandre Courbot, Eliot Courtney（NVIDIA）
之前 Nova 驱动卸载时，GSP 固件还在"傻傻地跑着"，继续访问内核已经回收的共享内存区域——这就像你搬走了但快递还往老地址送，特别是在没有 IOMMU（硬件地址隔离）的机器上会导致严重问题。这组补丁（共 5 个）让驱动卸载时发送 UNLOADING_GUEST_DRIVER 命令，告诉 GSP"我要走了，你也停下来吧"。同时还重构了卸载流程的代码结构，让错误处理更清晰。

📬 查看补丁
2. 新增 warn_on_err 宏 + 在 Nova 中使用（v2）

作者：Alexandre Courbot, Eliot Courtney（NVIDIA）
在 Rust 里经常需要检查一个操作是否出错（Result 类型），如果出错了想打一条内核警告。之前需要手写好几行代码，现在新增了一个 warn_on_err! 宏来简化这个模式——一行代码搞定"如果出错就警告"。Nova 的驱动卸载流程是第一个使用者，用它来在卸载失败时发出警告，同时重构了卸载代码让它在出错时提前退出。

📬 查看 warn_on_err 宏 📬 查看 Nova 使用
3. 重构 GSP 固件命令的导入方式（v2）

作者：Alexandre Courbot, Eliot Courtney（NVIDIA）
之前代码里把所有固件（FW）层面的命令都 use 导入到了 GSP 命令模块里，导致一个问题：看到命令名字时分不清它是固件层的还是 GSP 层的。而且如果 GSP 层想创建一个同名命令就会冲突。这个补丁停止了"全量导入"的做法，改为显式地用模块路径来引用固件命令，让代码层次更清晰。这就像把"张三"改成"北京的张三"和"上海的张三"，避免混淆。

📬 查看补丁
4. 内存管理：TLB 刷新支持（v11，20 个补丁）

作者：Joel Fernandes（NVIDIA）
这是一组大型补丁系列，已经迭代到第 11 版了！TLB（Translation Lookaside Buffer）是 GPU 里的一块"地址翻译缓存"——GPU 用它来快速查找"虚拟地址→物理地址"的映射。当内存映射发生变化时，必须"刷新"（清空）TLB，否则 GPU 会访问到错误的内存地址。这组补丁为 Nova 的内存管理模块（mm）添加了 TLB 刷新的完整支持，是让 GPU 正确管理显存的关键基础设施。

5. VBIOS 解析：拒绝多余的固件镜像（v2，11 个补丁）

作者：Timur Tabi, Joel Fernandes（NVIDIA）
VBIOS 是显卡的"启动固件"，里面包含多种镜像（PCI-AT 引导镜像、FWSEC 安全固件镜像等）。之前的解析代码如果遇到多个同类型的镜像，不会报错，可能导致加载了错误的镜像。这组补丁加了严格检查：如果发现多余的 PCI-AT 或 FWSEC 镜像，直接拒绝，防止潜在的安全和稳定性问题。

📬 查看补丁
6. 从 GSP 固件提取可用显存区域（v11，20 个补丁）

作者：David Airlie, John Hubbard, Joel Fernandes（NVIDIA）
NVIDIA 显卡的显存（Framebuffer）并不是全部都能随便用的——GSP 固件自己也要占一部分。这组补丁让驱动在初始化时向 GSP 查询"哪些显存区域是我可以用的"，然后只在安全区域内分配内存。这就像搬进合租房时先问清楚"哪些柜子是我的"，避免占了室友的空间。这个系列已经到第 11 版，说明社区在反复打磨细节。

📬 查看补丁
7. VBIOS：计算 FWSEC 相对偏移（v3）+ GA100 支持（v5）

作者：Eliot Courtney, Alexandre Courbot（NVIDIA）
两个较小的改进。FWSEC（Firmware Security）是显卡安全启动相关的固件，补丁修正了其 Falcon 数据偏移量的计算方式。另一个补丁为 GA100（Ampere 架构的数据中心 GPU）添加了 FbHal::frts_size() 接口支持，让 FRTS（Falcon Runtime Staging）区域大小的查询适配更多 GPU 型号。

️ 架构支持：POWERPC 的 RUST 之旅

️ 架构支持：PowerPC 的 Rust 之旅

1. 修复 "multiple candidates for rmeta dependency core" 编译错误（V12）

作者：Gary Guo, Mukesh Kumar Chaurasiya（IBM）
这是一个很有意思的 bug：当在 PowerPC 64 位小端（ppc64le）上用 LLVM=1（使用 Clang 编译器）并且 -j1（单线程编译）时，rustc 编译器会报错说"找到了两个 core 库的候选项"。一个是系统自带的标准库里的 core，另一个是内核自己编译的 libcore.rmeta。这就像你电脑上装了两个版本的 Python，运行时不知道该用哪个。补丁通过调整编译搜索路径来解决了这个冲突，确保 rustc 只看到内核自己编译的 core 库。这个系列已经迭代到第 12 版，可见解决这类工具链兼容问题需要反复调试。

📬 查看补丁
2. 添加 PowerPC Rust 支持 + 启用 ppc64le（V12）

作者：Gary Guo, Mukesh Kumar Chaurasiya（IBM）
配套的两个补丁：一个在 Rust 构建系统中添加了 PowerPC 架构的支持代码，另一个在 PowerPC 的 Kconfig 里打开了 HAS_RUST 开关，让 ppc64le 平台可以编译带 Rust 的内核。目前 Rust-for-Linux 已经支持 x86_64、ARM64、RISC-V 和 LoongArch，PowerPC 将成为又一个支持的架构。这对 IBM 的 POWER 服务器用户来说是个好消息——以后也能享受到 Rust 带来的内存安全保障了。

⚙️ 驱动核心与平台框架

⚙️ 驱动核心与平台框架

1. 修复 NULL module_kset 空指针解引用（v2）

作者：Greg Kroah-Hartman, Shashank Balaji（Sony）
这个 bug 发生在系统启动的极早期：module_kset（内核用来管理模块的一个数据结构）是在 subsys_initcall 阶段才初始化的，但如果有内建驱动（built-in driver）在这之前就尝试注册自己，并且设置了 mod_name，就会尝试访问还没初始化的 module_kset，触发空指针崩溃。补丁很直接：在 lookup_or_create_module_kobject() 函数里加了一个 NULL 检查——如果 module_kset 还没准备好，就安全地跳过。这种"启动时序"问题在内核里很常见，就像你还没开店就有顾客来了，得先挂个"尚未营业"的牌子。

📬 查看补丁
2. 平台驱动注册时传递模块名（v2）

作者：Greg Kroah-Hartman, Shashank Balaji（Sony）
之前内建驱动（编译进内核镜像的驱动，不是 .ko 模块）在 sysfs 里看不到模块符号链接（module symlink），因为注册时没有传入 KBUILD_MODNAME。这个补丁修改了平台驱动的注册宏，让它把模块名传进去，这样驱动核心就能在 sysfs 里创建正确的模块链接了。重要的是，Rust 侧的平台驱动适配器也同步更新了，把模块名传递到新的参数里。补丁在 QEMU 上测试通过，覆盖了 x86 和 ARM64 配置。

📬 查看补丁
3. 为更多内建驱动启用 sysfs 模块符号链接（v2）

作者：Shashank Balaji（Sony）
这是上面两个补丁的"总包"封面信（cover letter）。目标是让更多编译进内核的驱动在 /sys 文件系统下能正确显示模块信息。这对系统管理工具（比如 udev、systemd）了解"这个设备是哪个驱动在管"非常有用。之前只有可加载模块（.ko）有这个链接，现在内建驱动也能有了。

RUST 内核基础设施

Rust 内核基础设施

1. task_work 抽象：让 Rust 代码也能"排队等回调"

作者：Ashutosh Desai，审阅：Greg KH
内核的 task_work API 允许任何内核线程向某个特定任务"排队"一个回调函数，这个回调会在目标任务下一次返回用户空间（或退出）时执行。这在很多场景下非常有用，比如"等用户进程准备好了再执行某个清理操作"。但 C 语言的接口需要手动管理 callback_head 的内存生命周期，还要小心初始化顺序——这正是 Rust 擅长的领域。这个补丁为 task_work 提供了安全的 Rust 封装，利用 Rust 的所有权系统自动管理回调的生命周期，避免 use-after-free 之类的内存错误。

📬 查看补丁
2. maple_tree：实现 Send 和 Sync trait

作者：Onur Özkan，审阅：Gary Guo, Joel Fernandes（NVIDIA）
Maple Tree 是 Linux 内核里的一种高效数据结构，主要用于内存管理中追踪虚拟内存区域（VMA）。之前 Rust 对 Maple Tree 的封装没有实现 Send 和 Sync 这两个 trait，意味着它不能安全地在多个线程间共享。这个补丁为 MapleTree 实现了这两个 trait，因为底层的 C 实现本身就是线程安全的（通过内部锁保护）。这让 Rust 代码可以在多线程环境中使用 Maple Tree，是后续更多内存管理功能 Rust 化的基础。

📬 查看补丁
💡 小白科普
Send 和 Sync：Rust 的两个核心 trait。Send 表示"这个类型可以安全地移动到另一个线程"，Sync 表示"这个类型的引用可以安全地在线程间共享"。它们是 Rust 编译器保证线程安全的基石。 了解更多 →
Maple Tree：Linux 6.1 引入的数据结构，替代了旧的红黑树来管理虚拟内存区域，查找和插入性能更好。名字来源于枫树（Maple）的分支结构。 了解更多 →

1. Synology NAS 微控制器驱动（v8）

作者：Markus Probst，审阅：Miguel Ojeda, Ilpo Järvinen, Krzysztof Kozlowski
如果你用过群晖（Synology）NAS，你可能知道它上面有指示灯、蜂鸣器、风扇控制等功能——这些都是由一颗叫"MicroP"的嵌入式控制器芯片管理的。这个补丁系列（已经到 v8）用 Rust 编写了这颗芯片的驱动程序！这是一个非常好的 Rust 驱动实例，因为它是一个真实的、有人用的硬件驱动，而不仅仅是示例代码。配套的还有设备树绑定（dt-bindings）的补丁，用 YAML 格式描述这颗芯片的硬件属性，让内核知道"这是什么设备、怎么跟它通信"。

2. AS5600 磁编码器传感器驱动（RFC 讨论）

作者：Muchamad Coirul Anwar，审阅：Miguel Ojeda
AS5600 是一种磁性旋转位置传感器（磁编码器），常用于电机控制、机器人关节等场景——它能通过检测磁场来精确测量旋转角度。这个 RFC（Request for Comments，征求意见稿）补丁用 Rust 为它编写了一个 IIO（Industrial I/O）子系统下的驱动。目前还在讨论阶段，社区在审阅代码结构是否合理、API 使用是否正确。这是 Rust 驱动生态扩展到工业传感器领域的一个信号。

社区讨论与工具链

社区讨论与工具链

1. Zulip 讨论：模块/驱动级别的数据作用域

参与者：Onur Özkan, Alvin Sun, Gary Guo, Daniel
在 Zulip 的 #General 频道，社区正在讨论一个架构层面的问题：如何在 Rust 驱动中管理"模块级别"和"驱动级别"的数据。核心挑战是：有些数据需要在驱动的整个生命周期内存在，但又需要能被安全地撤销（revocable）。Gary Guo 提到，当前的"惰性可撤销"（lazy revocable）抽象不能用于 Nova 驱动，因为 Nova 的撤销操作需要睡眠（sleeping），而当前的实现不支持在持有锁时睡眠。这个讨论涉及 Rust-for-Linux 最核心的设计问题之一：如何用 Rust 的类型系统来表达内核中复杂的数据生命周期。

💬 查看讨论
2. GitHub Issue #353：bindgen 功能需求与 bug 修复跟踪

维护者：ojeda（Miguel Ojeda）
这是一个长期跟踪的 meta issue，收集了社区对 bindgen（自动把 C 头文件转换成 Rust FFI 绑定的工具）的各种功能需求和已知 bug。bindgen 是 Rust-for-Linux 的关键工具链组件——没有它，Rust 代码就无法调用内核的 C 函数。这个 issue 上标记了 toolchain 和 meta 标签，说明它属于基础设施级别的重要事项。如果你对改进 Rust-for-Linux 的工具链感兴趣，这是一个很好的入口点。
