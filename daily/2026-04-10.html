️ GPU / DRM 驱动

nova-core：统一页表条目包装枚举（v10，21 个补丁）

NVIDIA 的 Alexandre Courbot 提交了 nova-core 驱动中内存管理（mm）模块的第 10 版大型补丁系列，共 21 个补丁。这组补丁的核心目标是为 GPU 的页表条目（Page Table Entry, PTE）提供统一的 Rust 枚举包装。简单来说，GPU 和 CPU 一样，需要通过"页表"来管理内存地址的映射关系——就像一本翻译字典，把程序使用的"虚拟地址"翻译成硬件实际使用的"物理地址"。之前 nova-core 中页表条目的表示方式比较零散，这组补丁将它们统一成枚举类型，让代码更整洁、更不容易出错。这个系列已经得到了 John Hubbard、Joel Fernandes 等 6 位审阅者的评审，迭代到了第 10 版，说明社区对代码质量的要求非常严格。

nova-core：修复 SysmemFlush 在启动失败时未注销的问题

John Hubbard 和 Eliot Courtney 修复了 nova-core 帧缓冲（fb）模块中的一个资源泄漏问题。在 GPU 初始化（Gpu::new）过程中，如果在 SysmemFlush（系统内存刷新组件）创建之后某一步失败了，之前的代码不会自动注销 SysmemFlush，因为它需要手动注销。补丁的解决方式是给 SysmemFlush 添加了一个 Drop 实现——这是 Rust 的"析构函数"机制，当对象被销毁时会自动调用，相当于给它加了一个"自动清理"的保险。同时保留了手动注销路径，因为手动路径可以保证不会失败（infallible），而 Drop 路径依赖于运行时状态。这是一个很好的案例，展示了 Rust 的 RAII（资源获取即初始化）模式如何帮助防止内核中的资源泄漏。

Tyr GPU 驱动：使用 register! 宏定义 GPU_CONTROL 寄存器块（v5，6 个补丁）

Collabora 的 Daniel Almeida 和 Deborah Brouwer 为 Tyr GPU 驱动提交了第 5 版补丁，使用内核的 register! 宏来定义 GPU_CONTROL 寄存器块。之前这些寄存器的地址、位掩码（bit mask）和位移量（shift）都是手写的常量，就像用一堆"魔法数字"来操作硬件——不仅容易出错，读起来也很痛苦。新的做法是用宏来自动生成类型化的寄存器定义，编译器可以帮你检查类型是否匹配，大大降低了写出 bug 的概率。手写的 GpuId 解析代码也被替换掉了。Gary Guo 参与了代码审阅。

⚡ IO_URING 异步 I/O

⚡ io_uring 异步 I/O

Rust io_uring 命令抽象层 for miscdevice（v4，5 个补丁）

Linux 内核维护者 Greg Kroah-Hartman（简称 GKH）提交了为 miscdevice（杂项设备）构建 Rust io_uring 命令抽象层的第 4 版补丁。io_uring 是 Linux 上最先进的异步 I/O 接口——传统的 I/O 操作就像排队买咖啡，你必须站在柜台前等；而 io_uring 像是手机下单，你提交请求后可以去做别的事，咖啡好了会通知你。这组补丁让 Rust 编写的杂项字符设备驱动也能利用 io_uring 的高性能异步机制来处理用户态命令。GKH 亲自推动这项工作，说明 Rust 在内核驱动开发中的地位正在稳步提升。

📬 查看补丁
修复 io_uring_cmd_prep() 中的未定义行为（v4）

同样来自 GKH 的补丁，修复了 io_uring_cmd_prep() 函数中 PDU（Protocol Data Unit，协议数据单元）未被零初始化的问题。在 C 语言中，如果你声明了一个变量但没有给它赋初始值，它里面的内容是"随机"的（实际上是之前残留在内存中的数据）。如果后续代码依赖这些未初始化的数据做判断，就会产生未定义行为（Undefined Behavior, UB）——程序可能以任何不可预测的方式运行。这个补丁通过在准备阶段将 PDU 清零来消除这个隐患。这类问题恰恰是 Rust 语言天然可以避免的（Rust 不允许使用未初始化的变量），也从侧面说明了为什么内核社区在推动 Rust 进入内核。

为 Rust 绑定添加 io_uring 头文件（v4，5 个补丁）

Sidong Yang 提交的补丁在 bindings_helper.h 中添加了 io_uring 相关的 C 头文件。这是 Rust 绑定的基础工作——Rust 代码要调用 C 内核函数，需要先通过 bindgen 工具从 C 头文件生成 Rust 端的函数签名。把 io_uring 的头文件加入到绑定辅助文件中，是让上面 GKH 的 io_uring Rust 抽象层能正常工作的前提条件。

允许 clippy::uninlined_format_args 规则（2 个补丁）

Tamir Duberstein 提交的补丁修改了内核的 Rust 构建配置（kbuild），放宽了 Clippy 中 uninlined_format_args 这条 lint 规则。Clippy 是 Rust 的"代码检查助手"，会在你写代码时给出各种改进建议。这条规则原本会要求你把 format!("{}", x) 写成 format!("{x}")（内联格式参数）。但在内核代码中，由于需要兼容多个 Rust 编译器版本，有时不能使用最新的语法特性，所以选择允许（allow）这个规则而不是强制执行。Gary Guo 参与了审阅，共 2 位审阅者确认了这个修改的合理性。

内核宏与 Rust 语言特性

用 const expect 替换 match + panic

Google 的 Alice Ryhl 提交了一个精巧的小补丁，将内核 Rust 宏中 match + panic 的模式替换为 const expect。在编译期常量上下文（const context）中，之前的做法是用 match 检查某个值，如果不符合预期就 panic!。但这种写法比较冗长，而且意图不够明确。新的 const expect 方式更加简洁直观——就像用 .expect("错误信息") 替换 if x.is_none() { panic!("错误信息") } 一样。这是一个代码风格上的改进，让内核中的 Rust 宏更加地道和易读。

📬 查看补丁
为 system_dfl_wq 添加 system_default() 包装（v5，2 个补丁）

Marco Crivellari 提交的补丁为内核的默认工作队列（system_dfl_wq）添加了一个 Rust 包装函数 system_default()。工作队列（workqueue）是内核中延迟执行任务的机制——有些工作不需要立即完成，就丢进队列里，让内核在合适的时候再处理，就像餐厅里的"待办订单队列"。system_dfl_wq 是内核内置的默认工作队列，几乎每个子系统都会用到。给它加上 Rust 包装函数，让 Rust 驱动可以方便地使用这个基础设施。这个补丁已经到了第 5 版。

安全协议：SPDM 的 Rust 实现讨论

RFC v3：用 Rust 在内核中实现 SPDM 协议（27 个补丁）

Alistair Francis 发起了关于在 Linux 内核中用 Rust 实现 SPDM（Security Protocol and Data Model）协议的第三轮 RFC 讨论。SPDM 是一种安全协议，用于设备之间互相验证身份和加密通信——想象你插了一块 PCIe 显卡，系统怎么知道这块显卡是"真的"而不是一个恶意设备？SPDM 就是解决这类信任问题的标准协议。这是一个规模很大的工作（27 个补丁！），用 Rust 实现可以在协议解析、密码学运算等安全敏感的代码中获得更强的内存安全保障。作为 RFC（Request for Comments），目前还在设计讨论阶段，距离合入主线还有一段路要走。

Zulip：KUnit 测试中设置自定义 kconfig 选项的踩坑记

Lyude 在 Zulip 的 #Help 频道分享了自己在运行 KUnit（内核单元测试框架）时遇到的配置难题。问题是：她想在 KUnit 测试的 .config 中启用某些选项（比如 CONFIG_RUST），但怎么也设不上。折腾了一番后发现关键卡点在于 randstruct（随机化结构体布局，一种安全加固措施）没有正确设置。她把它设成 Y 后，还发现需要 diff 对比修改前后的 config 文件才能确认配置选项真正生效了——因为 KUnit 的配置生成流程有一些不太直观的依赖关系。最终测试成功运行！Miguel Ojeda 也参与了讨论。这个经历对其他想跑 Rust 内核测试的新手来说非常有参考价值。
