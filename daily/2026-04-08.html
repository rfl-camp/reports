构建系统与工具链升级

今天最重要的一组变更围绕编译器最低版本升级展开。Miguel Ojeda 合并了一个补丁系列，正式宣布 Rust-for-Linux 将跟随 Debian Stable（Trixie）的 Rust 版本作为最低支持版本。这是在去年 LPC 2025 和维护者峰会上讨论后达成的共识。简单来说，以前你可能需要用很旧的 Rust 编译器来编译内核，现在门槛提高了——但好处是开发者能用上更多好用的新特性，不用再为兼容老版本写一堆绕行代码。

📬 查看合并提交

配合版本升级，Miguel 还新增了一套按编译器版本控制全局编译选项的机制（support global per-version flags）。在 Kbuild 构建系统中，有时候某个 Rust 版本的 lint（代码检查规则）可能有误报，我们需要针对特定版本关掉它。以前只能手动处理，现在有了统一的辅助函数，像 rustc-min-version 一样方便地按版本门控全局 flag。这让维护多版本兼容变得更干净。

文档大扫除

配合工具链版本升级，今天有一大波文档清理补丁被合并，主要集中在快速入门指南（Quick Start Guide）。这些改动的核心思路是：既然最低 Rust 版本提高了，很多老旧的发行版安装变通方案就不再需要了。

Ubuntu 相关更新（3 个补丁）：Ubuntu 25.04 已停止支持，25.10 成为当前最新支持版本（自带 Rust 1.85.1）。同时，即将在几周后发布的 Ubuntu 26.04 LTS（代号 Resolute Raccoon）被提前加入了文档，因为它的 Rust 工具链版本足够新。Ubuntu 的版本化 Rust 包（如 rust-1.85）也做了相应更新。这对 Ubuntu 用户来说是个好消息——升级到最新 LTS 就能直接开始内核 Rust 开发，不用折腾手动安装编译器。

📬 更新版本化包

发行版简化（3 个补丁）：删除了 Nix 需要"unstable channel"的说明（现在 stable channel 就够了）、删除了 Gentoo 需要"testing"分支的说明（stable 分支已满足要求）、以及 openSUSE 现在直接提供 rust-src 包不再需要版本号后缀。这些改动反映了一个积极信号：各大 Linux 发行版对 Rust 工具链的支持越来越好，入门门槛在持续降低。

️ 设备驱动基础设施

️ 设备驱动基础设施

新增 SizeConstants trait（已合并）：John Hubbard 提交了一个非常实用的补丁。内核里有一组 SZ_* 常量（如 SZ_4K、SZ_1M），用来表示内存大小。但这些常量的类型是 usize，匹配的是 CPU 的指针宽度（比如 64 位系统就是 64 位）。问题是——设备的地址空间宽度和 CPU 不一样！比如，一个 32 位的 MMIO 窗口、一个 64 位的 GPU 显存空间，它们需要不同位宽的常量。以前驱动开发者只能手动做类型转换（cast），既麻烦又容易出错。这个补丁引入了一个 SizeConstants trait，让不同位宽的地址空间类型都能方便地获取正确类型的大小常量。这就像给不同尺寸的螺丝配上了对应的螺母，不再需要每次都手动找转接头。

📬 查看补丁

IoPort 资源访问函数（PR #1199）：alexandruradovici 提交了一个 PR，为内核的 I/O 端口资源添加 Rust 绑定。I/O 端口是 x86 架构上一种古老但仍在使用的硬件通信方式（比如和键盘控制器、串口芯片通信）。这个 PR 的目标是提供一个安全的 Rust 函数，让驱动可以获取对 IoPort 资源的引用，而不用直接写 unsafe 的底层操作。虽然现代设备大多使用 MMIO（内存映射 I/O），但 I/O 端口在嵌入式和传统设备上依然重要。

📬 查看 PR #1199

️ 安全机制：控制流完整性（CFI）

️ 安全机制：控制流完整性（CFI）

修复 lru_status 的 CFI 编码（已合并）：Alice Ryhl 提交了一个针对 CFI 安全机制的重要修复。bindgen（自动将 C 头文件转为 Rust 绑定的工具）默认会把 C 语言的 enum lru_status 转换成一个整数类型的 typedef。在大多数情况下这没问题——同样大小的整数和枚举在调用时二进制层面是兼容的。但在启用了 CFI（控制流完整性）后就出问题了：CFI 会检查函数调用时参数的类型签名是否匹配，整数和枚举虽然大小一样但类型签名不同，于是 CFI 就会拦截这个调用，认为它是非法的。Alice 的修复方法是为 lru_status 声明正确的 cfi_encoding，让 CFI 知道这两种类型其实是兼容的。

📬 查看补丁

启用 KCFI 支持（PR #1034）：maurer 提交的这个 PR 目标更宏大——为 Rust-for-Linux 全面启用 KCFI（内核控制流完整性）。KCFI 是一种安全防护技术，防止攻击者通过篡改函数指针来劫持程序执行流程（这是很多内核漏洞的利用方式）。启用 KCFI 后，每次通过函数指针调用函数时，系统都会验证目标函数的类型签名是否正确。这个 PR 让 Rust 代码也能参与到这套保护机制中，是 Rust 在内核安全领域的重要一步。这个 PR 和上面 Alice 的修复密切相关——正是因为要推进 KCFI 支持，才发现了像 lru_status 这样的类型兼容性问题。

📬 查看 PR #1034
💡 小白科普

内核同步原语

内核同步原语

Mutex / Spinlock / Condvar（PR #990）：wedsonaf 持续推进内核同步原语的 Rust 绑定。在内核中，多个 CPU 核心可能同时访问同一块数据，必须使用锁来保证数据安全。这个 PR 为三种核心同步机制提供 Rust 封装：Mutex（互斥锁，等待时会让线程休眠）、Spinlock（自旋锁，等待时 CPU 忙等，适用于很短的临界区）、Condvar（条件变量，允许线程等待某个条件满足后再继续）。这些是写任何并发内核代码的基础积木。

📬 查看 PR #990

Pinned Module（PR #997）：同样来自 wedsonaf 的这个 PR 涉及 Rust 的 Pin 机制在内核模块中的应用。在内核中，很多数据结构被初始化后就不能再被移动（因为其他地方可能持有指向它的指针），Pin 正是 Rust 提供的保证"这个值不会被移动"的机制。这个 PR 为内核模块提供了更好的 Pin 支持，确保内核数据结构的内存安全。

📬 查看 PR #997
💡 小白科普

Pin：Rust 的一种智能指针，保证被包裹的值不会在内存中移动。在内核中特别重要，因为很多 C 数据结构用了"侵入式链表"等依赖固定地址的设计。📖 了解更多

if_cfg! 条件编译宏（PR #1218）：KaiserGranatapfel 提交了一个新的过程宏 if_cfg!，用于根据内核配置选项进行条件编译。在 Rust 中，标准的 #[cfg(...)] 属性只能用于编译器已知的配置项，但内核的配置选项（来自 Kconfig）需要特殊处理。这个宏让 Rust 内核代码可以更优雅地写"如果启用了某个内核配置就编译这段代码，否则编译另一段"的逻辑，而不是用一堆笨重的条件编译指令。

📬 查看 PR #1218
