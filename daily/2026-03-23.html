DMA（直接内存访问）抽象层大更新

DMA（直接内存访问）抽象层大更新

1. DMA 通用容器：不再只支持"切片"

今天 DMA 子系统是最热闹的方向，一共有三组补丁在同步推进，目标都是让 Rust 代码能更安全、更灵活地和硬件打交道。简单来说，DMA 就是让硬件设备（比如显卡、网卡）直接读写内存，不用 CPU 当"搬运工"——速度飞快，但如果管理不好，硬件可能读到错误的内存内容，甚至引发系统崩溃。

Gary Guo 和 Aditya Rajan 提交了第二版补丁（共 8 个 patch），核心变化是：以前 Rust 的 DMA 抽象只能处理"切片"（slice，一种连续的数组视图），现在扩展为可以支持任意类型的数据结构。打个比方，以前这个"快递箱"只能装一排排整齐的小物件，现在可以装各种形状的东西了。这意味着驱动开发者可以直接把自定义的结构体放进 DMA 缓冲区，不需要再手动做类型转换，既方便又减少了出错的可能。这组补丁已经收到了 3 条评审意见，社区在讨论 API 的命名和边界检查细节。

2. 移除 DMA_ATTR_NO_KERNEL_MAPPING 公开属性

Alexandre Courbot 提交了 2 个 patch，把 DMA_ATTR_NO_KERNEL_MAPPING 这个属性从 Rust 的公开 API 中移除了。这个属性的意思是"分配的 DMA 内存不需要在内核里有可访问的地址映射"——通常用于一些特殊硬件（比如某些嵌入式 GPU），它们自己直接访问内存，内核根本不需要看到这块内存的内容。之所以要移除它，是因为接下来会用一个更安全、更清晰的 CoherentHandle 类型（见下一条）来替代它。Alice Ryhl 参与了评审，确认这个清理方向是对的。

📬 查看补丁
3. 新增 CoherentHandle：无内核映射的 DMA 分配

Danilo Krummrich 和 Alexandre Courbot 合作提交了 2 个 patch，引入了一个全新的类型 CoherentHandle。它代表一块"一致性 DMA 内存"，但内核自己不持有这块内存的指针——只有硬件设备能访问。这在 Rust 的类型系统里是一个很巧妙的设计：因为你拿不到指针，编译器就不会让你意外地去读写这块内存，从根源上防止了误操作。这比 C 语言里靠程序员"记住别碰"要安全得多。配合上面移除旧属性的补丁，DMA 的 Rust API 正在变得越来越"Rust 味"。

⚡ 编译优化：不用完整 LTO 也能内联 C 辅助函数

⚡ 编译优化：不用完整 LTO 也能内联 C 辅助函数

Miguel Ojeda 发布了第二版补丁（共 3 个 patch），标题是"Inline helpers into Rust without full LTO"。这是什么意思呢？在 Rust-for-Linux 里，很多 Rust 代码需要调用 C 语言写的内核函数。有些 C 函数非常小（比如只有一两行），如果每次调用都走"函数调用"的流程，会有性能损耗。理想情况下，编译器应该把这些小函数的代码直接"复制粘贴"到调用处——这叫做内联（inline）。

问题是，要跨越 C 和 Rust 的边界做内联，以前需要开启"完整 LTO（Link-Time Optimization，链接时优化）"。完整 LTO 会让编译器在链接阶段把所有代码当成一个整体来优化——效果好，但编译速度会慢很多，对于 Linux 内核这种巨型项目来说代价太大。Miguel 的这组补丁找到了一种更轻量的方式，让这些 C 辅助函数可以在不开启完整 LTO 的情况下被内联到 Rust 代码中。这对日常开发体验有很大改善——编译快了，开发者的反馈循环就短了。该补丁已经收到评审，Miguel 自己也在讨论帖中跟进了技术细节。

GPU/DRM 驱动：依赖队列层提案

GPU/DRM 驱动：依赖队列层提案

Intel 的 Matthew Brost 提出了一个 RFC（征求意见稿）补丁系列，总共 12 个 patch，其中第 2 个 patch 引入了一个全新的 "DRM 依赖队列（dependency queue）"层。这是在讨论什么呢？

在 GPU 驱动里，任务（比如渲染一帧画面）之间往往有复杂的依赖关系：任务 B 必须等任务 A 完成才能开始，任务 C 又依赖 B 和 D 都完成。目前各个 GPU 驱动（AMD、Intel、NVIDIA）都在自己内部处理这些依赖关系，代码重复且容易出 bug。Matthew 提议在 DRM 子系统层面建一个通用的"依赖队列"，让所有 GPU 驱动都能复用同一套依赖管理逻辑。

这个 RFC 还在早期讨论阶段，目前已经收到了 1 条评审意见。社区在评估这个抽象层的必要性、性能影响以及能否适配不同厂商的 GPU 架构。这对 Rust GPU 驱动（比如 nova 驱动）的开发也有潜在影响，因为依赖管理是驱动调度的核心环节。

️ 安全注释（SAFETY COMMENTS）补全

️ 安全注释（SAFETY Comments）补全

1. rust/print 模块的安全注释

在 Rust 里，unsafe 代码块是允许你绕过编译器安全检查的"特权区域"。Rust 社区有个重要规范：每个 unsafe 块前面都要写一段 // SAFETY: 注释，解释为什么这段代码是安全的。这就像在实验室里操作危险化学品时，旁边必须贴一张"为什么这样操作是安全的"说明牌。今天有两组补丁在做这件事：

这个补丁由多位作者共同完成（Miguel Ojeda、Shivendra Sharma、Albab Hasan），为 rust/kernel/print.rs 中的 unsafe 代码块补写了 SAFETY 注释。print 模块是 Rust 内核代码中用来打印日志消息的基础组件（类似 C 里的 printk）。虽然打印日志看起来简单，但底层需要调用 C 内核函数，涉及裸指针操作，所以需要 unsafe。这个补丁解释了这些 unsafe 操作为何不会导致未定义行为——比如指针从何而来、为什么一定有效。目前已收到 3 条评审意见。

2. rust/workqueue 模块的安全注释

Albab Hasan 为工作队列（workqueue）模块中 Pin<KBox<T>> 相关的两个 unsafe trait 实现补写了安全注释。之前这些地方只有占位符 // SAFETY: TODO.——现在被替换成了详细的安全论证。具体来说，WorkItemPointer 的安全性依赖于 __enqueue 通过 T::raw_get_work 获取 work_struct 的正确性，而 RawWorkItem 的安全性依赖于 Work::new 保证的内存布局。工作队列是内核中"延后执行任务"的核心机制，把安全注释补全对后续维护和审计非常重要。

内核驱动核心：7.0-RC5 修复合入

内核驱动核心：7.0-rc5 修复合入

Danilo Krummrich 提交了一个 GIT PULL 请求，将驱动核心（driver core）的修复合入 Linux 7.0-rc5。Linus Torvalds 亲自参与了讨论。pr-tracker-bot（一个自动化机器人）也跟踪了这个 PR 的状态。

GIT PULL 是 Linux 内核的一种工作流：子系统维护者把一段时间积累的补丁整理好，然后请求 Linus 把这些补丁"拉"进主线内核。rc5 表示 7.0 版本的第 5 个候选发布版——这个阶段通常只接受 bug 修复，不再加新功能。Danilo 是 Rust 设备驱动框架的核心维护者之一，这次的修复应该涉及驱动模型的稳定性改进。目前已经有 3 条评审/讨论消息。

RUST 宏：TRYFROM 和 INTO 派生宏 V5

Rust 宏：TryFrom 和 Into 派生宏 v5

Jesung Yang 提交了第五版补丁（共 4 个 patch），为内核中的 Rust 代码添加 TryFrom 和 Into 派生宏（derive macros）。这两个是 Rust 标准库中最常用的类型转换 trait：Into 用于无损转换（比如 u8 转 u32），TryFrom 用于可能失败的转换（比如 u32 转 u8，值太大就会失败）。

在内核代码中，经常需要在不同整数类型、枚举值和底层硬件寄存器值之间互相转换。目前这些转换代码大多是手写的，重复且容易出错。有了派生宏，只要在类型定义上加一行 #[derive(TryFrom, Into)]，编译器就会自动生成转换代码。这是第五版了，说明社区对 API 设计非常谨慎，反复打磨了多轮。目前已收到 1 条评审意见。

社区工具：PIN-INIT 自动化发布流程调试

社区工具：pin-init 自动化发布流程调试

在 Zulip 聊天的 #pin-init 频道中，Benno Lossin 和 Gary Guo 正在调试 pin-init 库的 staging（暂存）工作流。pin-init 是 Rust-for-Linux 中一个非常核心的库，用于安全地初始化被 Pin 固定的数据结构（在内核中很常见，因为很多结构体一旦初始化就不能被移动到别的内存地址）。

Benno 发现他设计的自动化工作流"完全按预期工作了"——他在一个 测试仓库 中验证了整个流程。不过发现了一个小问题：如果临时分支删除得太快，GitHub 来不及把对应的 PR 标记为"已合并"。他的解决方案是加一个短暂的延迟（sleep(1)），然后通过 API 检查合并状态，最多重试 10 次。这虽然是个小细节，但对于保持 CI/CD 流程的可靠性很重要。
