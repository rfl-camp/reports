里程碑：LINUX 7.0 发布与 RUST 工具链更新

里程碑：Linux 7.0 发布与 Rust 工具链更新

Linux 7.0 正式发布——Linus Torvalds 打上了 Linux 7.0 标签，标志着又一个主线版本的诞生。对 Rust-for-Linux 来说，这次发布非常重要，因为 Linus 同时合并了 rust-7.1 标签，里面包含了一系列 Rust 基础设施更新。

这次 Rust 更新的核心变化是：最低 Rust 版本从之前的版本提升到了 1.85.0，bindgen（一个自动生成 C/Rust 绑定的工具）的最低版本也提升到了 0.71.1。这个决定是在 2025 年的 Linux Plumbers Conference（LPC）和 Maintainers Summit 上讨论确定的。版本提升意味着内核开发者可以使用更多新的 Rust 语言特性，也让一些之前需要绕路实现的功能可以更简洁地完成。

此外，本次合并窗口还包含了大量上游子系统更新的合并，包括 VFS（虚拟文件系统）的完整性检查、扩展属性（xattr）、写回缓存（writeback）、RCU 无回调 CPU 管理、调度器修复、定时器修复等，这些都是 Linux 内核正常迭代的一部分。

Nova-core：GA100 全面支持

Nova-core：GA100 全面支持

NVIDIA 的开源 GPU 内核驱动 nova-core 今天提交了一组共 6 个补丁的系列，目标是让 GA100 这款高端计算 GPU 能够正常工作。GA100 是 NVIDIA Ampere 架构中一个比较特殊的芯片——它是纯计算卡，没有显示输出硬件（就像一台没有屏幕的超级计算器）。这带来了几个需要特殊处理的地方：

Nova-core：统一页表项抽象（v11，20 个补丁）

Nova-core：统一页表项抽象（v11，20 个补丁）

这是一个相当大的补丁系列！GPU 有自己的内存管理单元（MMU），就像 CPU 的 MMU 一样，用来把虚拟地址翻译成物理地址。NVIDIA GPU 有两个版本的 MMU（v2 和 v3），它们的页表项格式不同。之前的代码需要为每个版本写一套逻辑，非常繁琐。

这组补丁引入了 Pte、Pde、DualPde 三个统一的枚举类型，每个类型内部包裹了 v2 和 v3 两种格式的变体。它们在构造时接受一个 MmuVersion 参数来决定用哪种格式，然后把具体操作委托给底层实现。这样，上层的页表遍历器（page table walker）和虚拟内存管理器（VMM）就可以写一套代码同时支持两种 MMU 版本了——就像写一个通用的充电器适配器，可以同时给不同接口的手机充电。

Nova-core：跳过 IFR 头部（6 个补丁）

Nova-core：跳过 IFR 头部（6 个补丁）

GPU 的 ROM（只读存储器）里存着 VBIOS（视频 BIOS），它是 GPU 启动时需要读取的固件。有些 GPU 的 ROM 开头多了一个叫 IFR（Init-from-ROM）的头部，真正的 PCI 扩展 ROM 镜像藏在后面。这个补丁让驱动在读取 ROM 时先检查有没有 IFR 头部，如果有就解析它，找到真正的 VBIOS 起始偏移量，然后从正确的位置开始读。大多数 GPU 没有这个头部，但必须处理这种情况。

Nova-core：VBIOS 安全加固（2 个补丁）

Nova-core：VBIOS 安全加固（2 个补丁）

这两个补丁修复了 VBIOS 解析代码中的潜在安全问题。具体来说，在 setup_falcon_data 和 FwSecBiosImage::ucode 函数中，有些值来自固件（不可信的外部数据），之前直接做算术运算可能会溢出。修复方法是使用 Rust 的 checked 算术操作（比如 checked_add、checked_mul），如果溢出就返回错误而不是继续执行。这是 Rust 在安全性上的典型优势——用编译器和标准库的力量防止低级错误。

📬 setup_falcon_data 补丁 📬 FwSecBiosImage 补丁
Nova-core：SysmemFlush 资源泄漏修复

Nova-core：SysmemFlush 资源泄漏修复

这是一个 bug 修复。在 GPU 初始化过程中（Gpu::new），如果 SysmemFlush 资源创建成功后，后续步骤失败了，SysmemFlush 不会被自动清理——因为它需要手动注销。补丁的解决方案是给它加一个 Drop 实现（Rust 的析构函数），这样在出错时会自动清理。同时保留了手动注销的路径，因为手动路径可以保证不会出错（infallible），而 Drop 路径依赖于其他资源所以可能失败。

Nova-core：Hopper/Blackwell GPU 识别（v10，28 个补丁）

Nova-core：Hopper/Blackwell GPU 识别（v10，28 个补丁）

这是一个大型补丁系列，为 nova-core 添加了对 Hopper 和 Blackwell 架构 GPU 的基础识别支持。这两个是 NVIDIA 最新的 GPU 架构（Hopper 用于数据中心 AI 计算，Blackwell 是其后继者）。目前这组补丁主要做的是让驱动能认出这些 GPU，还不涉及完整功能支持。已经迭代到了第 10 版，说明社区对代码质量要求很高，反复打磨。

Zulip 讨论：DMA Fence 的信号上下文用什么数据结构

Zulip 讨论：DMA Fence 的信号上下文用什么数据结构

在 DRM 驱动中，DMA Fence（DMA 栅栏）是 GPU 和 CPU 之间同步的机制——就像一个信号灯，告诉 CPU "GPU 已经完成这个操作了"。Philipp Stanner 在 Zulip 上提出，存储需要触发信号的 DMA Fence 指针时，XArray（一种类似数组的内核数据结构）可能是最好的选择，因为 fence 需要按顺序触发信号，而且经常需要跳过中间的某些 fence。Boris Brezillon 表示赞同，并分享了他在自己的分支中已经用 XArray 实现的代码。

Zulip 讨论：新人如何开始写 DRM 驱动

io_uring 命令抽象（v4，5 个补丁）

io_uring 命令抽象（v4，5 个补丁）

这个系列把 io_uring 的命令直通（passthrough）功能引入了 Rust 世界。io_uring 是 Linux 最新一代的高性能异步 I/O 框架，可以理解为"给内核发一堆工作请求，然后高效地等结果回来"。其中 IORING_OP_URING_CMD 允许驱动自定义命令。这个补丁系列的做法是：先把 io_uring 的 C 头文件引入 Rust 绑定，然后构建核心抽象，最后把它接入 miscdevice（杂项设备）框架，这样 Rust 驱动就可以处理 io_uring 的直通命令了。来自 Greg Kroah-Hartman（Linux 稳定版维护者）的评审是个好信号，说明这个方向得到了上游大佬的关注。

Workqueue 安全取消和状态方法（v4，3 个补丁）

资源依赖管理：Dependent DevRes

资源依赖管理：Dependent DevRes

Gary Guo 和 Danilo Krummrich 在 Zulip 上深入讨论了一个棘手的设计问题：当一个设备资源（DevRes）依赖于另一个资源时，如何确保它们按正确的顺序释放？具体场景来自 nova-core 的 SysmemFlush 修复——SysmemFlush 需要在它依赖的资源之前被注销。

Danilo 指出 Gary 提议的方案可能导致自引用结构（一个结构体的某个字段借用了同一个结构体的另一个字段），这在 Rust 中是不允许的。Gary 反驳说这并不算自引用，因为从结构体的视角看，它只是引用了"别的东西"。Danilo 提出了另一种方案，但 Gary 认为那种方案只解决了 Drop（析构）的问题，而他想要的是一个更通用的解决方案，让资源在存活期间也能安全地引用其依赖。这是 Rust 所有权模型在内核场景中的典型挑战。

Wenzhao Liao 在 Zulip 上介绍了自己正在做的项目：一个将 Linux 内核 C 驱动翻译成"可审查的"Rust 代码的工具。他希望找到愿意花时间评审 AI 生成代码的维护者来合作改进工具。Miguel Ojeda 仔细询问了项目的定位——是需要维护者评审 AI 生成的代码来改进翻译工具？还是希望翻译的代码直接上游？Wenzhao 澄清说，他的重点是让翻译出的代码没有 unsafe，可以被人类审查和理解，而不是直接提交上游。这是一个很有潜力的方向，但也面临着"AI 生成的代码该如何评审"这个新挑战。
