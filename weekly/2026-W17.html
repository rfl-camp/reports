️ GPU 驱动：Nova-Core 全力推进 GA100 支持

本周最密集的活动来自 Gary Guo，他提交了一组 5 个补丁（均为 v5 版本，属于同一个 6 补丁系列），目标是让 Nova 驱动能够正确支持 NVIDIA GA100 GPU——也就是大家熟悉的 A100 数据中心加速卡。这些补丁各有分工，下面逐一介绍：

1. 修正 GA100 的固件签名（fwsignature）
GA100 的固件签名格式与其他 GPU 不同。这个补丁确保驱动在加载固件时使用正确的签名格式来校验固件的完整性，防止加载到错误或损坏的固件。就好比不同型号的钥匙要配对应的锁——GA100 需要它专属的"钥匙模板"。

📬 查看补丁

2. 正式启用 GA100 支持
在所有前置补丁就位后，这个补丁将 GA100 的 PCI ID 添加到驱动的支持设备列表中，让内核在检测到 A100 硬件时能自动加载 Nova 驱动。这是整个系列的"画龙点睛"之笔。

📬 查看补丁

3. 拒绝无效的 PCI ROM 头签名（0xBB77）
某些 GPU 的 PCI ROM（存储固件/配置的只读存储器）中会出现值为 0xBB77 的假签名。这个补丁让驱动在解析 ROM 时跳过这类无效数据，避免将"垃圾数据"当成有效固件来处理，从而防止潜在的初始化失败。

4. 仅在 FRTS 区域已分配时才引导
FRTS（Fault Replay Table Storage）是 GPU 显存中用于故障恢复的一块特殊区域。并非所有 GPU 配置都会分配这块区域。这个补丁增加了一个判断：如果 FRTS 区域未被分配，就跳过引导步骤，而不是尝试操作一块不存在的内存——这可以避免空指针访问或硬件错误。

📬 查看补丁

5. 为 GA100 添加 FbHal::frts_size() 方法
GA100 的 FRTS 区域大小与其他 GPU 不同，需要通过硬件抽象层（HAL）返回正确的尺寸。这个补丁为 GA100 实现了 frts_size() 方法，让驱动知道该分配多大的 FRTS 区域。这是上一条补丁的前置条件。

📬 查看补丁
💡 小白科普
Nova 驱动：这是一个用 Rust 语言编写的全新 NVIDIA GPU 开源驱动，目标是替代现有的 C 语言 Nouveau 驱动。它目前还在早期开发阶段，正在逐步添加对不同 NVIDIA GPU 型号的支持。
HAL（硬件抽象层）：一种软件设计模式，它在驱动和硬件之间加了一层"翻译"，让同一套驱动代码可以通过不同的 HAL 实现来支持不同的硬件型号——就像同一个遥控器 App 配上不同的红外码就能控制不同品牌的电视。

构建系统与 RUST 工具链

构建系统与 Rust 工具链

HeeSu Kim 提交了一组 v6 补丁（共 2 个），解决的核心问题是：内核在用 rustdoc 生成文档和运行文档测试时，需要传递一个编译器参数 -Cunsafe-allow-abi-mismatch=fixed-x18 来绕过 Rust 编译器的两个 bug（#144521 和 #146465）。但这个 workaround 只对特定版本的 rustc 有意义——1.90.0 之后修复了文档生成 bug，1.92.0 之后修复了文档测试 bug。

补丁 1：引入 rustc-lt-version 宏
现有的构建系统里已经有 rustc-min-version（用来检查"编译器版本 ≥ X"），但缺少反向检查"编译器版本 < X"的能力。这个补丁在 scripts/Makefile.compiler 中添加了 rustc-lt-version 宏，采用严格小于（非包含）的比较方式，让版本边界表达更清晰。打个比方：以前只能说"至少要 18 岁才能进入"，现在还能说"未满 18 岁走这个通道"。

📬 查看补丁

补丁 2：精确限定 workaround 的生效版本范围
有了上面的新宏，这个补丁就可以在 rust/Makefile 中用条件判断来精确控制 workaround 只在受影响的编译器版本上生效。对于已经修复了 bug 的新版本编译器，不再添加这个多余的参数。这是一种很好的工程实践：workaround 不应该"永远存在"，它应该随着上游修复而自动退出。值得注意的是，这已经是 v6 版本了，说明补丁经历了多轮社区审查和打磨。

📬 查看补丁
💡 小白科普
rustdoc：Rust 语言自带的文档生成工具，它可以把代码中的注释自动转换成漂亮的 HTML 文档，还能把文档中的代码示例当作测试来运行（doctest），确保文档和代码保持一致。
Workaround（变通方案）：当上游工具存在已知 bug 时，下游项目添加的临时修补措施。好的 workaround 应该标记清楚它应对的是哪个 bug，并在 bug 被修复后及时移除。

同步原语：为 GLOBALGUARD 补上安全网

同步原语：为 GlobalGuard 补上安全网

Ashutosh Desai 提交了一个简短但重要的补丁：为 GlobalGuard 和 GlobalLock::try_lock 添加 #[must_use] 标注。

在内核开发中，锁（Lock）是保护共享数据的核心机制。当你"获取锁"时，会得到一个"守卫"对象（Guard），只要这个守卫还活着，你就持有锁。当守卫被丢弃（drop）时，锁就会自动释放。问题在于：如果开发者调用了 try_lock() 获取守卫，却忘记把它赋值给一个变量，守卫会被立刻丢弃，锁瞬间释放——这意味着你以为自己在锁的保护下操作数据，其实完全没有保护。

内核中已有的 Guard 类型已经加了 #[must_use]（编译器会警告你"这个值不应被忽略"），但封装它的 GlobalGuard 却漏掉了。这个补丁填上了这个空缺，让编译器帮你抓住这类隐蔽的并发 bug——想象一下你以为门锁上了，其实钥匙一转就弹开了。

📬 查看补丁
💡 小白科普
#[must_use]：Rust 编译器提供的一个属性标注。加上它之后，如果你调用了一个函数却没有使用它的返回值，编译器会发出警告。这对锁守卫、错误类型（Result）等"丢弃就有副作用"的值特别重要。
RAII（资源获取即初始化）：一种编程范式——资源的生命周期绑定到对象的生命周期。锁守卫就是典型的 RAII：创建守卫 = 获取锁，销毁守卫 = 释放锁，不需要手动"解锁"。

设备驱动：RUST I2C 生态持续壮大

设备驱动：Rust I2C 生态持续壮大

本周设备驱动方面有两个值得关注的动态，它们共同指向一个趋势：Rust 正在从内核基础设施层向具体硬件驱动层扩展。

1. AS5600 磁旋转位置传感器驱动（RFC）
贡献者 Muchamad Coirul Anwar 提交了一个 RFC（征求意见）补丁，用 Rust 为 ams OSRAM AS5600 磁旋转位置传感器编写了初始驱动。AS5600 是一种通过 I2C 总线通信的传感器，常用于电机角度测量、旋钮输入等场景。这个驱动目前覆盖了基本的 I2C 通信功能，属于 IIO（工业 I/O）子系统的一部分。虽然还处于 RFC 阶段（即"我写了个初版，大家来看看思路对不对"），但它展示了 Rust 驱动开发正在逐步触达更多具体硬件。

📬 查看 RFC

2. Rust I2C 绑定合入请求（GIT PULL）
I2C 子系统维护者 Wolfram Sang 发出了 rust-i2c-next 分支的合并请求，目标是进入 Linux v8.0 内核。这意味着 Rust 语言的 I2C 总线抽象层已经通过了子系统维护者的审查，即将正式成为主线内核的一部分。这是一个里程碑事件——一旦 I2C 的 Rust 绑定进入主线，后续像上面 AS5600 这样的 I2C 设备驱动就可以直接基于主线 API 来开发了，不再需要依赖外部补丁。

📬 查看合并请求
💡 小白科普
I2C（Inter-Integrated Circuit）：一种非常常见的低速串行通信协议，用两根线（数据线 SDA 和时钟线 SCL）连接多个芯片。你的手机里的传感器、笔记本的触控板、智能家居设备内部，大量使用 I2C 通信。
IIO（Industrial I/O）：Linux 内核中专门管理传感器的子系统，包括加速度计、陀螺仪、温度传感器、磁传感器等。它提供统一的接口让用户空间程序读取传感器数据。
