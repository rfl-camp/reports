RUST 工具链版本大升级

Rust 工具链版本大升级

本周最重要的事件之一：Rust-for-Linux 的最低 Rust 编译器版本从之前的版本正式提升到了 1.85.0，同时 bindgen（用于自动生成 C 与 Rust 绑定代码的工具）的最低版本也一并升级。这个决定遵循了在 LPC 2025 和维护者峰会上达成的共识——跟随 Debian Stable（Trixie）发行版中附带的 Rust 版本。这是一种务实的策略：既能使用较新的语言特性，又不会把门槛设得太高，因为 Debian Stable 是很多服务器和开发者的常用发行版。

这次升级不只是改了一个版本号，而是由 Miguel Ojeda 提交的一个包含 33 个补丁的大型补丁集，已经全部合入 rust-next 分支。John Hubbard、Gary Guo、Tamir Duberstein 等核心开发者参与了 Review，至少有 4 位 Reviewer 给出了意见。升级带来的主要变化包括：

代码简化：由于新版本中多个之前不稳定的 API 已经稳定化，大量代码得以简化。比如 extract_if（从集合中按条件提取元素）在 1.87.0 稳定后，宏代码可以直接使用它；split_at_checked() 在 1.80.0 稳定后简化了类型转换代码；NonNull::add() 稳定后简化了内存分配代码。这些变化体现了一个良性循环：随着 Rust 语言的成熟，内核代码可以越来越多地使用标准库功能，减少自己维护的"黑科技"代码。

📬 查看版本升级补丁集

Clippy lint 调整：clippy::incompatible_msrv 这个 lint（代码检查规则）被全局允许了。这个 lint 原本是提醒你"你用了比你声明的最低版本更新的 API"，但在内核场景下它的误报太多，维护者们多次讨论后决定直接关掉它。另一个变化是 clippy::precedence——1.85.0 中这个 lint 被扩展到检查位运算和移位的优先级，但因为触发太频繁，1.86.0 又拆分成了独立的 precedence_bits lint。为了兼容使用 1.85.x 编译器的开发者，补丁添加了按版本条件禁用的机制。

📬 clippy::incompatible_msrv 补丁 📬 clippy::precedence 补丁

构建系统改进：新增了"按编译器版本设置全局编译标志"的机制。这解决了一个实际痛点——有时某个 lint 只在特定编译器版本有误报，你需要只在那个版本关掉它。此外还清理了多个历史遗留的兼容性 hack：移除了 --remap-path-prefix 的变通代码、移除了旧版 bindgen 需要的"假参数"技巧、移除了旧版 bindgen 与 libclang 19.1 兼容性问题的警告。

📬 per-version flags 补丁

GPU NOVA-CORE 驱动开发

GPU Nova-Core 驱动开发

Nova-Core 是正在开发中的 NVIDIA GPU 开源内核驱动，用 Rust 编写，本周在多个方面取得了进展。

TLB 刷新支持已推进到 v10：Joel Fernandes 提交的内存管理模块 TLB（Translation Lookaside Buffer）刷新支持补丁已迭代到第 10 版，共包含 21 个补丁。TLB 是 GPU 中用于加速虚拟地址到物理地址转换的高速缓存，当页表发生变化时必须刷新 TLB，否则 GPU 可能访问到错误的内存。这个功能对 GPU 内存管理至关重要。3 位 Reviewer 参与了审查。

📬 查看 TLB 刷新补丁 v10

GSP 固件相关修复：Danilo Krummrich 提交了 GSP（GPU System Processor）命令队列中未定义行为的修复，已迭代到 v5。GSP 是 NVIDIA GPU 上运行固件的一个协处理器，驱动通过命令队列与它通信。"未定义行为"在 Rust 中是非常严肃的问题——它意味着编译器不保证代码行为的正确性，可能导致难以排查的 bug。此外，John Hubbard 和 Eliot Courtney 审查的 NV_STATUS 错误码绑定也推进到 v3（共 9 个补丁），这为从 Rust 代码中更好地处理 NVIDIA 固件返回的状态码奠定了基础。

📬 NV_STATUS 错误码绑定 v3

Falcon 固件 DMA 对齐修复：Alexandre Courbot 发现了一个因为切换 DMA 内存分配类型而引入的 bug。之前 nova-core 使用自定义的 DmaObject 类型，它会自动将分配大小对齐到页面大小（通常是 4KB）。切换到内核全局的 dma::Coherent 类型后，这个自动对齐消失了，导致 Falcon 微控制器的固件 DMA 对象大小可能不满足硬件要求的块对齐。v2 修复了这个问题。

📬 Falcon DMA 对齐修复 v2

I/O 抽象层类型泛化

I/O 抽象层类型泛化

Alexandre Courbot 本周提交了一组 8 个补丁的系列，对内核 Rust I/O 抽象层进行了一次重要的类型系统重构。

从地址到指针类型：第一个变化是将 I/O 操作从使用原始地址（整数）改为使用 Rust 的指针类型。这看起来是个小改动，实际上意义重大——Rust 的指针类型携带了更多的类型信息和安全保证，编译器可以做更多的静态检查，减少"不小心把一个普通数字当地址用"这类错误。

📬 指针类型补丁

Mmio 泛化为任意类型：更大的变化是将 Mmio（Memory-Mapped I/O，内存映射 I/O）结构从只能映射固定的寄存器类型，泛化为可以映射任意类型。Gary Guo 参与了审查，两人就设计细节展开了技术讨论。这种泛化使得驱动开发者可以用类型安全的方式访问各种 MMIO 设备的寄存器布局——你定义一个 Rust 结构体来描述寄存器布局，然后通过 Mmio 安全地读写它们，编译器会帮你检查偏移量和类型是否正确。

📬 Mmio 泛化补丁 📬 Gary Guo 的审查讨论

DMA 错误码修正：一个小但重要的修复——当 DMA 分配大小发生溢出时，将返回的错误码从 ENOMEM（内存不足）改为 EOVERFLOW（溢出）。这是 kernel test robot 自动发现的问题。正确的错误码有助于调试：如果你看到 ENOMEM 你会去查内存是否不足，但实际原因是请求的大小溢出了——使用正确的错误码能省去排查的弯路。

📬 DMA 错误码修复
🔰 小白科普
MMIO（Memory-Mapped I/O）：一种让 CPU 通过读写普通内存地址来控制硬件设备的技术。想象你家的智能家居面板——你按一个按钮（写一个内存地址），灯就亮了（硬件执行了操作）。MMIO 就是这个"面板"在计算机里的实现方式。
类型投影（Type Projection）：从一个大的类型中安全地"取出"某个字段对应的子视图。比如你有一个描述所有寄存器的大结构体，类型投影让你只操作其中的某个寄存器，且编译器保证你不会越界。

新设备驱动进展

新设备驱动进展

群晖 NAS 微控制器驱动 v6：Markus Probst 提交的 Synology（群晖）NAS 设备微控制器驱动已经迭代到第 6 版。群晖是一家知名的 NAS（网络附属存储）设备厂商，他们的 NAS 内部有一个微控制器通过串口与主处理器通信，用来控制 LED 指示灯、风扇转速、蜂鸣器，以及处理关机/重启按钮和风扇故障检测。这个用 Rust 编写的驱动目前支持控制电源 LED、状态 LED 和警告 LED，未来会扩展更多功能。该补丁系列依赖于尚在开发中的 Rust LED 抽象层。设备树绑定的补丁也引发了 Krzysztof Kozlowski 等 4 位开发者的审查讨论。

📬 驱动代码补丁 📬 设备树绑定讨论

Goldfish 虚拟设备驱动 RFC v3：来自中国人民大学的 Wenzhao Liao 提交了 Goldfish Address Space 驱动的第 3 版 RFC（征求意见稿），共 6 个补丁。Goldfish 是 Android 模拟器使用的虚拟设备平台，这个驱动实现了其中地址空间管理的 ioctl 子集。补丁系列涉及多个基础设施增强：为 PCI 设备添加了共享 BAR 内存映射支持，为页面管理添加了 ping 状态辅助函数，还加固了 misc 设备的注册流程和文件操作不变量。作为 RFC，目前还没有 Review 回复，这组补丁主要是在探索可行的实现路径。

📬 Goldfish 驱动 RFC v3

ACPI PRP0001 匹配修复：Markus Probst 修复了 Rust ACPI 驱动中一个设备匹配的问题。在 ACPI（高级配置与电源接口）标准中，PRP0001 是一个特殊标识符，表示设备的属性应该从设备树（Device Tree）的兼容字符串中查找。之前 Rust 代码在这种设备上会得到空的匹配数据（id_info 为 None），因为 Rust 存储的是索引值而不是直接的数据指针，所以不能使用通用的 device_get_match_data 函数。修复方法是导出并使用 acpi_of_match_device 函数来专门处理这种情况。Greg Kroah-Hartman 参与了审查。

📬 ACPI PRP0001 修复 v2
🔰 小白科普
设备树（Device Tree）：一种描述硬件配置的数据结构，告诉操作系统"这台机器上有哪些硬件、连在哪里"。就像一份电脑的"零件清单"。嵌入式设备（如 NAS、手机）广泛使用设备树来描述硬件。
RFC（Request for Comments）：在内核社区中表示"这是一个初步方案，我知道还不完善，请大家提意见"。与正式补丁相比，RFC 更侧重于讨论方向是否正确，而不是代码细节。

内核基础设施小改进

内核基础设施小改进

时间子系统的调试改进：Miguel Ojeda 为 fsleep（灵活睡眠）函数添加了 pr_warn_once! 宏调用，在睡眠时间参数被截断时打印一次警告。这个补丁引发了 Greg KH 等 4 位开发者的讨论。fsleep 是一个"智能"睡眠函数，会根据请求的时长自动选择最合适的底层睡眠机制（微秒级用忙等待，毫秒级用定时器），但对过大的值会截断。现在如果发生截断，开发者能在内核日志中看到警告，方便调试。

📬 fsleep 警告补丁
🔰 小白科普
fsleep：内核中的"灵活睡眠"函数。就像你设闹钟——如果只需等几微秒，就原地打转等（忙等待）；如果要等几毫秒，就真的"睡一觉"让出 CPU（定时器睡眠）。fsleep 自动帮你选最省资源的等待方式。

社区互动与新手求助

社区互动与新手求助

I2C 驱动调试：在 Zulip 的 #Help 频道，Christian Meusel 遇到了一个常见的新手问题——编写了一个 Rust I2C 驱动后看不到调试输出。经过排查发现，原来是忘记在驱动的设备 ID 表中注册 I2C 设备（一个 GPIO 扩展器）。这个问题随后被 Onur Özkan 标记为已解决。这类"驱动怎么没反应"的问题通常就是设备 ID 没有正确匹配——内核靠设备 ID 来决定哪个驱动应该被加载来管理哪个设备。

💬 查看 Zulip 讨论

寻找合适的 Rust 驱动练手目标：在 Zulip 上有人寻求建议——想找一个适合初学者、容易通过 Review 的 Rust 驱动目标来练手。Miguel Ojeda 回复了讨论，对关于"严格结构化的管线用于强制安全边界"的说法提出了疑问，指出仅仅添加一些静态检查仍然属于工具生成的范畴。这个讨论反映了社区对"什么样的驱动适合用 Rust 重写"这个问题的持续思考。

💬 查看 Zulip 讨论

GitHub 议题追踪更新：本周有 3 个 GitHub 议题活跃，均为 Miguel Ojeda 维护的元追踪议题。其中最值得关注的是 #1223（内核所需的 Rust 不稳定特性——已完成）和 #2（内核所需的 Rust 不稳定特性），随着本周 Rust 版本升级，一些之前标记为不稳定的特性已经稳定化，这些追踪议题得到了相应更新。#1132 追踪 LLVM 和 Clang 的需求与 bug 修复。

🔰 小白科普
I2C：一种简单的两线通信协议，用于连接低速外设（传感器、LED 控制器、EEPROM 等）。你可以把它想象成一条"公共汽车"线路——多个设备共享同一条总线，每个设备有自己的"站名"（地址），主设备喊站名来和特定设备通信。
设备 ID 匹配：内核通过比对"设备说自己是什么"和"驱动说自己支持什么"来自动配对。如果驱动没有正确声明支持的设备 ID，内核就不会把设备交给这个驱动管理，驱动的代码根本不会运行。
