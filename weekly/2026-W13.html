将 C 内联辅助函数引入 Rust（无需完整 LTO）—— v2 热烈讨论

将 C 内联辅助函数引入 Rust（无需完整 LTO）—— v2 热烈讨论

这是本周最重磅的话题。目前内核中有很多用 C 写的"辅助函数"（helper），它们是 Rust 代码调用 C 内核 API 的桥梁。问题在于：这些小函数如果不能被"内联"（即把函数体直接嵌入调用处），会带来额外的函数调用开销。此前的方案需要开启完整的 LTO（链接时优化），但 LTO 会大幅拖慢编译速度。Miguel Ojeda 提交的 v2 方案通过编译器的 always_inline 属性，让这些辅助函数即使不开 LTO 也能被内联到 Rust 代码中。本周围绕此补丁的讨论非常活跃（超过 20 条回复），涉及 ARM、x86 等多个架构的兼容性问题，Russell King、Arnd Bergmann、Marek Szyprowski 等资深内核开发者都参与了讨论。Marek 在三星 ARM 平台上测试发现了一些问题，推动了进一步迭代。

KASAN 消毒器：移除 -Zsanitizer-recover 标志

KASAN 消毒器：移除 -Zsanitizer-recover 标志

KASAN（Kernel Address Sanitizer）是内核中用来检测内存越界访问的工具。Alice Ryhl 发现 Rust 编译器被传入了 -Zsanitizer-recover 标志（遇到错误时继续运行而不停止），但 C 编译器 Clang 端并没有传同样的标志。为了保持 Rust 和 C 行为一致，该补丁移除了这个标志，使得内存错误发生时程序会立即停止，更有利于发现问题。

dma::Coherent 泛化：从字节切片到任意类型 —— v2

dma::Coherent 泛化：从字节切片到任意类型 —— v2

DMA（直接内存访问）是设备绑过 CPU 直接读写内存的方式，而"一致性 DMA 内存"（Coherent DMA）是 CPU 和设备都能看到相同数据的特殊内存区域。此前 Rust 的 Coherent 类型只能包装字节切片（[u8]），本周 Aditya Rajan 的 v2 补丁将其扩展为可包装任意类型——比如一个结构体或自定义数据。这让 DMA 内存的使用变得更加灵活。Andreas Hindborg 和 Danilo Krummrich 就 API 设计展开了深入的技术讨论。

外部 crate 使用讨论

有人在 Zulip Modules 频道询问是否可以在内核模块中使用外部 Rust crate。Miguel Ojeda 明确表示：内核不太可能支持任意外部 crate——内核代码有特殊的内存分配器、错误处理和 ABI 要求，标准库的 crate 很难直接适配。

小白科普
BPF（Berkeley Packet Filter）：一种让用户在内核中安全运行小程序的技术。最初用来过滤网络包，现在已经扩展到了跟踪、安全策略、甚至 I/O 调度等领域。好比在内核中开了一个"小沙盒"，让用户可以安全地自定义内核行为。
RCU（Read-Copy-Update）：内核中一种高效的并发数据访问机制，允许读者在不加锁的情况下访问数据，写者则创建副本修改后再替换。
